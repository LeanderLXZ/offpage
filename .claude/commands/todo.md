# /todo — todo_list 状态总览 + 下一步建议

把 `docs/todo_list.md` 解析成结构化总览（按优先级 / 工作量 / 阻塞依赖 / 待决策项数标列），然后基于当前状态给出 2–4 条具体下一步建议（包含一个"跳过 todo 干别的"逃生口）。**只读**——不改 todo_list、不改代码、不 commit。`$ARGUMENTS` 可选作为关键字过滤（如 `schema` 只显示 ID 含 schema 的条目）；不传则列全部。

## 0. 读 todo_list

读 `docs/todo_list.md` 全文。仅识别**实际任务条目**——形如 `### [T-{SLUG}] 标题`，按所属二级标题（`## 立即执行` / `## 下一步` / `## 讨论中（未定案）`）分段。文件说明（`## 文件说明` 段）跳过。

文件不存在或无任务条目 → 打印"⚠️ docs/todo_list.md 缺失或为空"并停手。

## 1. 解析每条任务

对每个 `### [T-XXX]` 条目，从其下面的子段提取：

- **ID**：方括号里的 slug，如 `T-CHAR-SNAPSHOT-13-DIM-VERIFY`
- **标题**：方括号后的中文短语
- **简介**：从「上下文」段提取**首句**（截断到 60 字内），落到不下贴整段
- **段落归属**：立即执行 / 下一步 / 讨论中
- **预估**：抓「**预估**：S(...)」/「**预估**：M」/「**预估**：L」；缺则记 `—`
- **依赖**：抓「**依赖**：xxx」首句；`无` / `已完成` 类标 ✅ ready，其他标 ⚠️ blocked + 简短依赖说明
- **待决策项数**：数 `**待决策项**` 段下的列表条目数；缺该段 → 0
- **改动文件数**：数「**改动清单**」下出现的文件路径（可选简显）

## 2. 推断"重要等级"

不靠 todo_list 自带字段（它没有），基于段落归属 + 阻塞状态合成：

| 重要等级 | 触发条件 |
|---|---|
| 🔴 **高** | 段落 = 立即执行 |
| 🟡 **中** | 段落 = 下一步 且 依赖 ready 且 待决策项 ≤ 1 |
| 🟢 **中低** | 段落 = 下一步 且（依赖 blocked 或 待决策项 ≥ 2） |
| ⚪ **低 / 待定** | 段落 = 讨论中 |

"重要等级"是建议性视觉标签，用户可以推翻。

## 3. 推断"立即可做性"

| 标签 | 触发条件 |
|---|---|
| ✅ **可 /go** | 依赖 ready 且 待决策项 = 0 |
| 💬 **需先讨论** | 待决策项 ≥ 1 |
| ⏸ **被阻塞** | 依赖中含具体阻塞名（外部 CLI、未实装模块、未发生事件 等） |

## 4. 输出主表

按段落顺序（立即执行 → 下一步 → 讨论中）逐段打印。每段一个 markdown 表：

```markdown
## 🔴 立即执行（{n} 条）

| ID | 简介 | 重要 | 立即可做 | 预估 | 待决策 | 依赖 |
|---|---|---|---|---|---|---|
| `T-XXX` | 一句话简介… | 🔴 高 | ✅ 可 /go | S | 0 | 无 |
| `T-YYY` | 一句话简介… | 🔴 高 | 💬 需先讨论 | M | 2 | 无 |

## 🟡 下一步（{n} 条）

…

## ⚪ 讨论中（{n} 条）

…
```

每条 ID 用 ` ``  ` markdown 行内代码包裹，便于点击复制。如果 `$ARGUMENTS` 传入关键字，先过滤 ID 含该关键字的条目。

## 5. 总览统计

主表后打印一行汇总：

```
共 {N} 条 — 🔴 立即执行 {a} ｜ 🟡 下一步 {b} ｜ ⚪ 讨论中 {c} ｜ ✅ 可 /go {d} ｜ 💬 需讨论 {e} ｜ ⏸ 被阻塞 {f}
```

## 6. 给出下一步建议（2–4 条）

基于状态合成。固定模板：

- 立即执行段有 ≥1 条 ✅ 可 /go → **建议 1**：`/go {第一条 ID}` —— 直接落地最高优先级
- 立即执行段全部 💬 需先讨论 → **建议 1**：选一条讨论待决策项（列出该条 ID + 待决策项条数）
- 立即执行段为空 → **建议 1**：从下一步提升一条到立即执行（列出下一步首条 ID）
- 任意状态都给 → **建议 2**：列出 1–2 条 🟡 中等 / 待决策少的下一步条目，问"要不要现在拉到立即执行段"
- 任意状态都给 → **建议 3**（**逃生口**）：跳过 todo，问用户是否有别的想做的事（"或者你心里有别的事？说一下"）

如果有 ⏸ 被阻塞的条目长期卡着，可补一条建议："清理 todo——这些 X 条是否还需要保留？"（仅当被阻塞条目 ≥ 3 才提）

输出格式：

```markdown
## 接下来想干啥？

1. **直接 /go T-XXX**：{为什么这条值得先做的一句话}
2. **讨论 T-YYY 的待决策项**：{有哪 N 个分歧 / 待拍板点}
3. **聊聊别的**：你有想做的别的事吗？告诉我即可

（讨论中 5 条、被阻塞 4 条都依赖外部条件未变；本轮无需触动）
```

建议数量 2–4 条；多了用户不看，少了没选项。逃生口"聊聊别的"必出。

## 7. 等待用户响应

打完建议后**停手**。不要进入 `/go`、不要改代码、不要改 todo_list、不要 commit；等用户从建议里挑一条，或自己说要做别的事。

## 约束

- **只读**：不改任何文件、不 commit、不 push
- **不省略**：所有 ID 都要出现在主表里（除非 `$ARGUMENTS` 关键字过滤）；不抓"前 5 条"之类
- **简介必须 ≤ 60 字**：用户扫读时看的是简介，长了反而看不进去
- **重要等级是建议**：用户可以说"我觉得 T-XXX 该升立即执行"；这时不动 todo_list，建议下一轮 `/go` 时同步重排
- **逃生口必出**："聊聊别的"建议永远在；用户可能根本不想跟着 todo 走
