# 项目需求文档

本文档是用户对 persona-engine 的核心需求规范。所有架构设计、Schema 定义、
提取流程和运行时加载策略都应以本文档为最终权威。

---

## 一、总体目标

建立一个长期稳定的小说角色扮演系统，能够：

- 从长篇小说中提取角色的"灵魂"
- 支持任意指定角色的深度扮演
- 让角色在不同剧情阶段有不同的状态、性格和表现
- 支持长期对话和记忆积累
- 未来可接入多种终端（AI agent、前端 app、移动端 MCP）

核心原则：**深度扮演，而非表面模仿。**

---

## 二、剧情阶段模型

### 2.1 基本规则

- 每个提取 batch 对应一个剧情阶段（batch N = 阶段 N）
- **分批规划是分析阶段最核心的产出**——每个 batch 边界直接成为系统的 stage 边界，
  世界快照、角色快照、记忆时间线、运行时阶段选择全部建立在此切分之上。
  切分的质量直接决定角色体验的连贯性。
- 分批按自然剧情边界确定，剧情边界的准确性优先于章节数量均匀，
  每个 batch 的章节数可以不同
  （默认目标 10 章，最小 5 章，最大 20 章，可在作品 config 中调整）
- 阶段 N 是累积的：选择阶段 N 意味着提取阶段 1..N 的全部内容

### 2.2 时间性原则

- 当前所选阶段是角色的"现在"
- 之前的阶段是按顺序发生的历史事件
- 角色能记住发生过的事情，但必须用**当前阶段的状态、性格和口吻**与用户对话
- 世界状态同理——只有当前阶段的快照反映"此刻"的世界

### 2.3 世界信息的架构

#### 2.3.1 世界基础设定层

每部作品应有一组世界基础设定文件，描述不随剧情推进而大幅变化的底层规则：

- **世界观设定**：世界的基本架构（如仙侠、科幻、现代等）
- **力量体系**：修炼等级、魔法系统、科技水平等
- **宇宙观/天道规则**：世界运行的底层逻辑

基础设定可能在后续阶段被补充或修正（如新发现推翻早期认知），修正应记录在
世界快照的 `foundation_corrections` 中。

#### 2.3.2 世界实体追踪

系统应追踪以下世界级实体：

- **主要事件**：影响整个世界或多个角色的重大事件。世界层只记录大事件，
  小事件和个人经历归角色层
- **地点**：重要的场景、地域、建筑。地点可能随剧情变化（毁灭、新建等）
- **势力/阵营**：组织、门派、国家等。势力可能合并、分裂、消亡
- **主要角色索引**：主角团和高频配角的基本信息索引。世界层只收主要角色，
  不收大量龙套和一次性小角色。详细角色信息归角色包

#### 2.3.3 世界级关系

世界层的关系分为两类：

- **固定关系**：不随阶段变化的结构性纽带（如父子、兄妹、血亲），始终有效
- **动态关系**：随剧情演化的关系状态（如恋人、盟友、仇敌、师徒），按阶段记录

#### 2.3.4 世界阶段快照

每个世界阶段快照应包含：

- 对基础设定的修正与补充
- 累积历史事件
- 当前世界状态
- 关注的人物关系的转变
- 人物的状态（生死、恋爱、等级等可能随时间变化的状态）
- 地点的变化
- 地图的变化（势力范围、领土变动等）

世界阶段快照与角色阶段快照使用相同的 `stage_id`，确保世界状态与角色状态
在同一时间线上对齐。运行时加载某个阶段时，同时加载该阶段的世界快照和角色
快照。

### 2.4 角色信息的剧情阶段

每个角色阶段快照应包含：

- 人物从上个阶段到此已经经历的事件
- 目前状态（生死、恋爱、等级等可能随时间变化的状态）
- 目前的性格、口吻发生的转变
- 心情
- 一切可能随着时间变化的东西

### 2.5 角色间关系的阶段演化

角色的性格、口癖、行事作风，以及对其他特定角色的态度，都会随剧情阶段不断
变化。这些变化由角色之间共同经历的事件、记忆、隐瞒是否被揭露等因素驱动。
角色之间也可能相互影响对方对世界的认知和性格。

系统必须支持以下能力：

- **对特定角色的态度随阶段变化**：同一角色在不同阶段对同一对象可能从敌意
  转为亲密，再从亲密转为悲痛。阶段快照必须精确记录当前阶段对每个重要角色
  的态度、信任度、亲密度。
- **事件驱动的关系转变**：态度变化不是凭空发生的，必须关联到具体事件
  （如共同经历、误会澄清、背叛、牺牲、死亡与复活等）。
- **知识边界影响行为**：角色在某个阶段可能不知道某件事（如对方的真实身份、
  对方是否已死亡），这会直接影响角色的反应。当用户以某个身份出现时，角色
  应基于当前阶段的知识状态做出真实反应（如惊讶、质疑、感动）。
- **隐瞒与揭露的阶段追踪**：角色对某些信息的隐瞒状态可能在某个阶段被揭露，
  揭露前后角色的行为和态度应有明显差异。
- **相互影响**：角色之间可能通过互动改变彼此的世界观、性格和行为模式。
  这些影响应该被记录，并在对应阶段的快照中体现。

**示例**：

1. 阶段 A：男女主初遇，相互抱有敌意
2. 阶段 B：经历若干事件后态度逐渐缓和
3. 阶段 C：相爱，相互许诺
4. 阶段 D：男主死亡，女主悲痛欲绝——此时若用户以男主身份对话，女主应惊讶
   反问"你不是死了吗？"，确认复活后感动流泪
5. 阶段 E：男主复活，女主难以置信但极度感动，重新开心

系统加载任意阶段时，AI 必须能完整复刻该阶段的世界观、性格、对每个角色的
态度和语气，做出符合该阶段的真实反应。

### 2.6 跨阶段历史回忆

角色在当前阶段对话时，必须能够回忆过去阶段的具体行为细节，而不仅仅是
事件摘要。这些细节包括但不限于：

- 过去阶段对特定角色使用的称呼和口癖
- 过去阶段对特定角色的态度和行为模式
- 过去阶段的知识边界（当时知道什么、不知道什么）
- 过去阶段隐瞒或被隐瞒的信息
- 过去阶段关系状态的具体原因

角色在回忆时，必须使用**当前阶段的性格和语气**来谈论过去，而不是切换回
过去的性格。回忆应体现角色对过去经历的**当前认知和态度**。

**示例**：

- 阶段 1：女主和男主是仇家，女主叫男主"臭小子"，男主隐瞒自己是女主前世
  爱人的身份，女主不知道男主的真实身份。
- 阶段 10：男女主已经相爱，女主性格仍然害羞，且已知道男主的真实身份。
- 加载阶段 10 后，用户（男主）问："你以前叫我什么？"
- 女主应当能回答（以阶段 10 害羞的语气）："我当时虽然叫你臭小子，但是
  咱们那个时候还是仇家嘛。人家又不知道你的真实身份。"

这要求系统在以下层面提供足够的历史细节：

1. **记忆条目**：memory_timeline 中的记忆不能只是事件摘要，还应记录当时
   的行为细节（称呼、口癖、态度、知识状态等）。
2. **关系演变概述**：relationship_history_summary 应包含关系演变中的关键
   行为标记（称呼变化、态度转折点、知识揭露时点等）。
3. **按需深度加载**：当以上信息不足时，系统应支持按需加载过去阶段的快照
   以获取更完整的历史细节。

### 2.7 阶段选择

- 每个角色的每个剧情阶段，需要有一个一句话总结，用于用户选择
- 用户在开始对话时选择一个阶段，该阶段在整个对话上下文中固定

## 三、深度扮演的三大目标

### 目标 1：系统化的角色资料结构

将小说中与角色相关的信息，从原始剧情中提炼出来，并整理为结构化资料。
这些资料至少应包括：

- 角色基础身份信息
- 性格特征与内在驱动力
- 情绪触发点与行为模式
- 关键关系网络
- 重要经历与时间线记忆
- 语言风格与典型台词特征
- 禁止偏离的人设边界
- 容易写崩或误判的点

### 目标 2：建立角色视角的记忆体系

不是只总结剧情，而是要把剧情转化为"角色本人记得的内容"。
需要从角色的视角梳理：

- 角色经历过哪些关键事件
- 角色在不同阶段知道什么、不知道什么
- 角色误会过什么、隐瞒过什么
- 角色对其他角色的态度是如何变化的
- 角色在情感上的变化轨迹是什么

目标是形成一份可供角色扮演直接调用的"角色记忆库"或"记忆时间线"。

### 目标 3：建立稳定的语言与行为模拟规则

AI 不仅要"知道角色是谁"，还要"说得像角色、反应得像角色"。
需要提炼：

- 角色常用的语气
- 角色偏好的表达方式
- 角色在不同情绪下的语言变化
- 角色面对不同对象（亲近者、陌生人、敌对者）时的说话差异
- 角色在暧昧、愤怒、委屈、嘴硬、关心、吃醋等状态下的典型反应模式

目标是总结出一套可以直接指导角色扮演的语言风格规则和行为规则。

## 四、运行时加载需求

当 AI 进行角色代入，加载某个角色的某个剧情阶段时，需要能迅速代入以下六类信息：

### A. 基本人设

- 姓名
- 年龄
- 身份
- 外貌特征
- 社会地位
- 重要关系

### B. 性格核心

- 表层性格
- 内核性格
- 优点
- 缺点
- 情绪触发点
- 逆鳞 / 底线
- 安全感来源
- 常见防御机制

### C. 说话风格

- 语气
- 用词习惯
- 是否毒舌 / 傲娇 / 冷淡 / 温柔 / 高压
- 是否爱反问
- 是否会直接表达感情
- 在不同对象面前说话的区别

### D. 记忆库

- 经历过的重大事件
- 知道的秘密
- 误会过什么
- 失去过什么
- 和其他角色的重要互动时间线

### E. 行为规则

- 遇到挑衅时会怎样反应
- 遇到暧昧时会怎样掩饰
- 生气时怎么说话
- 吃醋时怎么说话
- 受伤时会不会嘴硬
- 会不会主动示弱

### F. 禁止偏离项

- 绝不会做什么
- 绝不会说什么
- 容易被误写崩的点有哪些

### 运行时行为规则

除了加载数据外，运行时还需要以下行为规则指导 AI 的实时反应：

- **认知冲突处理**：当角色的已有认知与对话中出现的现实冲突时（如角色认为
  某人已死但该人出现），AI 应基于角色当前的知识状态、性格和关系做出真实反应
  （惊讶→质疑→确认→情绪爆发），而非直接接受新信息。这些行为不预设场景，
  而是由数据（知识边界、关系、性格）+ 规则（处理步骤）实时推理
- **历史回忆处理**：角色回忆过去时，优先使用已加载的记忆条目和关系演变概述；
  不足时按需加载过去阶段快照；始终用当前阶段人格表达；禁止编造历史、混淆
  阶段、泄漏角色当时不知道的信息

## 五、用户流程与会话生命周期

### 5.1 新用户创建流程

1. 输入用户 ID
2. 选择作品（从已入库作品中选择）
3. 选择要模拟的目标角色
4. 选择目标角色的剧情阶段
5. 选择自身角色：
   - 可以选择作品中已有的角色（characters/ 中已建好的）
   - 可以选择新建自定义角色（保留入口，后续细化）
   - 自身角色的剧情阶段与目标角色保持一致
6. 创建用户包：
   - 将所选角色的基础信息、剧情阶段、当前状态写入用户信息
   - 创建用于模拟角色修正、用户事件记忆等的目录结构
7. 创建新 context，加载对应剧情阶段的世界信息、目标角色信息、用户信息

### 5.2 已有用户登录流程

1. 输入已有用户 ID
2. 显示当前账户信息
3. 检查是否有可恢复的 context：
   - 如果有：列出可选 context 让用户选择
   - 如果没有：提示用户并创建新 context
4. 加载对应剧情阶段的世界信息、目标角色信息、用户信息

### 5.3 关键约束

**一次性设定锁定**：每个用户只能在初始创建时进行设定选择（作品、目标角色、
自身角色、剧情阶段）。创建完成后设定锁定，无法更改。如需更改必须创建新用户
包或执行显式迁移。

**会话关闭与合并**：当用户说出结束对话的关键词时：

1. 退出当前对话
2. 询问用户是否将当前对话内容并入用户长期历史
3. 如果用户同意合并：
   - 将 context 中积累的角色情绪/性格变化、事件、记忆点提炼写入用户的
     长期 profile（`long_term_profile.json`）和关系核心
     （`relationship_core/`）
   - 事件和记忆点是**只增添不覆盖**的
   - context 状态变为 `merged`
4. 如果用户拒绝合并：
   - context 保持原状，不影响长期 profile

**实时更新 vs 合并更新**：

- Context 内的内容（对话记录、角色实时状态、事件、记忆点）是**实时更新**的
- 用户的长期 profile 和关系核心**只在合并确认后才更新**
- 这两层严格分开：context 是临时工作区，长期 profile 是持久化的人格积累

**Context 内实时追踪的状态**：

对话进行中，context 应持续追踪以下增量状态：

- 角色情绪变化
- 性格微漂移（因对话事件产生的临时性格变化）
- 语气微调（因对话情境产生的临时语气变化）
- 双方达成的约定/承诺
- 关系增量（态度、信任度、亲密度的微调）
- context 内发生的事件
- context 内产生的新记忆

这些状态在 context 内实时可用，但只在合并时才被筛选性地提升为长期状态。

**崩溃恢复**：

系统应支持对话中断后的恢复能力：

- 每轮对话应有日志记录机制，确保用户输入不会因中断而丢失
- 中断后恢复时，应能检测到未完成的轮次并安全恢复

**对话归档**：

合并确认后的对话记录应归档为不可变的账户级历史记录：

- 归档内容包含：对话摘要、关键时刻、完整会话记录
- 归档记录是不可变的历史——合并后不再修改
- 运行时启动时仅加载归档摘要和索引，完整记录按需加载

## 六、数据分离原则

- 客观剧情事实 vs 角色主观认知——必须分开
- 原著正典 vs 合理推理——必须标注区分
- 角色 canon 数据 vs 用户对话产生的数据——用户数据不得污染角色 canon
- 不同角色的知识边界——角色不能知道它不该知道的事
- 不同阶段的状态——角色不能表现出未来阶段才有的特征
- 不同作品的命名空间——每部作品是独立的命名空间，所有运行时请求和持久化
  文件必须显式携带 `work_id`，不能仅依赖目录路径来区分作品

**内容语言一致性**：作品的语言设定（由 manifest 中的 `language` 字段声明）
应作为该作品下所有生成内容的默认语言。这不仅适用于作品级 canon（世界包、
角色包），也适用于用户侧与该作品相关的数据（关系描述、长期 profile 中的
作品相关内容、context 内的角色状态描述等）。

## 七、信息分层与加载原则

### 7.1 信息分层

系统不是一个巨大的 prompt，而是分层组织的：

1. **不变层**：identity、hard_boundaries、failure_modes——不随阶段变化，
   运行时始终加载
2. **自包含阶段快照**：stage_snapshot 包含该阶段完整的语气、行为、边界、
   关系、性格、心情、知识状态——运行时直接加载，不需要与 baseline 合并
3. **历史记忆**：memory_timeline 按阶段拆分为独立文件（每个阶段一个
   `{stage_id}.jsonl`），加载阶段 N 时读取阶段 1..N 的所有记忆文件；
   过去阶段的完整快照可按需加载以支持深度历史回忆
4. **对话可变层**：context 内的角色状态（情绪变化、性格微漂移、语气微调、
   关系增量）每轮实时更新
5. **跨对话积累层**：long_term_profile、relationship_core——仅在会话关闭
   并确认合并后更新

Baseline 文件（voice_rules、behavior_rules、boundaries）仍然存在，
但仅作为提取时的参照锚点，**不在运行时加载**。

### 7.2 分层加载

运行时应按需分层加载，而非全量装入：

- **启动核心层**：进入对话前必须加载的最小集——角色不变层、当前阶段快照、
  历史记忆、世界快照、用户状态摘要。启动层只加载摘要，不加载完整对话记录
- **结构化按需层**：对话中按需加载——世界事件详情、地点/势力信息、过去阶段
  快照、旧 context 摘要、旧会话摘要等
- **对话记录层**：仅在需要精确回忆时加载——完整对话记录、逐轮摘要
- **原文验证层**：仅在摘要冲突或需要高精度引用时加载——原始章节正文

每部作品可通过配置文件自定义各层的加载范围。

## 八、作品入库与规范化

### 8.1 支持的原始格式

系统应能接受以下格式的原始小说文件：

- **TXT**：纯文本文件，可能有不同编码（UTF-8、GBK 等）
- **EPUB**：电子书格式，需要解包提取正文
- **MOBI**：Kindle 格式，需要转换后提取正文
- **HTML**：网页抓取的章节，需要去除标签和格式
- **用户手动提供的片段**：部分章节或摘录

### 8.2 规范化流程

原始文件入库后必须经过规范化处理，产出可供后续提取使用的标准化文本：

1. **原始保存**：原始文件不做任何修改，保存在 `sources/works/{work_id}/raw/`
2. **格式转换与清洗**：
   - 统一编码为 UTF-8
   - 去除格式伪影（HTML 标签、排版控制符、广告水印等）
   - 修复明显的编码错误和乱码
   - 产出干净的纯文本，保存在 `normalized/`
3. **章节拆分**：
   - 按章节边界拆分为独立文件，保存在 `chapters/`
   - 每章一个文件，零填充编号（如 `0001.txt`）
   - 每个文件包含完整的一章正文
4. **元数据生成**：
   - `metadata/book_metadata.json`：书名、作者、章节数、来源信息
   - `metadata/chapter_index.json`：有序章节列表，含标题和文件路径

### 8.3 Source Package 边界

- Source package 只包含原始素材和规范化产物
- 分析、提取和规范化产出归属 `works/{work_id}/`，不属于 source package
- Source package 是输入层，下游流程不应修改它
- Source package 默认不纳入版本控制（通过 `.gitignore` 排除）

### 8.4 作品 Manifest

每本作品入库时必须创建 `sources/works/{work_id}/manifest.json`，至少包含：

- `work_id`：必须与目录名一致
- `title`：原语言标题
- `language`：ISO 639-1 语言代码（如 `"zh"`、`"en"`）
- `source_types`：原始格式标签数组（如 `["epub"]`）
- `ingestion_status`：`"pending"` / `"active"` / `"complete"`

中文作品的 `work_id` 应直接使用中文书名。

## 九、提取流程

### 9.1 提取原则

- 增量提取，不从头重来
- 中文作品使用中文标识和路径
- 提取结果需要有原文证据支撑
- 矛盾和修订必须显式记录
- 不要将分析简化为普通文学摘要——要聚焦于对角色扮演有用的信息

### 9.2 端到端提取流程

从原始小说到可运行角色包的完整流程如下：

1. **作品入库**：原始文件规范化、章节拆分、元数据生成（见 §八）
2. **全书总体分析**：基于元数据和章节标题做轻量分析，评估体量，建议分批策略
3. **源文件分批规划**：制定 batch plan，按自然剧情边界切分（默认目标 10 章，
   最小 5 章，最大 20 章），每个 batch 的章节数可因剧情节点而不同
4. **候选角色识别**：基于前几批章节，识别可建包的候选角色
5. **活跃角色确认**：用户从候选中选择要建包的目标角色
6. **协同批次提取**：每个 batch 同时产出世界信息更新和角色信息更新
7. **针对性补充提取**：协同批次完成后，对明显缺口进行定向补充
8. **包验证与发布**：验证产出物的完整性和一致性

### 9.3 协同提取模式

每个 batch 读取一次正文后，应同时产出：

- **世界信息更新**：世界快照、阶段目录、基础设定修正、动态关系等
- **角色信息更新**：角色快照、记忆时间线、baseline 文件（仅 batch 1）等

不要求先把整条世界线完整抽完再回头做角色线。只要已确认活跃角色集合，
就应采用"同批读一次、双层同时产出"的方式。

### 9.4 自包含快照的生成规则

- **阶段 1 快照**：先填充 baseline 文件（提取锚点），然后据此生成阶段 1
  的自包含快照。阶段 1 快照 ≈ baseline 内容 + 阶段特有字段
- **阶段 N 快照**：以 baseline + 前一阶段快照为参照，产出完整的当前阶段
  状态。未变化的内容也必须包含在快照中——快照是自包含的
- **stage_delta**：记录从上一阶段的变化（信息性，便于理解演变弧线）
- **Baseline 的角色**：baseline 是提取时的参照锚点，帮助判断"相比初始状态
  有什么变化"。Baseline 不在运行时加载

### 9.5 信息来源标注

提取产出的所有结构化数据必须标注信息来源：

- `canon`：原文明确描述
- `inference`：基于原文的合理推断
- `ambiguous`：原文存在多义性，当前解读可能有误

当来源为 inference 或 ambiguous 时，必须附带说明推断依据或多义性所在。

### 9.6 提取产出物清单

每个 batch 可能产出或更新的文件：

**世界层**：
- `world/stage_catalog.json` — 追加新阶段条目
- `world/stage_snapshots/{stage_id}.json` — 当前阶段的世界快照
- `world/foundation/` — 基础设定修正
- `world/social/stage_relationships/{stage_id}.json` — 动态关系
- 按需：events、locations、factions、cast

**角色层（每个活跃角色）**：
- Baseline 文件（batch 1 时创建，后续仅在必要时修订）：
  identity.json、voice_rules.json、behavior_rules.json、boundaries.json、
  failure_modes.json
- 阶段文件（每 batch 产出）：
  stage_catalog.json、stage_snapshots/{stage_id}.json、
  memory_timeline/{stage_id}.jsonl

## 十、稀释保护

长时间运行的 AI 会话（无论是提取阶段还是运行时扮演阶段）都面临"稀释"
风险——随着上下文增长，AI 对早期加载的指令、数据和对话内容的注意力递减，
导致行为偏离预期。系统必须在架构层面提供防稀释机制。

### 10.1 运行时扮演的稀释保护

角色扮演 AI 在长对话中面临两类稀释：

**角色信息稀释**：启动时加载的角色数据随对话推进被推离注意力焦点，导致：

- 语气漂移：角色特有的说话方式逐渐退化为泛化 AI 口吻
- 关系姿态漂移：对对话者的态度从精确定位退化为通用友好
- 知识边界泄漏：角色开始给出超出当前阶段知识范围的信息
- 情绪状态重置：对话前段因事件产生的情绪在后段被遗忘

**对话内容稀释**：对话早期的事件和承诺被推出有效注意窗口，导致：

- 承诺/约定遗忘
- 已揭露的信息未被后续对话延续
- 情绪弧线断裂
- 用户提到的具体细节被遗忘

系统应提供以下防稀释能力：

1. **角色锚点重注入**：每轮或每 N 轮向 AI 重新注入压缩的角色核心信息
   （语气签名、硬边界、当前关系姿态、知识边界、情绪状态），防止角色漂移
2. **滚动对话状态**：定期生成对话内容的滚动摘要（情绪轨迹、已揭露信息、
   承诺/约定、关系微变化、未解决话题），确保早期内容不丢失
3. **深度校准检查点**：在重大事件后或每 15-20 轮触发深度校准，重读关键
   角色字段，检测并纠正漂移

### 10.2 提取阶段的稀释保护

多批次提取跨越长时间和多次 AI 会话，提取 AI 面临以下稀释风险：

**指令与规范遗忘**：

- 忘记 schema 的具体字段要求（特别是 stage_snapshot 的复杂结构）
- 忘记自包含快照模型的核心规则（每个快照必须完整，不可省略未变化字段）
- 忘记证据引用格式、source_type 标注要求等约定
- 忘记 baseline 是提取锚点而非运行时加载件

**批次间一致性断裂**：

- 后续 batch 产出的 stage_snapshot 与前面 batch 风格不一致
- 对同一角色的性格描述在不同 batch 间出现矛盾
- 之前标记为"不确定"的点被重新判断但结论不同
- 术语和命名在 batch 间发生漂移

**输出质量递减**：

- stage_snapshot 字段填充从详尽变为粗糙
- evidence_refs 引用越来越少
- source_type 标注越来越随意
- emotional_voice_map、relationship 等复杂字段被敷衍填充

系统应提供以下防稀释能力：

1. **Schema 强制重读**：每个 batch 开始写入前，必须重读目标 schema
   （而非仅回看 prompt 段落），确保字段要求仍在注意力范围内
2. **架构模型重申**：每个 batch 开始前，必须重读自包含快照模型的核心规则
   （baseline_merge.md 或等效说明），防止退化为增量模式
3. **前一 batch 输出校验**：新 batch 开始时，必须加载前一阶段的
   stage_snapshot 作为参照基准，确保风格和详细度一致
4. **输出质量自检**：每个 batch 写入完成后，对照 schema 检查必填字段是否
   齐全、复杂字段是否与前一 batch 同等详细
5. **跨会话连续性**：当提取跨越多个对话 session 时，新 session 必须通过
   progress 文件和前一 batch 输出恢复上下文，而不是仅依赖交接摘要

## 十一、自动化提取编排

### 11.1 总体架构

系统提供自动化编排脚本（`automation/`），通过 CLI 调用（`claude -p` 或
`codex`）驱动多批次提取。编排脚本负责：

- 批次间的状态传递（通过文件系统，不依赖 session 内存）
- 质量检查（两层：程序化校验 + 语义审校）
- 自动回滚与重试
- 断点恢复
- Git 安全操作

### 11.2 Pipeline 流程

1. **分析阶段**：LLM agent 执行全书分析、分批规划、候选角色识别
2. **用户确认**：用户选择角色、确认批次规划（由分析阶段按剧情边界生成）、设定提取范围
3. **提取循环**：对每个 batch 执行：
   a. Git preflight check（工作区干净、分支正确）
   b. 构建上下文感知 prompt（含文件清单、前批参照、schema 引用）
   c. 运行协同提取 agent（世界 + 角色同步产出）
   d. 程序化校验（JSON schema 合规、必填字段、文件数量一致性）
   e. 语义审校（独立 agent 检查质量、一致性、边界正确性）
   f. 通过 → git commit；失败 → rollback + 注入反馈重试

### 11.3 Agent 上下文模型

- 每个 batch 是一个全新的 `claude -p` 调用，完全无共享内存
- 批次间上下文完全靠文件系统传递
- 编排脚本预先为每个 batch 组装 prompt，明确列出该读哪些文件
- 这是优势而非缺陷：每个 agent 从干净 context 启动，无稀释风险

### 11.4 两层质量检查

**第一层：程序化校验（不花 token）**

- JSON schema 合规性
- 必填字段存在性
- stage_catalog 条目数 = stage_snapshots 文件数
- memory_timeline 文件存在性
- evidence_refs 和 source_type 非空
- 世界-角色 stage_id 交叉一致性

**第二层：语义审校（独立 LLM agent）**

- 与前一 batch 的风格/详细度一致性
- 数据边界正确性（世界层 vs 角色层）
- 信息充分性（关键字段的条目数和深度）
- 时间性正确（当前 vs 历史）
- 知识边界合理性

### 11.5 失败处理

- 程序化校验失败 → 自动回滚 + 重试（带错误反馈）
- 语义审校失败 → 自动回滚 + 注入 reviewer 反馈后重试
- 最大重试次数 2 次，超过停下等用户介入
- Rate limit → 自动等待后重试
- 脚本崩溃 → 重启后从 progress 文件恢复

### 11.6 Git 集成

- 每个 batch 通过审校后自动 git commit
- 回滚 = git reset 到上一个 committed batch
- 提取在独立分支（`extraction/{work_id}`）进行，完成后合并回主分支
- 每个 batch 开始前执行 git preflight check

### 11.7 LLM 后端抽象

编排脚本通过统一接口支持多种 LLM 后端：

- Claude Code CLI（`claude -p`，使用 Claude subscription）
- Codex CLI（使用 Codex subscription）
- 可混合使用（如 Claude 提取 + Codex 审校）

### 11.8 Batch boundary 规则

分析阶段的分批规划应遵循：

- 尽量贴近自然剧情边界
- 最小批次 5 章，最大批次 20 章，默认 10 章
- 为每个阶段取有意义的中文名称（如"阶段1_南林初遇"）
- stage_id 在世界包和角色包间必须一致
