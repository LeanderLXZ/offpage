# 项目需求文档

本文档是用户对 persona-engine 的核心需求规范。所有架构设计、Schema 定义、
提取流程和运行时加载策略都应以本文档为最终权威。

---

## 一、总体目标

建立一个长期稳定的小说角色扮演系统，能够：

- 从长篇小说中提取角色的"灵魂"
- 支持任意指定角色的深度扮演
- 让角色在不同剧情阶段有不同的状态、性格和表现
- 支持长期对话和记忆积累
- 未来可接入多种终端（AI agent、前端 app、移动端 MCP）

核心原则：**深度扮演，而非表面模仿。**

---

## 二、剧情阶段模型

### 2.1 基本规则

- **阶段规划是分析阶段最核心的产出**——阶段边界决定世界快照、角色快照、
  记忆时间线、运行时阶段选择的切分方式。切分质量直接决定角色体验的连贯性。
- 阶段按自然剧情边界切分，边界准确性优先于章节数量均匀，
  每个阶段的章节数可以不同
  （默认目标 10 章，最小 5 章，最大 15 章，可在作品 config 中调整）
- 阶段 N 是累积的：选择阶段 N 意味着提取阶段 1..N 的全部内容

### 2.2 时间性原则

- 当前所选阶段是角色的"现在"
- 之前的阶段是按顺序发生的历史事件
- 角色能记住发生过的事情，但必须用**当前阶段的状态、性格和口吻**与用户对话
- 世界状态同理——只有当前阶段的快照反映"此刻"的世界

### 2.3 世界信息的架构

#### 2.3.1 世界基础设定层

每部作品应有一组世界基础设定文件，描述不随剧情推进而大幅变化的底层规则：

- **世界观设定**：世界的基本架构（如仙侠、科幻、现代等）
- **力量体系**：修炼等级、魔法系统、科技水平等
- **宇宙观/天道规则**：世界运行的底层逻辑

基础设定可能在后续阶段被补充或修正（如新发现推翻早期认知），修正应记录在
世界快照的 `foundation_corrections` 中。

#### 2.3.2 世界实体追踪

系统应追踪以下世界级实体：

- **主要事件**：影响整个世界或多个角色的重大事件。世界层只记录大事件，
  小事件和个人经历归角色层
- **地点**：重要的场景、地域、建筑。地点可能随剧情变化（毁灭、新建等）
- **势力/阵营**：组织、门派、国家等。势力可能合并、分裂、消亡
- **主要角色索引**：主角团和高频配角的基本信息索引。世界层只收主要角色，
  不收大量龙套和一次性小角色。详细角色信息归角色包

#### 2.3.3 世界级关系

世界层的关系分为两类：

- **固定关系**：不随阶段变化的结构性纽带（如父子、兄妹、血亲、师门传承），
  始终有效。存储在 `world/foundation/fixed_relationships.json`，
  Phase 2.5 产出骨架，后续 stage 维护修正
- **动态关系**：随剧情演化的关系状态（如恋人、盟友、仇敌），
  按阶段记录在 `world/stage_snapshots/{stage_id}.json` 的
  `relationship_shifts` 中

#### 2.3.4 世界阶段快照

每个世界阶段快照应包含：

- 对基础设定的修正与补充
- 本阶段事件（`stage_events`，≤ 10 条，仅本阶段发生的**世界级事件**，
  每条 50–80 字的一句话简述；供 `world_event_digest.jsonl` 程序化累积）。
  **仅记录世界
  公共层事件**（如势力变迁、地形/资源变化、规则揭示），角色个人剧情事件由
  对应角色的 `stage_events` 与 `memory_timeline` 承载，不进入此处
- 当前世界状态
- 关注的人物关系的转变
- 人物的状态（生死、恋爱、等级等可能随时间变化的状态）
- 地点的变化
- 地图的变化（势力范围、领土变动等）
- 引用的章节号列表（`evidence_refs`）

注意：世界快照只记录**本阶段**的事件，不累积历史。跨阶段的世界事件
时间线由 `world/world_event_digest.jsonl` 程序化累积维护（见 §11.3a）。
`stage_events` 每条 50–80 字（schema 两端硬门控），是快照中的事件清单，
也是 digest 条目的直接来源（唯一事件清单）。
快照正文字段（stage_events、current_world_state 等）不需要逐条标注
章节引用 `[NNNN]`，底部 `evidence_refs` 字段提供本阶段引用章节汇总

世界阶段快照与角色阶段快照使用相同的 `stage_id`，确保世界状态与角色状态
在同一时间线上对齐。运行时加载某个阶段时，同时加载该阶段的世界快照和角色
快照。

### 2.4 角色信息的剧情阶段

每个角色阶段快照应包含：

- 人物从上个阶段到此已经经历的事件
- 目前状态（生死、恋爱、等级等可能随时间变化的状态）
- 目前的性格、口吻发生的转变
- 心情
- 一切可能随着时间变化的东西

### 2.5 角色间关系的阶段演化

角色的性格、口癖、行事作风，以及对其他特定角色的态度，都会随剧情阶段不断
变化。这些变化由角色之间共同经历的事件、记忆、隐瞒是否被揭露等因素驱动。
角色之间也可能相互影响对方对世界的认知和性格。

系统必须支持以下能力：

- **对特定角色的态度随阶段变化**：同一角色在不同阶段对同一对象可能从敌意
  转为亲密，再从亲密转为悲痛。阶段快照必须精确记录当前阶段对每个重要角色
  的态度、信任度、亲密度。
- **事件驱动的关系转变**：态度变化不是凭空发生的，必须关联到具体事件
  （如共同经历、误会澄清、背叛、牺牲、死亡与复活等）。
- **知识边界影响行为**：角色在某个阶段可能不知道某件事（如对方的真实身份、
  对方是否已死亡），这会直接影响角色的反应。当用户以某个身份出现时，角色
  应基于当前阶段的知识状态做出真实反应（如惊讶、质疑、感动）。
- **隐瞒与揭露的阶段追踪**：角色对某些信息的隐瞒状态可能在某个阶段被揭露，
  揭露前后角色的行为和态度应有明显差异。
- **相互影响**：角色之间可能通过互动改变彼此的世界观、性格和行为模式。
  这些影响应该被记录，并在对应阶段的快照中体现。

**示例**：

1. 阶段 A：男女主初遇，相互抱有敌意
2. 阶段 B：经历若干事件后态度逐渐缓和
3. 阶段 C：相爱，相互许诺
4. 阶段 D：男主死亡，女主悲痛欲绝——此时若用户以男主身份对话，女主应惊讶
   反问"你不是死了吗？"，确认复活后感动流泪
5. 阶段 E：男主复活，女主难以置信但极度感动，重新开心

系统加载任意阶段时，AI 必须能完整复刻该阶段的世界观、性格、对每个角色的
态度和语气，做出符合该阶段的真实反应。

### 2.6 跨阶段历史回忆

角色在当前阶段对话时，必须能够回忆过去阶段的具体行为细节，而不仅仅是
事件摘要。这些细节包括但不限于：

- 过去阶段对特定角色使用的称呼和口癖
- 过去阶段对特定角色的态度和行为模式
- 过去阶段的知识边界（当时知道什么、不知道什么）
- 过去阶段隐瞒或被隐瞒的信息
- 过去阶段关系状态的具体原因

角色在回忆时，必须使用**当前阶段的性格和语气**来谈论过去，而不是切换回
过去的性格。回忆应体现角色对过去经历的**当前认知和态度**。

**示例**：

- 阶段 1：角色 A 和角色 B 是对手关系，A 用不友好的绰号称呼 B，B 隐瞒了
  自己的真实身份，A 不知道 B 的真实背景。
- 阶段 10：A 和 B 已经和解，A 性格仍然含蓄，且已知道 B 的真实身份。
- 加载阶段 10 后，用户（B）问："你以前叫我什么？"
- A 应当能回答（以阶段 10 的语气）："我当时那么叫你，是因为咱们那个时候
  还是对手嘛。我又不知道你的真实身份。"

这要求系统在以下层面提供足够的历史细节：

1. **记忆条目**：memory_timeline 中的记忆不能只是事件摘要，还应记录当时
   的行为细节（称呼、口癖、态度、知识状态等）。
2. **关系演变概述**：relationship_history_summary 应包含关系演变中的关键
   行为标记（称呼变化、态度转折点、知识揭露时点等）。
3. **按需深度加载**：当以上信息不足时，系统应支持按需加载过去阶段的快照
   以获取更完整的历史细节。

### 2.7 阶段选择

- 每个角色的每个剧情阶段，需要有一个一句话总结，用于用户选择
- 用户在开始对话时选择一个阶段，该阶段在整个对话上下文中固定

## 三、深度扮演的三大目标

### 目标 1：系统化的角色资料结构

将小说中与角色相关的信息，从原始剧情中提炼出来，并整理为结构化资料。
这些资料至少应包括：

- 角色基础身份信息
- 性格特征与内在驱动力
- 情绪触发点与行为模式
- 关键关系网络
- 重要经历与时间线记忆
- 语言风格与典型台词特征
- 禁止偏离的人设边界
- 容易写崩或误判的点

### 目标 2：建立角色视角的记忆体系

不是只总结剧情，而是要把剧情转化为"角色本人记得的内容"。
需要从角色的视角梳理：

- 角色经历过哪些关键事件
- 角色在不同阶段知道什么、不知道什么
- 角色误会过什么、隐瞒过什么
- 角色对其他角色的态度是如何变化的
- 角色在情感上的变化轨迹是什么

目标是形成一份可供角色扮演直接调用的"角色记忆库"或"记忆时间线"。

### 目标 3：建立稳定的语言与行为模拟规则

AI 不仅要"知道角色是谁"，还要"说得像角色、反应得像角色"。
需要提炼：

- 角色常用的语气
- 角色偏好的表达方式
- 角色在不同情绪下的语言变化
- 角色面对不同对象（亲近者、陌生人、敌对者）时的说话差异
- 角色在暧昧、愤怒、委屈、嘴硬、关心、吃醋等状态下的典型反应模式

目标是总结出一套可以直接指导角色扮演的语言风格规则和行为规则。

### 角色标识与名称跟踪

角色在小说中的名称和身份不是固定不变的。系统必须能够跟踪和管理以下场景
下的角色标识变化，确保无论角色以何种名称出现，数据始终归属到同一个角色
实体：

#### 场景 1：伪装 / 失忆 / 化名

角色因剧情原因（伪装潜入、失忆、故意隐瞒身份等）以另一个名称和身份活动
了一段时间。

- 系统需要知道化名 X 实际上是角色 Y
- 化名期间的事件、记忆、关系变化仍归入角色 Y 的时间线
- 角色视角的记忆中应反映"我当时以 X 的身份行事"
- 其他角色的认知中可能只知道 X，不知道 Y——这属于知识边界问题

#### 场景 2：后期获得名称

角色前期以代称存在（如"小龙""那把剑"），后期被正式命名。

- 前期的数据不应丢失，应在命名后自动关联
- 提取过程中前期可用代称作为临时标识，命名后更新为正式名称，并保留代称
  作为别名

#### 场景 3：名称变更

角色因拜师、封号、转世等原因更换了名称（如本名→道号、前世名→今世名）。

- 所有名称都应记录，并标注各名称的生效阶段
- 角色的 character_id 应保持稳定（选择最通用或最终的名称），其他名称作为
  别名
- 不同阶段的快照中应反映当时的"当前名称"

#### 场景 4：双重身份 / 双名并行

角色同时拥有两个名称被交替使用（如武器名和人形名、真名和封号）。

- 两个名称应指向同一个角色实体
- 在不同语境中使用不同名称是正常的，系统不应将其视为两个角色
- 应记录哪些场景下使用哪个名称（如战斗时用武器名、日常用人名）

#### 场景 5：称呼与昵称

角色被不同人以不同方式称呼（如本名、绰号、亲昵称呼、敌对称呼）。

- 这些称呼应记录在角色的别名列表中，并标注来源关系（"XX对该角色的称呼"）
- 运行时扮演中，不同角色提及目标角色时应使用符合关系的称呼
- 称呼可能随阶段变化（如从敌对称呼变为亲昵称呼），应按阶段记录

#### 分析阶段的身份合并

全书分析（归纳 + 剧情阶段划分 + 候选角色识别）阶段也必须处理角色身份一致
性问题。章节归纳以分组（chunk）为单位独立进行，每个 chunk 无法看到其他
chunk 的内容，因此同一角色可能在不同 chunk 中以不同名称出现。

要求：
- 章节归纳阶段：当 chunk 内出现疑似已知角色的新名称时，应在摘要中标注
  可能的关联（如"角色B疑似即角色A"），但不强制合并（因为单个 chunk 信息
  不充分）
- 全书分析阶段（读取所有 chunk 摘要后）：**必须** 执行跨 chunk 的角色身份
  合并，将同一角色在不同阶段的不同名称统一到一个候选角色条目中。候选角色
  列表中每个角色应包含已知的所有别名，并标注各别名的首次出现位置
- 合并判断依据：同一叙事位置的角色特征一致、前后文暗示身份关联、角色行为
  模式延续、其他角色的反应一致等
- 不确定的合并应标注为推测，留待提取阶段用原文验证

示例：角色代称A（武器形态代称，前中期使用）→ 角色正式名（正式名，后期获知）→
角色昵称 / 简称（昵称，主角所取）应合并为同一候选角色"角色正式名"，并记录所有
别名。

#### 通用要求

- 每个角色有一个稳定的 `character_id`，在整个作品生命周期内不变
- 除 `character_id` 外，维护一个 **别名列表**（aliases），包含所有曾用名、
  化名、称呼、代称
- 别名应标注：名称文本、名称类型（本名 / 化名 / 代称 / 称呼 / 昵称 /
  绰号 / 封号 / 道号 / 武器名 / 其他）、生效阶段范围、来源（谁这样称呼 /
  为什么使用这个名称）。所有阶段（分析、baseline、提取）统一使用此类型枚举
- 提取过程中，当新名称出现时，应优先匹配已有角色的别名列表，避免重复建档
- 发现新别名时需 **双写**：更新 `identity.json` 的 `aliases`（追加新条目）
  并更新当前 `stage_snapshot` 的 `active_aliases`
- 提取 agent 的文件读取顺序应确保 `identity.json`（含已知别名）在原文章节
  之前被读取，以便阅读原文时进行别名交叉比对
- 运行时加载时，角色的别名列表应随阶段快照一起提供，供对话生成和角色识别
  使用

## 四、运行时加载需求

当 AI 进行角色代入，加载某个角色的某个剧情阶段时，需要能迅速代入以下六类信息：

### A. 基本人设

- 姓名
- 年龄
- 身份
- 外貌特征
- 社会地位
- 重要关系

### B. 性格核心

- 表层性格
- 内核性格
- 优点
- 缺点
- 情绪触发点
- 逆鳞 / 底线
- 安全感来源
- 常见防御机制

### C. 说话风格

- 语气
- 用词习惯
- 是否毒舌 / 傲娇 / 冷淡 / 温柔 / 高压
- 是否爱反问
- 是否会直接表达感情
- 在不同对象面前说话的区别

### D. 记忆库

- 经历过的重大事件
- 知道的秘密
- 误会过什么
- 失去过什么
- 和其他角色的重要互动时间线

### E. 行为规则

- 遇到挑衅时会怎样反应
- 遇到暧昧时会怎样掩饰
- 生气时怎么说话
- 吃醋时怎么说话
- 受伤时会不会嘴硬
- 会不会主动示弱

### F. 禁止偏离项

- 绝不会做什么
- 绝不会说什么
- 容易被误写崩的点有哪些

### 运行时行为规则

除了加载数据外，运行时还需要以下行为规则指导 AI 的实时反应：

- **认知冲突处理**：当角色的已有认知与对话中出现的现实冲突时（如角色认为
  某人已死但该人出现），AI 应基于角色当前的知识状态、性格和关系做出真实反应
  （惊讶→质疑→确认→情绪爆发），而非直接接受新信息。这些行为不预设场景，
  而是由数据（知识边界、关系、性格）+ 规则（处理步骤）实时推理
- **历史回忆处理**：角色回忆过去时，优先使用已加载的记忆条目和关系演变概述；
  不足时按需加载过去阶段快照；始终用当前阶段人格表达；禁止编造历史、混淆
  阶段、泄漏角色当时不知道的信息

### 5.1 新用户创建流程

1. 输入用户 ID
2. 选择作品（从已入库作品中选择）
3. 选择要模拟的目标角色
4. 选择目标角色的剧情阶段
5. 选择自身角色：
   - 可以选择作品中已有的角色（characters/ 中已建好的）
   - 可以选择新建自定义角色（保留入口，后续细化）
   - 自身角色的剧情阶段与目标角色保持一致
6. 创建用户包：
   - 将所选角色的基础信息、剧情阶段、当前状态写入用户信息
   - 创建用于模拟角色修正、用户事件记忆等的目录结构
7. 创建新 context，加载对应剧情阶段的世界信息、目标角色信息、用户信息

### 5.2 已有用户登录流程

1. 输入已有用户 ID
2. 显示当前账户信息
3. 检查是否有可恢复的 context：
   - 如果有：列出可选 context 让用户选择
   - 如果没有：提示用户并创建新 context
4. 加载对应剧情阶段的世界信息、目标角色信息、用户信息

### 5.3 关键约束

**一次性设定锁定**：每个用户只能在初始创建时进行设定选择（作品、目标角色、
自身角色、剧情阶段）。创建完成后设定锁定，无法更改。如需更改必须创建新用户
包或执行显式迁移。

**会话关闭与合并**：当用户说出结束对话的关键词时：

1. 退出当前对话
2. 询问用户是否将当前对话内容并入用户长期历史
3. 如果用户同意合并：
   - 将 context 中积累的角色情绪/性格变化、事件、记忆点提炼写入用户的
     长期 profile（`long_term_profile.json`）和关系核心
     （`relationship_core/`）
   - 事件和记忆点是**只增添不覆盖**的
   - context 状态变为 `merged`
4. 如果用户拒绝合并：
   - context 保持原状，不影响长期 profile

**实时更新 vs 合并更新**：

- Context 内的内容（对话记录、角色实时状态、事件、记忆点）是**实时更新**的
- 用户的长期 profile 和关系核心**只在合并确认后才更新**
- 这两层严格分开：context 是临时工作区，长期 profile 是持久化的人格积累

**Context 内实时追踪的状态**：

对话进行中，context 应持续追踪以下增量状态：

- 角色情绪变化
- 性格微漂移（因对话事件产生的临时性格变化）
- 语气微调（因对话情境产生的临时语气变化）
- 双方达成的约定/承诺
- 关系增量（态度、信任度、亲密度的微调）
- context 内发生的事件
- context 内产生的新记忆

这些状态在 context 内实时可用，但只在合并时才被筛选性地提升为长期状态。

**崩溃恢复**：

系统应支持对话中断后的恢复能力：

- 每轮对话应有日志记录机制，确保用户输入不会因中断而丢失
- 中断后恢复时，应能检测到未完成的轮次并安全恢复

**对话归档**：

合并确认后的对话记录应归档为不可变的账户级历史记录：

- 归档内容包含：对话摘要、关键时刻、完整会话记录
- 归档记录是不可变的历史——合并后不再修改
- 运行时启动时仅加载归档摘要和索引，完整记录按需加载

## 六、数据分离原则

- 客观剧情事实 vs 角色主观认知——必须分开
- 原著正典 vs 合理推理——必须标注区分
- 角色 canon 数据 vs 用户对话产生的数据——用户数据不得污染角色 canon
- 不同角色的知识边界——角色不能知道它不该知道的事
- 不同阶段的状态——角色不能表现出未来阶段才有的特征
- 不同作品的命名空间——每部作品是独立的命名空间，所有运行时请求和持久化
  文件必须显式携带 `work_id`，不能仅依赖目录路径来区分作品

**内容语言一致性**：作品的语言设定（由 manifest 中的 `language` 字段声明）
应作为该作品下所有生成内容的默认语言。这不仅适用于作品级 canon（世界包、
角色包），也适用于用户侧与该作品相关的数据（关系描述、长期 profile 中的
作品相关内容、context 内的角色状态描述等）。

### 7.1 信息分层

系统不是一个巨大的 prompt，而是分层组织的：

1. **不变层**：identity、hard_boundaries、failure_modes——不随阶段变化，
   运行时始终加载。为避免 identity 中 `aliases[].evidence_refs` 等
   章节引用列表线性膨胀 prompt，**loader 层在加载时按字段白名单过滤**：
   `evidence_refs` 等追溯字段仅用于提取期追溯，运行时不注入 prompt。
   schema 不改，**无需重跑 Phase 2 / 2.5**，仅调整 loader 的序列化逻辑即可
2. **自包含阶段快照**：stage_snapshot 是角色扮演的 **唯一运行时状态来源**
   （配合不变层）。它必须包含该阶段完整的语气、行为模式、边界、关系、
   性格、心情、知识状态、活跃别名——运行时直接加载，不需要也不应该与
   baseline 合并。**快照的完整度直接决定扮演质量**：如果快照中缺少某个
   维度的信息（例如缺少情绪语气矩阵、缺少对某个重要角色的关系描述），
   扮演 AI 就无法获得该信息，导致角色表现失真
3. **历史记忆**：memory_timeline 按阶段拆分为独立文件（每个阶段一个
   `{stage_id}.json`）。启动时加载近期 2 个阶段全量 + `memory_digest.jsonl`
   （stage 1 到用户选定 stage N 过滤加载）+ `world_event_digest.jsonl`
   （同范围过滤加载）；其余条目通过 FTS5 按需检索。
   过去阶段的完整快照可按需加载以支持深度历史回忆
4. **对话可变层**：context 内的角色状态（情绪变化、性格微漂移、语气微调、
   关系增量）每轮实时更新
5. **跨对话积累层**：long_term_profile、relationship_core——仅在会话关闭
   并确认合并后更新

Baseline 文件（voice_rules、behavior_rules、boundaries）仍然存在，
但仅作为提取时的参照锚点，**不在运行时加载**。

### 7.2 分层加载

运行时应按需分层加载，而非全量装入：

- **启动核心层**：进入对话前必须加载的最小集——角色不变层、当前阶段快照、
  历史记忆（近期全量 + digest 按 stage 过滤）、世界快照、世界固定关系、
  世界事件摘要、最近 M 个 scene_archive `full_text`（M 由
  `scene_fulltext_window` 控制，默认 10；**summary 字段不在启动加载**，
  仅作为 FTS5 / embedding 检索的索引字段）、用户状态摘要。
  启动层只加载轻量聚合，不加载完整对话记录
- **结构化按需层**：对话中按需加载——世界事件详情、地点/势力信息、过去阶段
  快照、旧 context 摘要、旧会话摘要等
- **对话记录层**：仅在需要精确回忆时加载——完整对话记录、逐轮摘要
- **原文验证层**：仅在摘要冲突或需要高精度引用时加载——原始章节正文

每部作品可通过配置文件自定义各层的加载范围。

## 八、作品入库与规范化

### 8.1 支持的原始格式

系统应能接受以下格式的原始小说文件：

- **TXT**：纯文本文件，可能有不同编码（UTF-8、GBK 等）
- **EPUB**：电子书格式，需要解包提取正文
- **MOBI**：Kindle 格式，需要转换后提取正文
- **HTML**：网页抓取的章节，需要去除标签和格式
- **用户手动提供的片段**：部分章节或摘录

### 8.2 规范化流程

原始文件入库后必须经过规范化处理，产出可供后续提取使用的标准化文本：

1. **原始保存**：原始文件不做任何修改，保存在 `sources/works/{work_id}/raw/`
2. **格式转换与清洗**：
   - 统一编码为 UTF-8
   - 去除格式伪影（HTML 标签、排版控制符、广告水印等）
   - 修复明显的编码错误和乱码
   - 产出干净的纯文本，保存在 `normalized/`
3. **章节拆分**：
   - 按章节边界拆分为独立文件，保存在 `chapters/`
   - 每章一个文件，零填充编号（如 `0001.txt`）
   - 每个文件包含完整的一章正文
4. **元数据生成**：
   - `metadata/book_metadata.json`：书名、作者、章节数、来源信息
   - `metadata/chapter_index.json`：有序章节列表，含标题和文件路径

### 8.3 Source Package 边界

- Source package 只包含原始素材和规范化产物
- 分析、提取和规范化产出归属 `works/{work_id}/`，不属于 source package
- Source package 是输入层，下游流程不应修改它
- Source package 默认不纳入版本控制（通过 `.gitignore` 排除）

### 8.4 作品 Manifest

每本作品入库时必须创建 `sources/works/{work_id}/manifest.json`，至少包含：

- `work_id`：必须与目录名一致
- `title`：原语言标题
- `language`：ISO 639-1 语言代码（如 `"zh"`、`"en"`）
- `source_types`：原始格式标签数组（如 `["epub"]`）
- `ingestion_status`：入库生命周期状态，枚举 `"planned"` / `"raw_imported"` /
  `"normalized"` / `"chunked"` / `"indexed"` / `"active"`。权威定义见
  `schemas/work_manifest.schema.json`。
  - `planned`：仅完成占位，尚未导入原始素材
  - `raw_imported`：原始文件已落盘 `raw/`，尚未规范化
  - `normalized`：已转 UTF-8、去噪、拆章到 `normalized/chapters/`
  - `chunked`：已完成分块或 RAG 切片（如有）
  - `indexed`：索引、词表等检索资源已生成
  - `active`：可供下游 `works/{work_id}/` 抽取和运行时使用

中文作品的 `work_id` 应直接使用中文书名。

#### 流程详细说明

从原始小说到可运行角色包的完整流程如下：

1. **作品入库**：原始文件规范化、章节拆分、元数据生成（见 §八）
2. **章节归纳**（Phase 0）：将全书按分组（chunk，约 20-25 章一组）逐组归纳，
   产出每章的结构化摘要（事件、出场角色、地点、情绪基调、身份变化线索、
   候选阶段边界标记）。每个 chunk 独立处理，多 chunk 并行执行
   （`--concurrency` 控制，默认 10）
3. **全书分析**（Phase 1）：基于所有章节摘要，执行以下分析任务（按顺序）：
   a. **跨 chunk 角色身份合并**：识别不同 chunk 中以不同名称出现的同一角色，
      建立统一的身份映射
   b. **世界观概览**：分析作品的世界观基础信息，包括题材类型、力量体系、
      主要势力/阵营、地理结构、大的时代/世界线划分、核心设定规则。产出
      世界观概览文件，供后续提取阶段作为背景认知
   c. **源文件阶段规划**：制定 stage plan，按自然剧情边界切分（默认目标
      10 章，最小 5 章，最大 15 章），每个 stage 的章节数可因剧情节点而
      不同。世界线的大转折、时代切换等应作为重要的阶段边界依据
   d. **候选角色识别**：基于身份合并后的角色出场信息，识别可建包的候选
      角色。同一角色的不同名称必须合并为一个候选条目
4. **活跃角色确认**（Phase 2）：用户从候选中选择要建包的目标角色
5. **Baseline 产出**（Phase 2.5）：基于全书摘要和确认的角色，产出世界
   foundation 和角色 identity/manifest 初稿
6. **协同阶段提取**（Phase 3）：每个 stage 读原文，世界提取与角色提取
   1+N 全并行产出。每个 stage 经过 JSON 修复 → 程序校验 → 语义审核 → git 提交。
   每个 stage 都可修正和补充 baseline（不限第一个 stage）
7. **跨阶段一致性检查**（Phase 3.5）：Phase 3 全部 stage 提交后，运行程序化
   跨阶段一致性检查（零 token），对标记的问题可选 LLM 裁定。详见 §11.10
8. **场景切分**（Phase 4）：逐章读原文，LLM 标注场景边界和元数据（起止行号、
   时间、地点、在场角色、摘要），程序根据行号从原文拼接 `full_text`。
   前置条件仅需 `stage_plan.json` 存在（Phase 1 产物），与 Phase 3
   独立，可并行执行。每章一次 LLM 调用，多章并行（`--concurrency` 控制，
   默认 10）。仅程序化校验（行号有效、场景不重叠、覆盖全章、alias 匹配），
   不做语义审校。详见 §11.11
9. **针对性补充提取**：协同阶段完成后，对明显缺口进行定向补充
10. **包验证与发布**：验证产出物的完整性和一致性

#### 世界观概览的内容要求

全书分析阶段产出的世界观概览应包含：

- **题材与基调**：作品的题材类型（如玄幻、科幻、都市等）和整体基调
- **世界结构**：基本的世界地理/空间结构、主要区域划分
- **力量体系**：修炼/能力体系的基本框架和等级划分（如有）
- **主要势力**：主要组织、阵营、国家/门派的基本格局
- **大世界线**：故事的大时代/篇章划分（如"天苍篇→仙界篇→蓝星篇"），
  每个大阶段的核心冲突和环境特征
- **核心设定规则**：影响剧情走向的关键世界规则（如转世机制、天道规则等）

世界观概览不需要穷尽细节，其目的是为后续逐 stage 提取提供全局背景认知，
避免提取 agent 在缺乏宏观视野的情况下工作。

#### 全书分析阶段的 Baseline 产出

全书分析阶段拥有全书视野和角色身份合并结果，是产出 **非阶段性 baseline
信息** 的最佳时机。以下内容应在全书分析阶段产出初稿，而非等到第一个 stage
提取时才从零开始：

**世界层 baseline**（写入 `world/foundation/`）：
- `foundation.json`：力量体系基本框架、世界地理/空间结构、主要势力/阵营
  格局、核心设定规则、大世界线/篇章划分
- `fixed_relationships.json`：世界视角的固定关系网络（血缘、师承、门派
  归属等不随阶段变化的结构性纽带），骨架初稿由 Phase 2.5 产出，后续
  stage 维护修正

**角色层 baseline**（为每个已确认的目标角色写入）：
- `identity.json`：canonical_name、完整的结构化 aliases（从身份合并直接
  转化）、gender、species、birth_origin、appearance_summary、
  background_summary、initial_social_position、affiliations、
  distinguishing_features、**core_wounds**（核心创伤——跨全故事的根源性
  心理创伤，含来源和行为影响）、**key_relationships**（核心人物关系——
  含初始关系、关系演变弧线、转折点）
- `manifest.json`：角色清单索引

**角色层 baseline 骨架**（全书视野初稿）：
- `voice_rules.json`：基线语言风格（跨阶段稳定的本性风格，阶段性语气
  变化由 stage_snapshot 覆盖）
- `behavior_rules.json`：基线行为模式（跨阶段稳定的本性行为）
- `boundaries.json`：角色底线禁忌
- `failure_modes.json`：角色易崩模式

这些文件记录的是**不随阶段变化的角色基底**，与 `identity.json` 和
`foundation.json` 定位一致。阶段性变化（语气转变、行为偏移、情感波动等）
由 `stage_snapshots` 覆盖。后续 stage 提取读到原文细节后可修正和补充。

**不在全书分析阶段产出的内容**（阶段性内容）：
- 阶段快照、记忆时间线等阶段性内容

**修正机制**：全书分析阶段的 baseline 是基于摘要的初稿。逐 stage 提取
阶段读到原文后，如果发现初稿有误或遗漏，应直接修正（而非忽略）。修正时
应在提取产出中注明"修正自全书分析初稿"。

### 9.3 1+N 并行提取模式

每个 stage 采用 **1+N** 并行提取架构，世界提取与角色提取拆分为独立调用，
**同一 stage 内全部 1+N 次调用并行执行**：

**世界提取（1 次调用）**
- 读取原文 + 上一个 world stage_snapshot（如有）+ world foundation + schemas
- 产出世界快照、基础设定（foundation / fixed_relationships）修正等

**角色提取（N 次调用，与世界提取并行）**
- 每个目标角色独立调用，读取：
  - 原文（本 stage 章节）
  - 角色自身全部 baseline（identity / voice_rules / behavior_rules / boundaries / failure_modes / manifest）
  - 角色最近一个 stage_snapshot（如有，用于 delta 计算 + 风格参照 + 未出场角色继承）
  - 角色最近一个 memory_timeline（如有，用于记忆连续性）
  - schemas
- **不读取世界快照**——角色和世界读同样的原文，客观事实一致性由审校阶段的
  跨通道检查保证（见 §11.5），不需要先等世界提取完成
- 产出角色快照、记忆时间线
- **每个 stage 都可修正和补充 baseline**（不限第一个 stage）——读到原文细节后
  修正此前的推测，或补充新发现的规则
- **不读取也不写入** `memory_digest.jsonl`、`world_event_digest.jsonl`、`stage_catalog.json`、`simulation/contracts/baseline_merge.md`
- `memory_digest.jsonl`、`world_event_digest.jsonl` 和 `stage_catalog.json` 由程序在提取完成后自动维护（见 §11.3a）
- 自包含快照的语义契约直接内嵌在 extraction prompt 中，不依赖 simulation 文档

世界和角色间无执行依赖——角色不读世界快照，世界不读角色快照。
角色间的主观体验差异是设计特性（不同视角），无需跨角色强制对齐。
客观事实一致性由 commit gate 的跨通道检查保证。

### 9.4 自包含快照的生成规则

**核心原则：stage_snapshot 是运行时角色扮演的唯一状态来源（配合不变层），
其完整度直接决定扮演质量。** 运行时不会加载 baseline，也不会做任何合并。
如果快照中缺少某个维度的信息，AI 就无法获得该信息。

#### 快照完整性检查清单

每个 stage_snapshot 必须包含以下全部维度（即使某维度相比上一阶段无变化）：

- `active_aliases`：当前活跃名称、隐藏身份、各角色对本角色的称呼
- `voice_state`：语气基调、语言习惯、用词偏好、口头禅、禁忌用语、
  **情绪语气矩阵**（emotional_voice_map，≤ 10 种情绪，覆盖主要情绪类型）、
  **对象语气矩阵**（target_voice_map，≤ 5 个对象，只对主要角色和重要配角
  详细记录，使用具体角色名，每个 target 至少 3-5 条对话示例（≤ 5 条），
  覆盖该对象下不同情绪和场景；泛化类型如"陌生人"简要描述即可，LLM 可从
  整体性格推断）、
  典型对话示例（dialogue_examples，至少 2-3 条，≤ 5 条）
- `behavior_state`：**core_goals**（理性目标）、**obsessions**（执念——
  非理性的心结，区别于可权衡调整的目标）、决策风格、情绪触发器、
  **情绪反应矩阵**（emotional_reaction_map，≤ 10 种情绪）、
  **对象行为矩阵**（target_behavior_map，≤ 5 个对象，与 target_voice_map
  平行且对齐，只对主要角色和重要配角详细记录，描述面对该对象时的肢体反应、
  距离感、习惯性动作、回避模式等，每个 target 至少 3-5 条行为示例（≤ 5 条）；
  泛化类型简要描述即可）、
  习惯性行为、压力应对
- `boundary_state`：当前阶段有效的软边界、容易被误判的点
- `relationships`（≤ 10 条）：对每个重要角色的完整关系状态——态度、信任度、
  亲密度、戒备度、语气变化、行为变化、驱动事件、认知中的对方状态、
  从阶段 1 到当前的关系演变概述。不包含边缘的不重要角色
- `knowledge_scope`：角色知道什么、不知道什么、不确定什么。**条数上限**：
  `knows` ≤ 50、`does_not_know` ≤ 30、`uncertain` ≤ 30；**每条 ≤ 50 字**
  （schema 硬门控）。超限时按裁剪策略丢弃：优先保留 ① 影响当前阶段决策或
  角色扮演的条目、② 与 `core_wounds` / `active_obsessions` / 活跃
  `relationships` 相关的条目；优先丢弃 ① 日常常识类条目、② 早期阶段已无
  触发点的细节、③ 已在 `memory_timeline` 中完整承载的条目
- `misunderstandings`：角色持有的误解（主观 vs 客观）。**条数上限 ≤ 20**
  （schema 硬门控）。已 resolved 的误解（`resolved_at_stage` 非空且 ≤ 当前阶段）
  应移除，优先保留影响当前决策的条目
- `concealments`：角色主动隐瞒的事情。**条数上限 ≤ 20**（schema 硬门控）。
  已 revealed 的隐瞒（`revealed_at_stage` 非空且 ≤ 当前阶段）应移除，
  优先保留当前仍在隐瞒的条目
- `emotional_baseline`：主导特质、**active_goals**（活跃理性目标）、
  **active_obsessions**（活跃执念）、活跃恐惧、活跃创伤
- `current_personality`（≤ 10 条）、`current_mood`（≤ 10 条）、
  `current_status`（≤ 10 条）
- `stage_delta`：从上一阶段的变化摘要（信息性，便于理解演变弧线）
- `character_arc`：角色从阶段 1 到当前阶段的 **整体弧线概览**——
  弧线摘要、关键节点列表（每个节点含 stage_id 和描述）、
  当前在弧线中的位置和趋势。第一个阶段可省略或仅写起点

#### 字段条数上限汇总（schema 硬门控）

| 字段 | maxItems | 裁剪策略 |
|------|----------|----------|
| `stage_events` | 10 | 保留最核心的事件 |
| `current_status` | 10 | 保留当前阶段最重要的状态 |
| `current_personality` | 10 | 保留已稳定显现的核心倾向 |
| `current_mood` | 10 | 保留当前最显著的情绪背景 |
| `knowledge_scope.knows` | 50 | 见上方裁剪策略 |
| `knowledge_scope.does_not_know` | 30 | 同上 |
| `knowledge_scope.uncertain` | 30 | 同上 |
| `concealments` | 20 | 移除已 revealed 的，保留当前隐瞒 |
| `misunderstandings` | 20 | 移除已 resolved 的，保留当前误解 |
| `relationships` | 10 | 只含重要角色，不含边缘角色 |
| `emotional_voice_map` | 10 | 保留最重要、最常见的情绪 |
| `emotional_reaction_map` | 10 | 保留最重要、最常见的情绪 |
| `target_voice_map` | 5 | 只含主要角色和重要配角 |
| `target_behavior_map` | 5 | 与 target_voice_map 对齐 |
| 各 `dialogue_examples` | 5 | 每个 target/emotion 内 |
| 各 `action_examples` | 5 | 每个 target 内 |
| 各 `typical_expressions` | 5 | 每个 target/emotion 内 |
| 各 `typical_actions` | 5 | 每个 emotion 内 |

**缺少任何一个维度都会导致扮演缺陷。** 例如：
- 缺少 emotional_voice_map → AI 在角色愤怒/吃醋/委屈时语气无法区分
- 缺少 target_voice_map 或内容过少 → AI 面对不同角色时说话方式千篇一律
- 缺少 target_behavior_map → AI 面对不同角色时行为反应无差异化
- 缺少 relationships 中某角色的条目 → AI 面对该角色时态度混乱
- 缺少 knowledge_scope → AI 可能泄露角色不该知道的信息
- 缺少 active_aliases.known_as → AI 不知道其他角色该怎么称呼本角色

### 9.5 提取产出物清单

每个 stage 可能产出或更新的文件：

**世界层**：
- `world/stage_snapshots/{stage_id}.json` — 当前阶段的世界快照（LLM 产出）
- `world/stage_catalog.json` — 阶段目录索引（程序自动维护，见 §11.3a）
- `world/world_event_digest.jsonl` — 世界事件增量索引（程序自动维护，见 §11.3a）
- `world/foundation/` — 基础设定修正（LLM 产出，含 fixed_relationships.json）
**角色层（每个活跃角色）**：

## 十、输出质量保护

系统在两个阶段面临不同类型的质量风险，需要不同的应对策略：

- **运行时扮演**（§10.1）：用户与角色的长对话 session 中，随着上下文
  增长，AI 对早期加载的指令和数据的注意力递减（"稀释"），导致角色漂移。
  应对方式是周期性重注入和校准。
- **自动化提取**（§10.2）：每个 stage 是独立的 `claude -p` 调用（全新
  context），不存在 session 内稀释。但跨 stage 仍可能出现风格不一致、
  字段详细度递减等**质量退化**。这些风险已由 automation 架构天然满足。

### 10.1 运行时扮演的防稀释机制

角色扮演 AI 在长对话中面临两类注意力稀释：

**角色信息稀释**：启动时加载的角色数据随对话推进被推离注意力焦点，导致：

- 语气漂移：角色特有的说话方式逐渐退化为泛化 AI 口吻
- 关系姿态漂移：对对话者的态度从精确定位退化为通用友好
- 知识边界泄漏：角色开始给出超出当前阶段知识范围的信息
- 情绪状态重置：对话前段因事件产生的情绪在后段被遗忘

**对话内容稀释**：对话早期的事件和承诺被推出有效注意窗口，导致：

- 承诺/约定遗忘
- 已揭露的信息未被后续对话延续
- 情绪弧线断裂
- 用户提到的具体细节被遗忘

系统应提供以下防稀释机制：

1. **角色锚点重注入**：每轮或每 N 轮向 AI 重新注入压缩的角色核心信息
   （语气签名、硬边界、当前关系姿态、知识边界、情绪状态），防止角色漂移
2. **滚动对话状态**：定期生成对话内容的滚动摘要（情绪轨迹、已揭露信息、
   承诺/约定、关系微变化、未解决话题），确保早期内容不丢失
3. **深度校准检查点**：在重大事件后或每 15-20 轮触发深度校准，重读关键
   角色字段，检测并纠正漂移

### 10.2 提取阶段的跨阶段质量退化防护

> **注意**：本节描述的风险是**跨 stage 的质量退化**，不是 session 内的
> 注意力稀释。自动化提取中每个 stage 都是独立的 `claude -p` 调用（全新
> context），不存在"上下文增长导致注意力递减"的问题。以下措施已由
> automation 架构天然满足，无需额外代码。

多阶段提取跨越长时间和多次独立 AI 调用，面临以下质量退化风险：

**阶段间一致性断裂**：

- 后续 stage 产出的 stage_snapshot 与前面 stage 风格不一致
- 对同一角色的性格描述在不同 stage 间出现矛盾
- 之前标记为"不确定"的点被重新判断但结论不同
- 术语和命名在 stage 间发生漂移

**输出质量递减**：

- stage_snapshot 字段填充从详尽变为粗糙
- evidence_refs 引用越来越少
- stage_events / memory_timeline 摘要越来越空泛
- emotional_voice_map、relationship 等复杂字段被敷衍填充

应对措施及 automation 中的实现方式：

| 措施 | 目的 | automation 实现 |
|------|------|----------------|
| Schema 引用 | 确保字段要求在注意力范围内 | `prompt_builder` 每次构建 prompt 时将 schema 路径列入读取清单 |
| 架构规则内嵌 | 防止退化为增量模式 | `prompt_templates/character_extraction.md` 内嵌自包含快照规则 |
| 前阶段输出参照 | 确保风格和详细度一致 | `prompt_builder` 将前一阶段 snapshot 和 memory_timeline 列为参照文件 |
| 输出质量检查 | 检测字段粗糙、引用缺失等退化 | `validator.py`（程序化）+ `semantic_review`（LLM）两层检查 |
| 跨阶段上下文传递 | 防止信息断裂 | progress 文件 + 前阶段输出 + schema，全部通过文件系统传递 |
| 跨阶段一致性检查 | 检测全局漂移和退化 | `consistency_checker.py`（Phase 3.5，全部 stage 完成后运行） |

3. **提取循环**：对每个 stage 执行：
   a. Git preflight check（工作区干净、分支正确）
   b. **智能跳过**：若本 stage 的提取产物已在磁盘上存在（world + 全部角色
      的 stage_snapshot），直接跳到 d（post_processing），不重复调用 LLM

      该 lane（消耗 `lane_retries`，配额耗尽才全 stage rollback；

      标记 FAILED

### 11.3 Agent 上下文模型

- 每个 stage 拆分为 1+N 次独立 `claude -p` 调用（1 次世界 + N 次角色），**同一 stage 内 1+N 全并行**，完全无共享内存
- 世界和角色间无执行依赖——角色不读世界快照，世界不读角色快照
- 阶段间和调用间上下文完全靠文件系统传递
- 编排脚本预先为每次调用组装 prompt，明确列出该读哪些文件
- **输入裁剪原则**：每次调用只传最近一个 stage_snapshot 和 memory_timeline（用于计算 delta 和延续格式），不传全部历史。Baseline 文件（identity.json, voice_rules.json 等）每个 stage 都传入（任意 stage 可修正）
- **显式排除**：以下文件不传入 extraction agent：
  - `simulation/contracts/baseline_merge.md` — 自包含快照的语义契约已内嵌在 extraction prompt 中
  - `memory_digest.jsonl` — 由程序在提取后自动生成（见 §11.3a）
  - `world_event_digest.jsonl` — 由程序在提取后自动生成（见 §11.3a）
  - `stage_catalog.json`（世界和角色均是）— 由程序在提取后自动维护（见 §11.3a）
- 这是优势而非缺陷：每个 agent 从干净 context 启动，无稀释风险，输入规模不随阶段推进膨胀

### 11.3a 程序化后处理（digest + stage_catalog）

提取 agent 只负责产出核心创造性内容（stage_snapshot、memory_timeline、
baseline 修正）。以下派生文件由编排脚本在提取完成后程序化维护，不消耗
LLM token：

**memory_digest.jsonl 生成**

从本阶段 `memory_timeline/{stage_id}.json` 自动生成压缩索引条目，写入
`characters/{character_id}/canon/memory_digest.jsonl`：

| memory_timeline 字段 | memory_digest 字段 | 映射方式 |
|---|---|---|
| `memory_id` | `memory_id` | 直接复制（`M-S{stage:03d}-{seq:02d}`） |
| `time` | `time` | 直接复制 |
| `location` | `location` | 直接复制 |
| `digest_summary` | `summary` | 直接复制（LLM 专写的 30–50 字 digest 摘要） |
| `memory_importance` | `importance` | 直接复制 |

规则：
- 与 memory_timeline 条目一一对应
- `summary` 由 LLM 在 memory_timeline 的 `digest_summary` 字段中专门撰写
  （30–50 字），**不是**从 `event_description`（150–200 字的完整描述）截取。
  这两个字段有不同职责：`event_description` 是条目详情，`digest_summary`
  是检索索引的一句话
- `stage_id` / `emotional_tags` / `involved_targets` 不再进入 digest
  （stage 从 `memory_id` 的 `S###` 段解析；情绪与关系信息按需通过
  FTS5 拉取 timeline 全文）
- 以 `memory_id` 为主键做 upsert，重跑不重复
- 长度违规（`digest_summary` < 30 或 > 50 字）直接由
  `memory_timeline_entry.schema.json` 判失败——后处理不做截断
- 写入前做 `memory_digest_entry.schema.json` 校验
- 每条 ~30–40 tokens

**world_event_digest.jsonl 生成**

从世界 `stage_snapshots/{stage_id}.json` 的 `stage_events` 字段自动生成
世界事件增量索引，写入 `world/world_event_digest.jsonl`。`stage_events`
条目本身就是 50–80 字的一句话摘要，作为 digest 的唯一来源。**仅记录世界
公共层事件**（势力变迁、地形/资源变化、规则揭示、跨角色公共事件），角色个人
剧情事件不得进入此处：

| world_stage_snapshot 字段 | world_event_digest 字段 | 映射方式 |
|---|---|---|
| `stage_events[i]` | `summary` | 逐条展开（条目本身 50–80 字） |
| 自动生成 | `event_id` | `E-S{stage:03d}-{seq:02d}`（按 stage_events 顺序） |
| `timeline_anchor` | `time` | 直接复制 |
| 从 `stage_events[i]` 推断 | `location` | 最佳匹配（可为空） |
| 从 `stage_events[i]` 推断 | `involved_characters` | 提取涉及角色 |
| 从 `stage_events[i]` 推断 | `importance` | 映射到五级（trivial..defining） |

规则：
- 每条 `stage_events[i]` 对应一条 digest 条目（1:1 拷贝）
- `event_id` 按 `stage_events` 数组顺序从 01 递增编号，**保持连续**。过滤由
  上游（提取+审校）完成——digest 只做机械展开，不做剔除
- `stage_id` 不再冗余存入 digest，从 `event_id` 的 `S###` 段解析
- 以 `event_id` 为主键做 upsert，重跑不重复
- 按 stage order 排序
- 运行时按 stage 范围过滤加载（stage 1 到用户选定 stage N）
- 写入前做 schema 校验（含 `summary` 50–80 字）

**世界事件过滤（在提取/审校阶段完成）**

`world_event_digest` 只承载世界公共层事件。过滤在源头（提取+审校）而非 digest
阶段完成，这样 digest 能保持纯机械的 1:1 展开：

- **提取阶段**：`world_extraction.md` prompt 在规则中明示"仅记录世界公共层
  事件"，并给出反例（"A 告诉 B 关于 C 的秘密" 属角色层，不写入世界）
- **语义审校**：world 审校通道额外读取 `characters/*/canon/memory_timeline/{stage_id}.json`，
  比对条目是否越界到角色层；越界条目列入 semantic_review 报告
- **targeted_fix**：按审校反馈从 world stage_events 删除越界条目
- **事件编号**：过滤发生在 digest 生成之前，`event_id` 在最终 stage_events
  上按顺序编号，天然连续

**stage_catalog.json 维护**

从 stage_snapshot 的元数据字段程序化维护阶段目录索引（世界和角色各自独立）。
stage_catalog 仅用于 bootstrap 阶段选择，**运行时不加载**：

| 来源 | stage_catalog 字段 | 映射方式 |
|---|---|---|
| snapshot `stage_id` | `stage_id` | 直接复制 |
| stage `order` | `order` | stage 序号 |
| snapshot `title` | `title` | 直接复制 |
| snapshot `snapshot_summary` | `summary` | 直接复制（世界事件概要，非主角剧情复述） |
| 计算 | `snapshot_path` | `canon/stage_snapshots/{stage_id}.json` |
| snapshot `timeline_anchor` | `timeline_anchor` | 直接复制 |
| stage `chapters` | `chapter_scope` | 解析章节范围 |

规则：
- 以 `stage_id` 为主键做 upsert，重跑不重复不损坏
- 按 `order` 排序
- 文件不存在时创建（含 `schema_version`、`work_id`、`character_id`）
- 写入前做 schema 校验

### 11.4a 三级 JSON 自动修复

LLM 产出的 JSON 经常有格式问题（内容完整但解析失败），例如在字符串值内使用
未转义的 ASCII 双引号充当中文书名号、尾部多余逗号、输出截断等。管线在判定
失败并触发重跑之前，自动尝试三级修复：

| 级别 | 方法 | 成本 | 处理的问题 |
|------|------|------|-----------|
| L1 | 程序化正则修复 | 0 token | 未转义的内部引号、尾部逗号、JSON 截断（自动补全闭合括号）、正文后的尾部垃圾 |
| L2 | LLM 修复（只发送坏 JSON，不重读源文件，超时 600s） | 少量 token | L1 无法处理的复杂格式问题 |
| L3 | 完整重跑（重新调用提取 agent，最多 1 次） | 全量 token | L1+L2 均失败时自动触发 |

L2 超时默认 600s（`repair_timeout` 参数可配置）。L3 由调用方实现，
`_summarize_chunk` 在 L1+L2 失败后自动重跑整个 chunk 归纳（最多 1 次）。

集成位置：

- **Phase 0 断点恢复**：已有 chunk 文件解析失败时，先尝试 L1→L2 修复再决定
  是否重跑
- **Phase 0 写后验证**：LLM 写完 chunk 后解析失败时，先 L1→L2 修复，
  仍失败则 L3 全量重跑（最多 1 次）
- **Phase 0 完成门控**：全部 chunk 处理完毕后，若有缺失 chunk 则阻断
  Phase 1（`sys.exit(1)`），下次运行自动断点恢复补跑
- **Phase 3 程序化校验**：`validator` 的所有 JSON 加载自动尝试 L1 修复
- **Phase 3 memory_timeline**：JSON 数组文件整体 L1 修复

memory_timeline 为 JSON 数组格式，与其他 JSON 文件使用相同的修复流程。

代码：`automation/persona_extraction/json_repair.py`

### 11.4 三层质量检查

**第一层：程序化校验（不花 token；`jsonschema` 为硬依赖）**

`jsonschema` 是 automation 包的必装依赖（见 `automation/pyproject.toml` 的
`dependencies`），未安装时 validator 直接报错而不是静默跳过。这样保证门控
强度在所有机器上一致，不会出现"本机 gate 通过、CI 失败"这类差异。

- JSON schema 合规性（含 `digest_summary` 30–50 字、`event_description`
  150–200 字、`stage_events` 50–80 字等长度下限的硬门控）
- 必填字段存在性
- stage_catalog 条目数 = stage_snapshots 文件数
- world_event_digest 条目与 world stage_snapshot `stage_events` 对应（逐条 1:1）
- memory_timeline 文件存在性
- evidence_refs（章节号列表）非空
- target_voice_map / target_behavior_map 样本数（importance-based 阈值，
  见 §11.4.1）

**第二层：语义审校（独立 LLM agent）**

- 与前一 stage 的风格/详细度一致性
- 数据边界正确性（世界层 vs 角色层）
- 信息充分性（关键字段的条目数和深度）
- 时间性正确（当前 vs 历史）
- 知识边界合理性
- 角色审校通道还需检查世界-角色交叉一致性（单向验证：角色内容是否与
  世界快照一致）

**第三层：提交门控（程序化，0 token，见 §11.4b）**

- 全通道 PASS 确认
- 世界-角色 stage_id 对齐
- 世界事件与角色阶段内容交叉一致性（程序化兜底）

### 11.4.1 Importance-based 质量阈值

target_voice_map 和 target_behavior_map 的最低 examples 数量不是固定的，
而是根据 target 角色在 `candidate_characters.json` 中的 importance 动态决定：

| target importance | 最低 dialogue_examples / action_examples |
|-------------------|------------------------------------------|
| 主角              | 5                                        |
| 重要配角          | 3                                        |
| 其他（次要配角/龙套/泛化类型） | 1                           |

**生成时告知**：extraction prompt 中注入每个具名 target 的 importance 和
对应最低 examples 数（从 `candidate_characters.json` 读取），使 LLM 在生成时
就知道标准。

**验证时兜底**：validator 和 consistency_checker 使用相同阈值做事后校验，
作为生成质量的最终保障。

### 11.4.2 各 Phase 出口验证

每个 Phase 完成后必须执行最基本的验证，避免错误延迟到下游 Phase 才发现：

| Phase | 验证内容 |
|-------|---------|
| Phase 0 (summarization) | 每个 chunk 输出 JSON 格式正确、非空 |
| Phase 1 (analysis) | candidate_characters / stage_plan / world_overview 结构完整、required 字段存在；stage plan 章节数硬性门控（5-15），违规则删除 plan 并带修正反馈重跑 LLM，重试耗尽则终止流程 |
| Phase 2.5 (baseline) | **关键文件**（缺失 = error，阻断 Phase 3）：identity.json、manifest.json、foundation.json — schema 校验 + required 字段非空。**骨架文件**（缺失 = warning，不阻断）：fixed_relationships.json、voice_rules.json、behavior_rules.json、boundaries.json、failure_modes.json — 存在时做 schema 校验，缺失仅警告（Phase 3 可补充） |
| Phase 3 (extraction) | validate_stage（已有） |
| Phase 3.5 (consistency) | consistency_check（已有）。存在 error 级问题时阻断 Phase 4（`--start-phase 4` 可跳过此门控独立运行） |
| Phase 4 (scene) | per-chapter validation（已有） |

验证失败（error 级）的 Phase 不进入下一 Phase，需修复后重试。warning 级问题不阻断流程。

### 11.4b 并行审校通道与提交门控

每个 stage 的后处理（校验 → 审校 → 修复）按实体拆分为独立的 **审校通道**，
并行执行以缩短耗时：

```
世界提取 + 角色提取 (1+N 全并行) → 程序化后处理 (§11.3a)
                                     │
                    ┌────────────────┼────────────────┐
                    ▼                ▼                ▼
              world 通道       char_1 通道       char_N 通道
              ┌─────────┐     ┌─────────┐      ┌─────────┐
              │ 校验     │     │ 校验     │      │ 校验     │
              │ 审校     │     │ 审校     │      │ 审校     │
              │ 修复(opt)│     │ 修复(opt)│      │ 修复(opt)│
              └────┬────┘     └────┬────┘      └────┬────┘
                   └────────────┬──┴─────────────────┘
                                ▼
                          提交门控 (0 token)
                                │
                         PASS → git commit
                         FAIL → 失败 lane 独立重试（≤2 次）
```

**通道写入边界**：每条通道只读写自己管辖的文件，通道间无写入重叠：
- world 通道：`world/stage_snapshots/{stage_id}.json`、`world/foundation/`
- char_X 通道：`characters/{char_X}/canon/stage_snapshots/{stage_id}.json`、
  `characters/{char_X}/canon/memory_timeline/{stage_id}.json`

**通道内流程**（串行，修复瀑布）：
1. Level 0: Schema autofix — 程序化修复 schema 违规（0 token）
2. 程序化校验（schema、字段完整性、evidence_refs）
3. 语义审校（独立 LLM agent，600s 超时）
4. 如果 FAIL 且可修复 → Level 2/3 定点修复 → 重跑校验+审校
5. 通道结果：PASS 或 FAIL（FAIL 时仅该 lane 回滚重试）

**审校输入收窄**：
- reviewer 不接收 `work_dir` 路径，只接收显式的文件路径列表
- world reviewer 只读：本阶段世界快照、前阶段世界快照（对比）、相关 schema、
  programmatic report
- char reviewer 只读：本阶段角色快照和 memory_timeline、前阶段角色快照、
  本阶段世界快照（交叉一致性单向验证）、相关 schema、programmatic report
- targeted fix 只读：被标记有问题的文件、本阶段原文章节、当前 stage 的
  chapter_summary chunk（而非全量 chunk）

**交叉一致性**：
- 角色通道的 reviewer 负责验证角色内容是否与世界快照一致（单向覆盖）
- world 通道的 reviewer 不需要看角色产物

**提交门控**（程序化，0 token）：

职责边界：门控只做**结构性与标识级**一致性兜底；**内容级**一致性（世界设定与
角色关系 / 认知 / 状态内容是否矛盾）由角色通道的语义审校负责（见 §11.4b
"交叉一致性"）。门控的检查项是最后一道可零 token 完成的护栏，不是跨实体语义
审校的替代。

每条门控失败项都按受影响文件归属到唯一 lane（world 或某个角色），
失败处理因此**复用与审校通道相同的 lane 独立重试路径**（见下方"失败处理"），
不再直接全 stage rollback。

1. 确认全部通道结果为 PASS
2. 结构存在性：world 快照与每个目标角色快照文件均存在
3. 标识对齐：world 快照和所有角色快照的 `stage_id` 字段等于当前 stage_id
4. Catalog 对齐：`stage_catalog.json`（world + 每个角色）**文件必须存在**
   且均包含当前 stage 条目
5. Digest 对齐：
   - 每个角色的 `memory_digest.jsonl` **文件必须存在**且包含当前 stage 的
     条目（stage 段编码在 `memory_id` 前缀 `M-S{stage:03d}-` 中，loader 按
     regex 过滤；digest 条目**不携带** `stage_id` 字段）
   - `world/world_event_digest.jsonl` **文件必须存在**且包含当前 stage 的条目
     （stage 段编码在 `event_id` 前缀 `E-S{stage:03d}-` 中）
6. 轻量跨实体引用：world 快照 `relationship_shifts` / `character_status_changes`
   中提及的角色名，能在世界 cast 或活跃角色 identity.json 的 aliases 中
   解析到（别名索引命中；解析失败仅告警，不阻断 — 因为角色可能尚未被用户
   列入活跃提取范围）

**门控 vs 语义审校的分工**：

| 风险类别 | 负责层 |
|----------|--------|
| 世界 / 角色快照文件缺失 | 提交门控（结构） |
| `stage_id` 字段写错 | 提交门控（标识） |
| digest / catalog 缺条目 | 提交门控（程序化维护产物） |
| 世界设定与角色认知内容矛盾 | 角色通道语义审校（单向验证） |
| 世界事件与角色记忆事实矛盾 | 角色通道语义审校 + §11.10 跨阶段检查 |
| 跨 stage 关系连续性 | §11.10 Phase 3.5 |

**失败处理**（lane 独立重试优先，全量回滚为最后手段）：

无论失败发生在初次提取、审校通道还是提交门控，都先尝试 lane 级修复，
把全 stage 回滚作为最后兜底。`stage.lane_retries` 计数在 **extraction
+ review + gate** 三处共享（≤ 2 次每 lane），仅在门控真正 PASS 后才清零，
避免"review 反复 PASS、gate 反复 FAIL"导致的预算逃逸。

*A0. 初次提取阶段 LLM 报错*（Step 2 — 1+N 并行抽取里某条 lane 失败）：
- 已成功的 lane 产物保留在磁盘，**不**全 stage 回滚
- 仅对失败 lane 调用 `rollback_lane_files` 清掉本次的部分产物 →
  消耗该 lane 一次 `lane_retries` → 仅重跑该 lane 的 extraction 调用
- 该 lane 耗尽 `lane_max_retries` 仍报错 → 升级为全 stage rollback +
  标记 ERROR（消耗 stage 级 `max_retries`）

*A. 审校通道内 FAIL*（修复瀑布走完仍 FAIL）：
- 仅回滚该 lane 的产物文件（world 快照 或 角色快照 + memory_timeline），
  已通过 lane 的产物保留在磁盘上
- 仅重新提取该失败 lane（单次 LLM 调用，注入该 lane 的 reviewer feedback）
- 重跑后处理（idempotent，upsert 仅该 stage 条目）→ 再走全部 lane 的
  修复瀑布（跨依赖：world reviewer 读各角色 memory_timeline）

*B. 提交门控 FAIL*（全部 lane 已 PASS 但门控失败）：

按修复成本递增分类处理，依次尝试：

| 门控失败类型 | 归属 lane | 修复方式 |
|-------------|-----------|----------|
| catalog / digest 文件缺失或缺当前 stage 条目 | 关联 lane（world catalog / world_event_digest → world；char catalog / memory_digest → 该角色） | **重跑后处理一次**（0 token，从已存在 snapshot upsert），再次门控 |
| snapshot 文件缺失 | world / 该角色 | lane 重提取（消耗 lane 配额） |
| snapshot `stage_id` 字段错位 | world / 该角色 | lane 重提取（消耗 lane 配额） |
| snapshot JSON 解析失败 | world / 该角色 | lane 重提取（消耗 lane 配额） |

如果"重跑后处理"仍未通过门控（说明 snapshot 本身就缺关键字段，post_processing
也救不回来），自动升级为 lane 重提取。

*C. 共同终止条件*（A、B 共用）：
- 每个 lane 独立维护重试计数（≤ `lane_max_retries`=2，review 与 gate 共享）
- 任一失败 lane 耗尽重试仍 FAIL → 全 stage rollback
  （`git checkout + clean`） + 整个 stage 标记 FAILED →
  进入 stage 级重试（≤ `max_retries`=2）
- 门控失败但**无可归属 lane**（极少见，例如 stage_id 解析失败）→
  直接全 stage rollback（无 lane 可重试）
- 重提取本身 LLM 报错（token 超限、速率限制等）→ 仅对该 lane 走
  `rollback_lane_files`，由下一轮 review/gate 通过 `snapshot_missing`
  类目继续走 cascade（消耗下一格 `lane_retries` 配额），直到该 lane
  配额耗尽才升级为全 stage rollback
- 不允许单个通道单独 commit（提交门控仍要求全通道 PASS）

**提交顺序契约**（防止伪 committed 状态）：

先执行 `git commit`，拿到 SHA 成功后再把 stage 状态迁移到 `COMMITTED`
并落盘 progress 文件。若 `git commit` 返回空（无变更或 commit 失败），状态
回退到 `FAILED` 并记录错误原因，不得写入 `COMMITTED`。顺序：

```
PASSED → git commit → [sha 非空] → transition(COMMITTED) → 保存 progress
                   ↘ [sha 为空] → transition(FAILED) → 保存 progress
```

**进度文件提交策略**：stage 的提取产物 commit 只包含 `works/` 下的数据产物；
`analysis/progress/` 本身在 `.gitignore` 中（本地 runtime 状态）。不再试图
让 progress 文件与 stage 产出落入同一 commit —— 该约束历史上导致"先落盘
progress、后 commit"的反向顺序，是门控误报之外最直接的 committed 状态漂移
来源。

### 11.5 失败处理

#### 修复瀑布（Fix Cascade）

程序化校验或语义审校发现问题后，按修复成本递增依次尝试四个层级。每个层级
修复后重新校验，通过则继续流程，不升级到更高层级：

```
提取完成
  ▼
Level 0: Schema Autofix（程序化，0 token，<1s）
  删除 schema 不允许的多余字段、截断超长数组/字符串、类型强转、
  填充缺失 required 字段、enum 大小写修正
  ▼ 修复后重跑程序化校验
Level 1: 程序化校验（0 token）
  ▼ FAIL 且 autofix 无法覆盖 → 升级 Level 2
  ▼ PASS → 进入语义审校
Level 2: 定点 LLM 修复（低 token，~30s）
  只发送出错字段的 JSON 片段 + 错误描述 + schema 约束
  ▼ 修复后重跑程序化校验 + 语义审校
Level 3: 语义 LLM 修复（中 token，~2min）
  出错片段 + 相关章节原文 + reviewer 的具体 findings
  ▼ 修复后重跑程序化校验 + 语义审校
```

最终仍然失败 → 该 lane 标记 FAIL → 仅该 lane 回滚重提取（见下方"Lane
独立重试"）。

#### Level 0: Schema Autofix

程序化校验失败时自动触发。根据 `jsonschema.ValidationError` 的结构化信息
（`validator` 类型 + `absolute_path`）对 JSON 文件做确定性修复：

| 错误类型 | 修复方式 |
|----------|----------|
| `additionalProperties` | 按 schema `properties` 键集合删除多余字段 |
| `maxItems` | 截断数组到上限 |
| `maxLength` | 截断字符串到上限 |
| `minLength` | 空/近空字符串填充占位符 |
| `type` | 简单类型强转（string↔number、scalar→array） |
| `required` | 缺失字段填入类型安全的空默认值 |
| `enum` | 大小写不敏感精确匹配 |

修复后写回文件并重新跑程序化校验。**不涉及 LLM 调用**。

#### Level 2–3: 定点 LLM 修复

- Level 2（轻量）：只传出错字段的 JSON 片段 + 错误描述 + 对应 schema 约束。
  适用于格式/结构基本正确但内容需要小幅调整的情况
- Level 3（语义）：出错片段 + 相关章节原文 + reviewer findings。适用于内容
  矛盾（如 knowledge_scope 与 memory_timeline 冲突）需要语义理解的情况
- **信息权威层级**：原文章节 > 章节摘要 > memory_timeline > stage_snapshot >
  world_snapshot。冲突时以高优先级来源为准
- 不重传整个提取 prompt（省 token，通常 <5% 原始提取成本）
- 指令为"最小化修改，只改审校指出的问题，不动其他内容"
- 修复后重跑程序化校验 + 语义审校确认修复无误

#### 系统性问题（触发 lane 重提取）

以下情况跳过修复瀑布，直接标记 lane FAIL 并触发该 lane 单独重提取：

- 整个文件缺失（如某角色的 snapshot 没有产出）
- 大段内容缺失（如 voice_map 为空）
- 整体语义偏差（如对阶段弧线理解方向错误）
- Error 数 > 5（批量错误，修复成本超过重提取）

#### Lane 独立重试

审校通道内 lane 失败时，**仅回滚该 lane 的产出文件并重提取该 lane**。
其他已通过的 lane 的产出保留在磁盘上，不受影响。这避免了"单 lane 失败
→ 全部 rollback"的浪费。

规则：
- 每个 lane 独立维护重试计数（≤ 2 次）
- 重提取时注入该 lane 的 reviewer feedback
- 全部 lane 通过后才进入提交门控
- 如果某 lane 耗尽重试次数仍失败 → 整个 stage 标记 FAILED

#### 流程图

```
提取 → [每个 lane 独立]:
        │
        ├─ Level 0: Schema Autofix (0 token)
        │   ↓ 重跑校验
        ├─ Level 1: 程序化校验
        │   ↓ PASS → 语义审校
        │   ↓ FAIL → Level 2
        ├─ Level 2: 定点 LLM 修复 (低 token)
        │   ↓ 重跑校验 + 审校
        ├─ 语义审校
        │   ↓ PASS → lane 通过
        │   ↓ FAIL + 可修复 → Level 3
        │   ↓ FAIL + 系统性 → lane 重提取
        ├─ Level 3: 语义 LLM 修复 (中 token)
        │   ↓ 重跑校验 + 审校
        │   ↓ 仍 FAIL → lane 重提取
        └─ lane 重提取（仅该 lane，≤ 2 次）

全部 lane 通过 → 提交门控 → commit
```

#### 其他失败场景

- Rate limit → 自动等待后重试（递增退避）
- **Token/context limit → 不重试**（相同 prompt 必定再次超限），直接标记
  ERROR 并回滚。与 rate limit 明确区分，避免浪费重试配额
- 每个 lane 最大重试次数 2 次（仅该 lane 重提取），全部 lane 耗尽后
  标记为 blocked
- 脚本崩溃 → 重启后从 progress 文件恢复
- **Baseline 恢复**：`pipeline.json` 追踪 Phase 2.5 完成状态。`--resume` 时
  如果 baseline 未完成（Phase 2.5 产出缺失——包括 foundation.json、
  fixed_relationships.json、identity.json），自动补跑 baseline production
  再进入提取循环。`--start-phase 2.5` 强制重跑 baseline（忽略完成标记）。
  **关键约束**：无论 baseline 来自新跑还是文件已存在的恢复路径，进入提取
  循环前**必须调用 Phase 2.5 exit validation**（`validate_baseline()`），
  通过后才 `mark_done("phase_2_5")`。"文件存在"不等于"通过校验"——外部损坏、
  旧格式或字段漂移的 baseline 必须在 Phase 3 之前被拦截
- **磁盘对账自愈**：每次启动加载 progress 后，Phase 0/3/4 各自调用
  `reconcile_with_disk()` 与磁盘产物对账。规则：(1) 终态但产物缺失
  → 回退 PENDING；(2) PENDING 但磁盘有产物 → 清掉产物（视为不完整的
  半成品）；(3) 任意中间态 → 清产物 + 回退。Phase 3 额外用
  `git cat-file -e` 校验 `committed_sha` 是否仍在 git 仓库中可达，
  reset/rebase 丢掉的 commit 视同产物缺失。覆盖任意状态被中断 + 手动
  清理 + 半成品 + git 回退等所有偏移场景
- **Phase 3 progress 重建**：若 Phase 2 已完成但 `phase3_stages.json`
  缺失或损坏，从 `stage_plan.json` 重建 progress（全部 stage 标 pending），
  避免落到 fresh-start 路径重新提示选角色 + 覆写 pipeline.json
- **同次运行自动重试**：Phase 3 和 Phase 4 均在同次运行内自动重试
  FAILED 的 stage/chapter（retry_count < max_retries 时直接重新处理），
  无需用户手动 resume。所有失败路径（LLM 失败、JSON 解析失败、校验失败）
  均递增 retry_count，超限升级为 ERROR（blocked）
- **Resume 自动重置**：`--resume` 时自动将所有 blocked（retry 耗尽的
  ERROR）重置为 pending，**retry_count 清零**（给用户全新的重试机会）。
  Phase 3 stage 重试由 targeted_fix 注入反馈信息；Phase 4 chapter
  重试同样注入上次错误信息到 prompt
- **Progress 与 `--end-stage` 分离**（同 Phase 4 模式）：`phase3_stages.json`
  始终包含完整 stage plan（Phase 2 确认时写入全量），`--end-stage` 仅作为
  运行时限制控制本次执行范围（`None`=全部，`0`=仅 baseline 不跑 stage），
  不影响进度数据结构。提取循环入口额外执行防御性补全（从 stage plan 追加
  缺失 stage），应对 plan 更新或手动编辑 progress 等边缘情况
- **`--end-stage` 语义（严格前缀运行）**：命中 `--end-stage` 的前缀停止路径
  不触发 Phase 3.5 / Phase 4 / squash-merge 提示 —— 这些收尾步骤必须等到
  所有 stage 都 `COMMITTED`。退出信息明确提示"前缀运行完成，剩余 stage 未
  覆盖；请重新运行（不带 `--end-stage`）以完成最终流程"。两种"退出循环"要
  在日志与后处理分支上区分开：
  - `stopped_by_limit`（前缀试跑）：只打印下次继续的提示，不进任何后处理
  - `all_done`（全部 committed）：跑 Phase 3.5 → squash-merge 提示 → Phase 4

### 11.6 Git 集成

- 每个 stage 通过审校后自动 git commit
- 回滚 = git checkout + git clean 全仓库范围（不仅限于 works/），
  确保 LLM agent 在任意目录写入的残留文件都被清除；但会显式保留
  `works/{work_id}/analysis/scene_splits/`、`analysis/progress/` 和
  `.scene_archive.lock`，避免 Phase 3 回滚误删 Phase 4 的中间产物与进度状态
- 提取在独立分支（`extraction/{work_id}`）进行，每个 stage 单独 commit
  以保持最细粒度的回滚能力
- 全部提取完成后，squash merge 回主分支——main 上只出现一个干净的提取
  commit，extraction 分支用完可删除
- 每个 stage 开始前执行 git preflight check

### 11.7 LLM 后端抽象

编排脚本通过统一接口支持多种 LLM 后端：

- Claude Code CLI（`claude -p`，使用 Claude subscription）
- Codex CLI（使用 Codex subscription）
- 可混合使用（如 Claude 提取 + Codex 审校）

### 11.8 运行保障

**后台运行**：提取流程可能持续数小时甚至数天。编排脚本应支持：

- 后台运行模式，SSH/终端断开后进程继续执行
- 输出重定向到日志文件，支持事后查看和实时跟踪
- 总运行时间限制，到期后在 stage 间优雅停止（不打断正在进行的 stage）

**进程互斥**：同一作品同时只允许一个提取进程运行。启动前应检查：

- 是否已有正在运行的提取进程（PID lockfile）
- Git 工作区是否干净

**自我保护**：防止进程因异常而无限运行占用资源：

- 每个 LLM 子进程有硬超时（提取 3600s、审校 600s）
- 每个 stage 有最大重试次数
- 可选的总运行时间上限

**进度可观测**：运行时应周期性显示：

- 当前子进程 PID 和内存占用
- 已用时间和分步耗时
- 基于历史的分步 ETA 预估

### 11.9 Stage boundary 规则

分析阶段的阶段规划应遵循：

- 尽量贴近自然剧情边界
- 最小阶段 5 章，最大阶段 15 章，默认 10 章
- 为每个阶段取有意义的中文名称（如"阶段01_主角初登场"）
- stage_id 在世界包和角色包间必须一致

#### 出口验证：超限 stage 自动重跑

LLM 产出的 stage plan 可能违反章节数上限（实测常见）。Phase 1 出口验证
检测到超限 stage 后，删除 `stage_plan.json` 并带修正反馈重跑 LLM：

1. 扫描所有 stage，检查 `chapter_count` 是否在 5-15 范围内
2. 如有超限，打印违规 stage 列表，删除 `stage_plan.json`
3. 构建修正反馈（列出具体违规 stage 和章节数），追加到下次 analysis prompt
4. 重跑 Phase 1 LLM（最多重试 2 次），其他已产出文件（world_overview、
   candidate_characters）保留不变
5. 若重试耗尽仍有超限，终止流程（`sys.exit(1)`），不允许带违规 plan 继续

### 11.10 跨阶段一致性检查（Phase 3.5）

Phase 3 全部 stage 提交后，运行跨阶段一致性检查。
主要为程序化检查（零 token 开销），仅对标记项可选 LLM 裁定。

#### 程序化检查项

| # | 检查 | 说明 |
|---|------|------|
| 1 | alias 一致性 | stage_snapshot 的 active_aliases 是否在 identity.json aliases 中有定义 |
| 2 | 快照字段完整性 | 每个 stage_snapshot 是否包含全部 13 个必填维度（personality, mood, voice_state, behavior_state, boundary_state, relationships, knowledge_scope, stage_delta 等） |
| 3 | 关系连续性 | 相邻 stage 间 attitude/trust/intimacy 变化是否有 driving_events 归因 |
| 4 | evidence_refs 覆盖率 | 快照和记忆条目中 evidence_refs 为空的比例 |
| 5 | memory_digest 对应 | memory_digest.jsonl 条目是否与 memory_timeline memory_id 一一对应 |
| 6 | target_map 样本数 | target_voice_map / target_behavior_map 样本数是否满足 importance-based 阈值（§11.4.1） |
| 7 | stage_id 对齐 | 世界/角色 stage_catalog 和 stage_snapshots 目录是否对齐 |
| 8 | world_event_digest 对应 | world_event_digest.jsonl 条目数是否与 world stage_snapshot `stage_events` 逐条对应 |

> 长度约束（`stage_events` 50–80 字、`event_description` 150–200 字、
> `digest_summary` 30–50 字等）由 schema minLength/maxLength 在每阶段
> 程序化校验时硬阻断，无需 Phase 3.5 再重复聚合检查。
> `knowledge_scope` 条数（knows ≤ 50、does_not_know ≤ 30、uncertain ≤ 30）
> 与单条字数（≤ 50 字）同由 schema maxItems/maxLength 在每阶段硬阻断。

#### LLM 裁定（可选）

对程序化检查标记的 warning/error 项，可调用独立 LLM agent 进行语义层面的
裁定（如判断关系变化是否合理、摘要是否空泛）。仅在有标记项时调用，
无标记项则跳过。

#### 产出

- `works/{work_id}/analysis/consistency_report.json`
- 控制台打印摘要（error/warning/info 计数）
- 有 error 级别问题时阻断 Phase 4（需人工处理后继续）

### 11.11 场景切分（Phase 4）

Phase 4 与 Phase 3 **数据独立**——它只读原文章节文件和 `stage_plan.json`
（chapter → stage_id 映射），不依赖 Phase 3 的角色/世界提取产出。前置条件仅为
`stage_plan.json` 存在（Phase 1 产物）。`stage_plan.json` 是
**scene_archive 的 `stage_id` 唯一真源**：若 Phase 1 重新生成了 stage plan
（例如 stage 命名变更），Phase 4 合并时必须据此**重算所有 scene 的
`stage_id` 与 `scene_id`**，始终以 stage_plan 为准。
在常规流程中，Phase 3.5 一致性检查存在 error 级问题时仍会阻断 Phase 4
（见 §11.4.2）；`--start-phase 4` 可跳过此门控独立运行。

#### 调用粒度

每章一次 `claude -p` 调用。LLM 只输出场景边界标注和元数据（起止行号、时间、
地点、在场角色、摘要），**不输出 full_text**——程序根据行号从原文中提取。

#### 并行执行

多章并行处理，`--concurrency N`（默认 10）控制最大并发数。使用
ThreadPoolExecutor，每个 worker 起一个 `claude -p` 子进程。

#### LLM 输出格式（每章）

```json
[
  {
    "scene_start_line": 1,
    "scene_end_line": 45,
    "time": "故事开始第一天清晨",
    "location": "某村外",
    "characters_present": ["角色A", "角色B"],
    "summary": "角色A在村外苏醒，发现角色B昏倒在旁..."
  }
]
```

程序后处理补充三个字段：
- `stage_id`：由章节号查 `stage_plan.json` 得出（唯一真源）
- `scene_id`：`SC-S{stage:03d}-{seq:02d}`（3 位阶段号支持 ≤999 阶段，
  2 位 seq 每阶段上限 99 个场景，符合 §12.5.1 "一个场景不跨章节边界"
  的粒度；seq 在同一 stage 内按章节顺序 + 章节内 scene 序递增）
- `full_text`：根据 `scene_start_line` / `scene_end_line` 从原文提取

#### 质量保障

仅程序化校验，不做语义审校：

| 检查项 | 说明 |
|--------|------|
| 行号有效 | 不超出原文总行数，start ≤ end |
| 场景不重叠 | 相邻场景的行号区间无交叉 |
| 覆盖全章 | 所有行均被某个场景覆盖，无遗漏 |
| 必填字段完整 | `time`, `location`, `characters_present`, `summary` 非空 |
| stage_id 来源一致 | 每个 scene 的 `stage_id` 必须匹配 `stage_plan.json` 当前内容；旧 scene_archive 中的 `stage_id` 不得沿用 |
| scene_id 格式 | 严格匹配 `SC-S\d{3}-\d{2}`，同一 stage 内 seq 唯一且从 01 开始连续 |
| alias 匹配 | characters_present 中的名字与已知角色 alias 一致（可选——scene archive 是 work-level 全局产物，不限于提取目标角色集） |

所有失败（LLM 失败、JSON 解析失败、校验失败）→ 递增 retry_count，同次运行内
自动重试（注入上次错误信息到 prompt），每章最多重试 2 次，超限进 error 状态。
`--resume` 时 ERROR 章节重置为 pending 且 retry_count 清零。

#### 状态机（per-chapter）

```
pending → splitting → split → validating → passed
   ↑                             │
   └──── retrying ←── failed ←───┘
         (retry ≤ 2)
```

#### 目录结构

```
works/{work_id}/
├── retrieval/
│   └── scene_archive.jsonl        ← 最终产物（.gitignore，文件过大）
├── analysis/
│   ├── .scene_archive.lock        ← Phase 4 独立 PID 锁（.gitignore）
│   ├── scene_splits/              ← 中间产物（.gitignore，本地断点恢复）
│   │   ├── 0001.json              ← 每章的场景标注
│   │   └── ...
│   └── progress/
│       └── phase4_scenes.json     ← Phase 4 进度追踪
```

运行时检索数据库 `fts.sqlite` 同样放在 `works/{work_id}/retrieval/`。

#### 进度追踪

Phase 4 进度在 `analysis/progress/phase4_scenes.json`：

```json
{
  "work_id": "<work_id>",
  "total_chapters": 500,
  "chapters": {
    "0001": {"state": "passed", "retry_count": 0},
    "0002": {"state": "splitting", "retry_count": 0},
    "0015": {"state": "failed", "retry_count": 2, "error": "..."}
  },
  "merged": false
}
```

#### 合并流程

全部章节 passed 后：
1. 按章节号顺序遍历 `splits/` 目录
2. 读每章标注 → 根据行号从原文提取 `full_text`
3. 查 `stage_plan.json` 补充 `stage_id`（解析 stage 的 `chapters` 字段，
   格式为 `"0001-0011"` 字符串，构建 chapter → stage_id 映射）——**始终以
   当前 stage_plan 为真源**
4. 生成 `scene_id`（`SC-S{stage:03d}-{seq:02d}`，seq 在同一 stage 内
   按"章节号 → 章节内 scene 序"递增，从 01 开始）
5. 整体重写 `scene_archive.jsonl`（每次合并完整重建，确保与 stage_plan 对齐）
6. 完整性检查：章节覆盖率、`scene_id` 唯一性、每个 `stage_id` 均在
   stage_plan 中存在

#### 断点恢复

- `--resume` 读 Phase 4 progress，跳过已 passed 的章节
- `--end-stage N` 限制只处理前 N 个 stage 对应的章节范围
- 已有的 `scene_splits/` 临时文件作为断点，不重复处理
- failed/error 章节在 `--resume` 时自动重置为 pending（保留 `retry_count`，
  不清零；当累积重试超过 `max_retries` 时升级为 error 永久跳过）
- **split 文件存在性校验**：每次启动通过 `reconcile_with_disk()` 校验
  passed 章节的 `splits/{chapter}.json` 是否实际存在；缺失则重置为
  pending 重新生成。同时清掉 PENDING/中间态遗留的半成品 split 文件，
  防止文件丢失或半成品导致最终 merge 失败
- Phase 4 使用独立 `.scene_archive.lock`；lock、`scene_splits/`
  中间产物和 `progress/` 目录均为本地忽略文件（.gitignore），
  **不得被 git track**。如 extraction 分支上已有 tracked 的 split 文件，
  需 `git rm --cached` 移除跟踪。否则 Phase 3 回滚（`git checkout -- .`）
  会覆盖 tracked 的 split 文件，导致已通过的章节产物丢失

#### Phase 选择与进度管理

CLI `--start-phase` 参数（0/1/2/2.5/3/3.5/4），默认 auto（自动检测）。

进度文件统一存放在 `works/{work_id}/analysis/progress/`：

- `pipeline.json` — 总 pipeline 状态（phase 级完成标记），`--start-phase`
  读取此文件判断跳转，指定具体 phase 时强制重跑该 phase
- `phase0_summaries.json` — Phase 0 chunk 级显式进度，chunk 完成后
  更新状态；启动时通过 `reconcile_with_disk()` 与磁盘 chunk 文件对账
  （done 但缺文件 → 回退 pending；pending 但磁盘有文件 → 清掉半成品）
- `phase3_stages.json` — Phase 3 stage 状态机（完整 stage plan +
  per-stage 状态、重试计数、commit SHA）
- `phase4_scenes.json` — Phase 4 per-chapter 场景切分进度
- `extraction.log` — 运行时日志（.gitignore）

`--end-stage` 语义：`None`（未指定）=全部，`0`=仅 baseline 不跑 stage，
正整数=跑 N 个 stage。

## 十二、记忆系统与运行时检索

### 12.1 总体目标

构建三层记忆系统，为运行时模拟引擎提供从聚合状态到原文细节的完整记忆支撑，
提升角色对话的真实性、语言风格的还原度和历史回忆的准确性。

### 12.2 三层架构

系统使用三层记忆结构，粒度从粗到细，职责不重叠：

| 层 | 名称 | 定位 | 视角 | 存储级别 |
|----|------|------|------|----------|
| 第一层 | stage_snapshot | 角色当前阶段的聚合状态 | 角色主观（聚合） | 角色级 |
| 第二层 | memory_timeline | 角色从头到现在的经历过程 | 角色第一人称主观（逐事件） | 角色级 |
| 第三层 | scene_archive | 小说原文按场景切分的档案 | 客观原文 | 作品级 |

三者的关系：
- stage_snapshot 是**结论**（"我现在信任他"）
- memory_timeline 是**过程**（"那次他替我挡剑之后我开始动摇"）
- scene_archive 是**原始证据**（完整的原文对话和叙述）

#### 记忆系统整体加载与检索流程

```
═══════════════════════════════════════════════════════════════
                    启 动 加 载 阶 段
═══════════════════════════════════════════════════════════════

    第一层: stage_snapshot
    ┌───────────────────────────────────────────────────────┐
    │ 加载当前 stage snapshot（自包含完整状态）              │
    │   └─ target_voice/behavior_map 按用户角色过滤         │
    │      └─ 未找到 → 逆序扫描前序 snapshot (fallback)     │
    └───────────────────────────────────────────────────────┘

    第二层: memory + event digest
    ┌───────────────────────────────────────────────────────┐
    │ ① 近期记忆: stage N + N-1 全量加载 (~10K tokens)      │
    │ ② 远期感知: memory_digest.jsonl stage 1..N 过滤加载   │
    │   └─ 压缩索引，覆盖到当前阶段，供 LLM 判断相关性      │
    │ ③ 世界事件: world_event_digest.jsonl stage 1..N       │
    │   └─ 世界级事件时间线，覆盖到当前阶段                  │
    └───────────────────────────────────────────────────────┘

    第三层: scene_archive
    ┌───────────────────────────────────────────────────────┐
    │ ① 最近 scene_fulltext_window 条 full_text (默认 10)    │
    │ ② 摘要**不进启动**——仅存 FTS5，命中时按需取用         │
    └───────────────────────────────────────────────────────┘

    检索基础设施
    ┌───────────────────────────────────────────────────────┐
    │ vocab_dict.txt → jieba 自定义词典                      │
    │ fts.sqlite（memory_timeline + scene_archive 的 FTS5） │
    └───────────────────────────────────────────────────────┘

═══════════════════════════════════════════════════════════════
                    对 话 检 索 阶 段（每轮）
═══════════════════════════════════════════════════════════════

    用户输入 + 情境关键词（地点、情绪、近期事件）
        │
        ▼
    jieba 分词 + vocab dict 匹配 (<10ms)
        │
    ┌───┴───────┐
    │           │
   命中       未命中 → 不额外检索，用已加载上下文回复
    │
    ▼
   FTS5 查询 memory_timeline + scene_archive (<10ms)
    │
    ▼
   top-K 候选摘要注入 prompt → LLM 判断相关性
    │
    ├─ 候选够用 → 生成回复（一次 LLM 调用）
    │
    └─ 候选不足 → LLM 调用 search_memory 工具
                   → embedding 语义检索 scene_archive
                   → scene_refs 反查 memory_timeline
                   → 生成回复（两次 LLM 调用）
```

#### 12.2.1 为什么不设独立对话语料库

不单独建立对话语料库。原因：

- 对话原文已包含在 scene_archive 的 `full_text` 中，单独抽取会导致数据重叠
- 每句台词标注说话人、情绪、语言模式的提取成本过高
- 运行时 LLM 从完整场景上下文中理解对话比脱离上下文的孤立标注更准确
- 角色的语言模式已在 `stage_snapshot.voice_state` 中结构化描述，
  场景原文作为 few-shot 样本补充即可

### 12.3 第一层：stage_snapshot（角色状态快照）

**定位**：角色在某个阶段的聚合状态——"角色现在是什么样"

**存储位置**：
```
works/{work_id}/characters/{character_id}/canon/stage_snapshots/{stage_id}.json
```

**粒度**：每阶段一份

**加载策略**：启动时只加载当前所选 stage

**内容**：
- 当前性格、情绪基调、行为模式
- 完整语言风格（voice_state，含情绪语气矩阵、对象语气矩阵）
- 完整行为规则（behavior_state，含情绪反应矩阵、关系行为矩阵、
  对象行为矩阵）
- 完整关系状态（对每个重要角色的态度、信任度、亲密度、关系演变概述）
- 知识边界（知道什么、不知道什么、不确定什么）
- 当前持有的误解和主动隐瞒
- 阶段变化摘要（stage_delta）

**运行时过滤加载与 Fallback**

`target_voice_map` 和 `target_behavior_map` 运行时不全量加载，而是按用户
扮演角色过滤。完整流程：

1. **加载当前 stage snapshot**：引擎从磁盘读取用户所选阶段的
   `stage_snapshots/{stage_id}.json`
2. **匹配 target 条目**：在 `target_voice_map` 和 `target_behavior_map` 中
   查找与用户角色匹配的条目
   - 用户扮演 canon 角色 → 精确匹配 `target_type`
   - 用户扮演 OC / 自定义角色 → 按 role_binding 中的设定特征匹配最接近的
     关系类型条目
3. **Fallback——向前扫描**：如果当前 snapshot 中找不到匹配条目（如该角色
   近期数个阶段未出场，提取时继承遗漏），引擎按阶段编号 **逆序遍历**
   之前的 `stage_snapshots/` 文件，找到第一个包含该 target 条目的快照，
   取出对应条目
4. **拼装 prompt**：将匹配到的 target 条目（无论来自当前快照还是 fallback）
   与其他启动数据一起拼入发送给 LLM 的 prompt

```
加载当前 stage snapshot
        │
在 target_voice/behavior_map 中查找匹配条目
        │
    ┌───┴───┐
   找到    未找到
    │       │
    │    逆序扫描前序 snapshot
    │       │
    │   ┌───┴───┐
    │  找到    均无
    │   │       │
    ▼   ▼       ▼
  使用条目    跳过
    │           │
    ▼           ▼
   拼装 prompt → LLM 调用
```

整个过程是 **纯代码层面的文件 I/O**，在 LLM 调用之前完成，不产生额外的
LLM 调用。LLM 收到的 prompt 中已包含正确的 target 条目，无需感知数据
来自哪个阶段。

**两层保障**：
- **提取层**：prompt 指令要求抽取 agent 将未出场角色的 target 条目从
  前一阶段原样继承（优先路径，保持快照自包含）
- **运行时层**：上述 fallback 扫描（安全网，应对提取遗漏）

**不包含**：事件经过的细节、角色对事件的主观体验过程

### 12.4 第二层：memory_timeline（角色主观记忆线）

**定位**：角色从故事开始到现在经历了什么——"角色记得发生过什么、
怎么走到这一步的"

**存储位置**：
```
works/{work_id}/characters/{character_id}/canon/memory_timeline/{stage_id}.json
```

**粒度**：每阶段一个文件，每个文件内包含该阶段的多条记忆条目

**加载策略（分层启动记忆）**：

启动时加载全历史 critical/defining 全文在阶段数较多时（30+）会超出 token
预算。改为三层：

1. **近期记忆**：启动时加载最近 2 个阶段（stage N + N-1）的
   memory_timeline **全文**（~10K tokens）
2. **远期记忆摘要**：启动时加载 `memory_digest.jsonl`（按 stage 1 到
   用户选定 stage N 过滤），压缩摘要（每条 ~80 tokens，见下方 §12.4.4），
   供 LLM 感知历史记忆全貌并判断是否需要检索详情
3. **按需检索**：其余条目通过 FTS5 索引 memory_timeline 按需检索

```
启动加载
  │
  ├── 近期记忆：stage N + N-1 memory_timeline 全文
  ├── 远期记忆：memory_digest.jsonl（stage 1..N 过滤）
  ├── 世界事件：world_event_digest.jsonl（stage 1..N 过滤）
  │
对话中
  │
  ├── LLM 根据 digest 判断相关性
  │     │
  │  需要详情 → FTS5 检索 memory_timeline 全文条目
  │     │
  │  仍不足 → search_memory 工具 → scene_archive embedding
  │                                  → scene_refs 反查 memory_timeline
  │
  └── jieba + vocab dict 主动关联（每轮自动）
        → FTS5 命中 memory_timeline / scene_archive
        → 候选摘要注入 prompt
```

memory_timeline 不需要单独做 embedding 索引——FTS5 + vocab dict 对归纳
文本命中率足够高；语义模糊查询由 scene_archive embedding 覆盖，通过
`scene_refs` 可反查对应的 memory_timeline 条目。

#### 12.4.1 归纳要求

- **角色第一人称主观视角**，不是第三人称客观叙述
- **是归纳，不是原文复制**——提炼事件对角色的意义，不是复述小说文字
- **必须包含心理活动和态度变化的因果**——不只说"关系变好了"，要说
  角色为什么觉得变好了
- **篇幅由事件复杂度决定，不设硬性字数限制**——简单事件可能几十字，
  复杂事件可能几百字
- 控制 token 量的方式是**控制条目数量**（按重要度筛选哪些事件值得记录），
  不是压缩单条篇幅

#### 12.4.2 条目结构

```json
{
  "memory_id": "M-S{stage:03d}-{seq:02d}",
  "stage_id": "所属阶段",
  "time": "故事内时间",
  "location": "事件发生地点",
  "event_description": "客观层面发生了什么的完整描述（第三人称，150–200 字，记录事件的关键起因、经过与结果，为 subjective_experience 提供事实锚点）",
  "digest_summary": "供 memory_digest 使用的一句话摘要（30–50 字，概括事件本体，不含主观情绪；post_processing 会直接复制到 memory_digest.summary）",
  "subjective_experience": "角色对这件事的主观体验（第一人称视角，核心字段）",
  "emotional_impact": "对角色的情感影响",
  "knowledge_gained": ["角色通过此事获得的新认知"],
  "misunderstanding": {
    "content": "角色误以为的内容",
    "truth": "实际真相"
  },
  "concealment": {
    "content": "隐瞒的内容",
    "target": "对谁隐瞒",
    "reason": "原因"
  },
  "relationship_impact": [
    { "target": "关系对象", "change": "关系变化描述" }
  ],
  "memory_importance": "trivial / minor / significant / critical / defining",
  "scene_refs": ["关联的 scene_archive scene_id（SC-S###-##）"],
  "evidence_refs": ["原文证据引用"]
}
```

**`event_description` vs `digest_summary` 的分工**：

- `event_description`（150–200 字）是条目内部的事实详情。长度下限硬门控，
  避免"XX 发生了"这类空摘要
- `digest_summary`（30–50 字）专供 `memory_digest.jsonl` 使用，由 LLM
  在提取阶段一并撰写（不是从 `event_description` 截断得到）。这样 digest
  的一句话摘要可以保留最高检索价值的关键词，而 `event_description` 保留
  完整因果链
- Schema 两边都设 `minLength`/`maxLength`——字段过短或过长时
  `jsonschema` 直接判失败，不走 warning 路径

`time` 使用故事内时间，格式不必严格统一（小说里的时间本身就是
模糊的），提取时保留原文的时间表述即可。全字段在整个系统中统一命名为
`time`。

`memory_id` 必须匹配正则 `^M-S\d{3}-\d{2}$`，其中 `S###` 与 `stage_id`
对应的阶段序号一致；同一阶段内 seq 从 01 开始连续递增。digest 从
`memory_id` 本身解析阶段号。

`scene_refs` 指向 scene_archive 中对应的场景 scene_id，形成从主观记忆到
原文的追溯链。非必填——有些记忆可能跨多个场景或无法精确对应。

#### 12.4.3 与 stage_snapshot 的分工

- stage_snapshot 的 `stage_events` 是**仅本阶段**发生事件的清单
  （string[]，每条 50–80 字的一句话），不累积历史、不展开过程
- memory_timeline 提供每件事的主观体验细节（全历史累积）
- 历史事件跨阶段视图由 `world_event_digest.jsonl` 程序化累积维护
- 三者不冗余——`stage_events` 是本阶段聚合结论、memory_timeline 是
  角色经历过程、world_event_digest 是跨阶段客观事件索引

#### 12.4.4 memory_digest（远期记忆摘要）

**定位**：memory_timeline 的压缩索引，供启动时加载，让 LLM 感知全历史
记忆的存在和大致内容，判断是否需要通过 FTS5 检索详情。

**存储位置**：
```
works/{work_id}/characters/{character_id}/canon/memory_digest.jsonl
```

**生成方式**：从 memory_timeline 全量条目自动生成（提取完成后或每阶段
追加）。每条 memory_timeline 条目对应一条 digest 条目。`summary` 直接复制
LLM 在 memory_timeline 中专门撰写的 `digest_summary` 字段，**不是**从
`event_description` 截断得到（见 §12.4.2）。

**条目结构**：
```json
{
  "memory_id": "M-S001-01",
  "time": "故事开始之时",
  "location": "起始地点",
  "summary": "角色A在某地苏醒并遭遇意外事件（30–50 字）",
  "importance": "significant"
}
```

**字段说明**：
- `memory_id`：直接复制 memory_timeline 的 `memory_id`；阶段号由
  `S###` 段解析，不再冗余存 `stage_id`
- `time`、`location`：直接从 memory_timeline 条目复制
- `summary`：直接复制 memory_timeline 的 `digest_summary`（LLM 专写的
  30–50 字 digest 摘要）。长度下限与上限均由 `memory_timeline_entry.schema.json`
  硬校验，digest 阶段不做任何截断或扩写
- `importance`：直接复制 memory_timeline 的 `memory_importance`，取值
  为 trivial / minor / significant / critical / defining 五级

**不包含**：
- `stage_id`、`emotional_tags`、`involved_targets`（历史冗余字段，
  节省启动 tokens——情绪与关系线索通过 FTS5 拉 timeline 全文获取）
- `subjective_experience`、`knowledge_gained`、`misunderstanding`、
  `concealment` 等详情字段——由 FTS5 按需检索

**每条约 30-40 tokens**（较旧版 ~70 tokens 压缩 ~50%）。

**加载规则**：
- 启动时按 stage 范围过滤加载（stage 1 到用户选定 stage N），
  不加载未来阶段的条目以避免知识泄露
- 过滤通过 `memory_id` 的 `S###` 段判定阶段归属
- 最近 2 个阶段的条目可跳过（已由 memory_timeline 全文覆盖）

#### 12.4.5 world_event_digest（世界事件摘要）

**定位**：世界级重要事件的增量索引，与 memory_digest 对称。memory_digest
记录角色主观记忆，world_event_digest 记录客观世界事件。

**存储位置**：
```
works/{work_id}/world/world_event_digest.jsonl
```

**生成方式**：从世界 stage_snapshot 的 `stage_events` 字段自动生成
（提取完成后程序化追加，见 §11.3a）。`stage_events` 条目本身即 50–80 字的
一句话摘要，作为 digest 条目的唯一来源。digest 只做 1:1 机械展开，**不做
过滤**——世界 vs 角色层边界在提取与语义审校阶段把控。

**条目结构**：
```json
{
  "event_id": "E-S001-01",
  "time": "故事时间锚点",
  "location": "事件发生地点",
  "summary": "某个世界级重大事件发生（50–80 字）",
  "importance": "significant",
  "involved_characters": ["角色A", "角色B"]
}
```

**字段说明**：
- `event_id`：`E-S{stage:03d}-{seq:02d}`，阶段号由 `S###` 段承载，
  不再冗余存 `stage_id`。按 `stage_events` 数组顺序连续编号
- `summary`：直接复制 `stage_events[i]`，50–80 字（schema 两端硬门控）
- `importance`：程序从 `stage_events[i]` + `current_world_state` 等
  上下文映射到五级（trivial..defining）；无法判定时默认 `significant`
- `time`、`location`、`involved_characters`：从 `timeline_anchor` 和
  `stage_events[i]` 文本中最佳匹配（可为空字符串 / 空数组）

**内容边界（世界层 vs 角色层）**：

只记录**世界公共层事件**。判断准则：事件对世界背景的长期状态有持续影响、或
涉及多个角色共同参与的公共事件。例：

| 属世界层（写入 stage_events） | 属角色层（写入角色 memory_timeline，不写世界） |
|---|---|
| 势力格局变迁（组织覆灭、联盟成立） | 角色之间的对话、误会、单次互动 |
| 地形 / 资源显著变化 | 某角色的经济活动或日常消费 |
| 世界规则首次被揭示 | 某角色的内心决定、情绪变化 |
| 跨角色的公共事件（战役、典礼） | 一方告知另一方的私人信息 |

过滤在提取 prompt 与 world 语义审校通道共同把控（见 §11.3a 的"世界事件
过滤"小节），digest 本身不做剔除，确保 `event_id` 与 `stage_events` 顺序
一致、连续。

**加载规则**：
- 启动时按 stage 范围过滤加载（stage 1 到用户选定 stage N）
- 过滤通过 `event_id` 的 `S###` 段判定阶段归属
- 不加载未来阶段的条目以避免知识泄露
- 提供跨阶段世界事件时间线的轻量全景

### 12.5 第三层：scene_archive（原文场景档案）

**定位**：小说原文按场景切分的检索库——"原文里到底怎么写的"

**存储位置**：
```
works/{work_id}/retrieval/scene_archive.jsonl
```

不提交 git（文件过大，.gitignore）。运行时检索数据库 `fts.sqlite` 同目录。
轻量索引可提交至 `works/{work_id}/indexes/`。

**粒度**：每个自然场景一条

#### 12.5.1 切分规则

- 以场景为自然切分单位，不按固定字数
- **一个场景不跨章节边界**——一章内可有多个场景，但不允许跨到下一章
- `stage_id` 由章节号查 `stage_plan.json` 得出，不需要 LLM 判断。
  `stage_plan.json` 是 scene_archive `stage_id` 的**唯一真源**，
  Phase 1 重生 stage plan 后必须整体重算 scene_archive 的 `stage_id` 和
  `scene_id`
- 场景篇幅无限制，由原文自然决定

#### 12.5.2 条目结构

```json
{
  "scene_id": "SC-S{stage:03d}-{seq:02d}",
  "stage_id": "{stage_id}",
  "chapter": "{chapter_identifier}",
  "time": "故事内时间",
  "location": "场景发生地点",
  "characters_present": ["{character_id}", "..."],
  "summary": "该场景的事件梗概（客观第三人称简述；不在启动加载，仅供 FTS5 检索索引）",
  "full_text": "完整场景原文，包含对话和叙述，无字数限制"
}
```

不对场景内的对话做逐句拆分或标注。`time` 字段（故事内时间）与
memory_timeline / world_event_digest 命名统一。

#### 12.5.3 存储位置

scene_archive 是作品级资产，不按角色拆分（同一场景可能涉及多个角色）。

#### 12.5.4 加载策略

1. **启动时加载 full_text**：当前阶段附近、与选定角色相关的最近 M 个
   场景的原文，用于精细推演和语言风格锚定。M 由运行时配置
   `scene_fulltext_window` 控制（**默认 10**）
2. **summary 不进入启动 prompt**：`summary` 字段仅作为 FTS5 / embedding
   检索的索引字段使用（表 / 索引中保留，prompt 不注入），避免启动 context
   随阶段数线性膨胀
3. **对话中按需检索**：FTS5 / embedding 命中 → 返回 `scene_id` + summary
   片段 → 按需取 `full_text` 或通过 `scene_refs` 反查 memory_timeline

角色过滤规则：模拟角色 A 时——
- full_text 启动加载：`characters_present` 包含角色 A 的场景中，
  按章节号距离当前阶段最近的 M 条
- FTS5 检索范围：`characters_present` 包含角色 A 的全部场景（不限阶段）
- 不加载：角色 A 不在场的场景（启动和检索均跳过）

#### 12.5.5 提取阶段

scene_archive 在 **Phase 4**（独立阶段）提取，与 Phase 3 数据独立：
- 前置条件仅需 `stage_plan.json`（Phase 1 产物），不依赖 Phase 3 产出
- 使用独立 PID 锁 `.scene_archive.lock`；中间目录
  `analysis/scene_splits/` 为本地忽略产物，可在断点恢复时复用
- 每章一次 LLM 调用：读原文 → 标注场景边界和元数据（不含 full_text）
- 程序根据行号从原文提取 full_text，补充 scene_id 和 stage_id
- 多章并行执行（`--concurrency` 控制，默认 10）
- 仅程序化校验（行号有效、不重叠、覆盖全章），不做语义审校
- 详见 §11.11

#### 12.5.6 ID 命名规范（统一）

memory_digest / world_event_digest / scene_archive 的主键 ID 采用统一
格式 `{TYPE}-S{stage:03d}-{seq:02d}`：

| 对象 | 前缀 | 示例 | 说明 |
|------|------|------|------|
| memory_timeline / memory_digest | `M` | `M-S001-01` | 每阶段 seq 从 01 起递增 |
| world_event_digest | `E` | `E-S001-01` | 对应 `stage_events[i]` 序号 |
| scene_archive | `SC` | `SC-S001-01` | 同一 stage 内按章节号 + 章内 scene 序递增 |

设计要点：
- **3 位阶段号**（`S###`）支持 ≤ 999 个阶段，覆盖长篇连载
- **2 位 seq**（`##`）每阶段上限 99，满足：
  - memory：每阶段记忆条目经验上 <40
  - world event：`stage_events` 每阶段通常 <20
  - scene：单 stage 场景数通常 <50（§12.5.1 规定场景不跨章节）
  - 若某阶段 seq 超出 99，提取期校验失败 → 需要拆分阶段或合并事件
- **阶段号由 ID 直接承载**：digest 条目不再冗余存 `stage_id`，loader
  按 stage 过滤时用正则 `S(\d{3})` 解析 ID 判断归属
- **ID 是主键**：upsert 时以 ID 为唯一键，任何重跑都幂等

提取端必须保证 ID 格式正确；运行时 loader 对不匹配 ID 的条目直接跳过
并记录 WARNING（不阻断启动）。

### 12.6 两级检索漏斗

scene_archive 和 memory_timeline 的运行时检索采用**两级漏斗**设计，
优先使用零延迟的轻量方案，仅在必要时升级到语义检索。

#### 12.6.1 第一级：jieba 分词 + 专有名词表 + FTS5（默认，<20ms）

每轮对话都执行，延迟可忽略：

1. **jieba 分词**：对用户输入进行中文分词
2. **专有名词表匹配**：将分词结果与作品级专有名词表比对，
   只保留命中的关键词。名词表在提取阶段自动产出，包含：
   - 人名（含别名、称呼）
   - 地名（宗门、城市、秘境等）
   - 功法 / 物品 / 阵法名
   - 事件关键词（大比、渡劫、结丹等）
3. **FTS5 查询**：用命中的关键词查询 SQLite FTS5 索引，
   按 BM25 排序取 top-K 条 summary（不是 full_text），
   再加 metadata 过滤（`stage_id` / `characters_present` /
   `importance`）缩小范围

**无命中时不检索**：如果没有任何词命中名词表，本轮不触发检索，
直接用已加载的上下文生成回复。

**FTS5 候选 + LLM 精筛**：FTS5 返回的候选摘要直接塞进主 LLM 调用
的 prompt 中，由 LLM 自行判断哪条相关、哪条丢弃。不额外调用 LLM。

示例：

```
用户说："我最近修炼总是突破不了"
jieba 分词：["我", "最近", "修炼", "总是", "突破", "不了"]
名词表命中："修炼"、"突破"
FTS5 查询：memory_fts MATCH '修炼 OR 突破' → top-10 summary
LLM 看到候选后，自行挑选相关记忆融入回复
```

#### 12.6.2 第二级：Embedding 语义检索（兜底，200-300ms）

仅在第一级无法满足时触发。使用 **LLM tool use** 机制：

- 引擎为 LLM 定义 `search_memory` 工具
- 当第一级候选不足以回答用户问题时，LLM 主动调用该工具，
  传入语义化的检索描述（如"确认关系之前两人的互动记忆"）
- 引擎执行 embedding 近似搜索（对 summary 的向量），返回结果
- LLM 基于新结果生成最终回复（第二次 LLM 调用）

tool use 定义示例：

```json
{
  "name": "search_memory",
  "description": "检索角色的历史记忆和原文场景。仅在候选记忆不足时使用。",
  "input_schema": {
    "properties": {
      "query": {
        "type": "string",
        "description": "要检索的内容描述"
      }
    },
    "required": ["query"]
  }
}
```

**绝大多数轮次在第一级就结束**，只有一次 LLM 调用。
第二级极少触发，延迟是按需递增的。

#### 12.6.3 角色主动联想

角色不应是被动回答问题的助手——当对话情境与某段记忆相关时，
角色应像真人一样自然地主动提起。

**实现方式**：引擎每轮不只分析用户输入，还从当前 context state
提取情境关键词（当前地点、近期事件、当前情绪），一并送入 jieba +
FTS5，让候选池稍宽。LLM 自行判断是否在回复中自然提起。

示例：

```
当前地点 = "天剑宗·演武场"（从 context state 提取）
jieba 匹配 "演武场" → FTS5 命中一条记忆：
  "小时候第一次在演武场被师兄打哭"
LLM 看到候选后，在回复中自然带出：
  "说到演武场……我小时候第一次来这里，被师兄打得鼻青脸肿。"
```

情境关键词来源：

- `context.current_location` — 当前对话场景地点
- `context.recent_events` — 近期发生的事件
- `context.current_emotion` — 角色当前情绪状态
- `context.conversation_partner` — 对话对象

**不额外调用 LLM，不增加延迟**。只是多提取几个情境关键词，
扩大 FTS5 候选范围，LLM 自主决定是否提起。

联想触发条件（在 system prompt 中指导角色）：

- 对方的某个动作 / 表情 / 语气让角色想起了某个人或某件事
- 当前场景（地点、天气、氛围）和过去某个经历相似
- 对方提到了某个话题，角色有相关的亲身经历
- 对方的某个行为和角色记忆中的他 / 她不同，引发感慨
- 角色心里一直在意某件事，在合适的时机终于提起

#### 12.6.4 对话语境驱动检索

综合第一级和第二级，引擎每轮的检索决策流程：

```
用户输入 + 当前情境
    │
    ▼
jieba 分词 + 名词表匹配（<10ms）
    │
    ├─ 命中 → FTS5 查询 top-K summary（<10ms）
    │         │
    │         ▼
    │     候选摘要塞入主 LLM 调用 prompt
    │     LLM 自行判断：
    │       ├─ 候选够用 → 直接生成回复（一次调用，结束）
    │       └─ 候选不足 → 调用 search_memory 工具
    │                      → 引擎执行 embedding 检索
    │                      → 结果传回 LLM → 生成回复（两次调用，结束）
    │
    └─ 未命中 → 不额外检索，直接生成回复（一次调用）
               （已加载的 stage_snapshot + 近期记忆 + memory_digest + world_event_digest 仍在上下文中）
```

#### 12.6.5 与现有加载层级的关系

检索不替代现有的分层加载策略（§7.2），而是作为补充：

- Tier 0（启动核心层）：增加 memory_timeline 预加载、scene_archive
  最近 M 个 `full_text` 预加载（M 由 `scene_fulltext_window` 控制，
  默认 10）、**专有名词表加载**。scene_archive 的 `summary` 字段
  **不进入启动 prompt**，仅作为 FTS5 索引字段
- Tier 1（结构化按需层）：scene_archive 和 memory_timeline 的
  FTS5 + 可选 embedding 检索归入此层；命中后按需取 `full_text`
- Tier 3（原文验证层）：scene_archive 的 `full_text` 可直接作为
  原文验证来源

### 12.7 运行时 Prompt 组装

LLM 不直接"加载文件"。模拟引擎从各层数据中选取当轮相关内容，拼接为
一个 prompt 发送给 LLM。所有加载和检索策略的核心约束是 context window 的
token 上限。记忆系统各层的加载量、检索条数、预加载范围都应在此预算内合理
分配。

LLM 的 system prompt 中应包含：

- **记忆检索指导**：告诉 LLM 有哪些候选记忆可用，以及 `search_memory`
  工具的使用时机和限制
- **主动联想指令**：告诉角色在什么情境下应自然联想过去的经历
- **候选记忆区域**：每轮动态填入的 FTS5 候选摘要

### 12.8 技术选型

- **中文分词**：`jieba`（加载作品级自定义词典，匹配速度 <10ms）
- **全文检索**：`sqlite FTS5`（BM25 排序，覆盖两库的 lexical 检索层，
  **默认主力方案**）
- **Embedding 模型**（可选兜底）：本地运行 `bge-large-zh-v1.5`
  （中文优化，约 1.3GB 显存），仅在 FTS5 不足时通过 tool use 触发
- **数据库**：**单个 SQLite 文件**统一存储 FTS5 索引和可选的
  embedding 向量列，无需额外的向量数据库进程

不使用独立的向量数据库（如 chromadb、faiss）。所有检索能力由
一个 SQLite 文件提供。Embedding 向量以 BLOB 列存储在同一数据库中，
仅作为可选的第二级兜底。
