# 最小结构读取入口

适用场景：

- 一个全新的 agent 即将开始处理 `persona-engine`
- 当前 prompt 已经说明了任务目标，但还没有系统确认“要写到哪里、这些目录各自负责什么”

用途：

- 让新 agent 不必通读整个项目，也不要靠猜目录开工
- 在正式读正文、做抽取、做运行时装载或写回前，先读取最小但足够的结构说明

核心原则：

1. 不需要先理解整个项目框架。
2. 但在写任何目录或结构化文件前，必须先确认目标目录的职责边界。
3. prompt 里已经写明的输出路径优先；结构说明用于校验和防止自创目录，而不是推翻当前 prompt。
4. 如果发现 prompt、现有目录、现有文件三者之间有冲突，先停下，不要继续猜。

最小读取规则：

1. 任何任务开始前，先读：
   - `README.md`
   - `ai_context/project_background.md`
   - `ai_context/current_status.md`
2. 如果任务涉及目录边界、运行时、schema、工作流或长期项目规则，再补读：
   - `ai_context/decisions.md`
   - 必要时：`ai_context/architecture.md`
3. 如果任务涉及运行时启动、装载、检索、持续写回或收尾合并，再补读：
   - `simulation/README.md`
   - 必要时：`simulation/retrieval/load_strategy.md`
   - 必要时：相关 `simulation/flows/*.md`
4. 如果要触碰 `works/{work_id}/...`，再读：
   - `works/README.md`
   - 如果存在：`works/{work_id}/manifest.json`
   - 如果存在：`works/{work_id}/analysis/progress/pipeline.json`
5. 如果要触碰 `users/{user_id}/...`，再读：
   - `schemas/context_manifest.schema.json`、`schemas/role_binding.schema.json`
     （枚举与必填字段以 schema 为准）
   - `users/_template/` 的目录骨架与最小清单文件（真源）
   - 上述确认后再读 `users/README.md` 作为设计层面的补充说明
   - 如果存在：`users/{user_id}/role_binding.json`
6. 如果要创建、更新或 review 结构化 JSON / JSONL / schema 相关文件，再读：
   - `schemas/README.md`
   - 对应 schema 文件
7. 如果你仍然不确定某一类目录、文件或包之间的关系，再补读：
   - `docs/architecture/data_model.md`
   - 必要时：`docs/architecture/system_overview.md`

就近读取规则：

1. 如果你要写某个具体目录，先看该目录下现有文件长什么样。
2. 如果你要新建某类文件，先看同层已有同类文件。
3. 如果你要更新某个 package，先看这个 package 的 `manifest.json`。
4. 如果已有相邻 README、analysis 说明或 progress 文件，优先读最近的一层，不要一口气把全仓库都扫一遍。
5. 如果已有当前对象的 extraction status / stage progress，先读状态文件，再决定是否需要继续读原文或更早阶段。
6. 如果任务属于 runtime，先读 `simulation/` 的总入口和相关子流程，再决定要不要扩读整份架构文档。

禁止事项：

1. 不要在没读结构说明前就自创新的顶层目录。
2. 不要在没确认边界前就把用户状态写进 `works/`。
3. 不要在没确认边界前就把作品 canon 写进 `users/`。
4. 不要因为当前 prompt 没写全，就假设“这个目录大概也能放”。

推荐心法：

1. 新 agent 不需要先“懂整个项目”。
2. 新 agent 仍应先拿到最小的项目长期记忆，而不是完全绕开 `ai_context/`。
3. 新 agent 需要先“知道这次任务应该落在哪一层、每层大致负责什么”。
4. 也就是说，默认策略不是“全项目通读”，而是“最小 `ai_context` + 最小结构读取 + 当前任务 prompt + 就近样例核对”。
