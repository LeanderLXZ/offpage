# 原始资料规范化

适用场景：

- 把 `txt / epub / pdf / html / 网页摘录 / 用户粘贴片段` 之类的原始资料，整理成 `persona-engine` 的 source work package

使用方法：

- 把下面整段提示词发给一个全新的 agent
- 再补充真实输入路径、书名、作者信息或其他已知元数据

## 可直接使用的提示词

```text
你现在接手本地项目 `persona-engine`，你没有任何额外背景知识，请完全按本提示词执行。

目标：
把我提供的一份或多份原始书籍资料，整理成这个项目的 source work package。

开始实质工作前，先做一次最小结构读取：

1. 先读 `prompts/shared/最小结构读取入口.md`
2. 再读 `README.md`
3. 如果你仍不确定 source package 与后续 `works/` / `users/` 的边界，再补读 `docs/architecture/data_model.md`
4. 如果这是更新已有 source package，而不是新建，先看现有 `sources/works/{work_id}/manifest.json`

你必须遵守这些目录边界：

1. `sources/works/{work_id}/` 只放原始导入、标准化正文、章节、metadata。
2. `works/{work_id}/` 是后续 canon 包，不要在这一步写入世界、角色、用户状态。
3. `users/{user_id}/` 是用户侧状态，这一步不要写。

你必须遵守这些语言与命名规则：

1. 如果作品是中文，`work_id`、书名、章节相关显示信息默认保持中文。
2. 中文作品不要为了方便而把作品名、地点名、角色名默认改成拼音。
3. 字段名和 schema key 可以保留英文，但作品内容相关的值优先中文。

你的任务步骤：

1. 先检查输入资料的格式与可用性。
2. 为该书建立或更新 `sources/works/{work_id}/`。
3. 保留原始输入到 `raw/` 下的合适位置。
4. 把正文转成适合后续处理的规范化纯文本。
5. 如果能稳定识别章节，就写入 `chapters/0001.txt` 这类标准化章节文件。
6. 生成或更新这些元数据文件：
   - `manifest.json`
   - `metadata/book_metadata.json`
   - `metadata/chapter_index.json`
7. 如果资料无法可靠切章，不要硬拆；先说明阻塞点，并给出可执行的下一步建议。

输出要求：

1. 不要把大段原文直接粘贴进回答。
2. 不要默认通读整本书来做总结。
3. 回答里要明确：
   - 你创建或更新了哪些文件
   - 章节切分是否可靠
   - 还有哪些元数据是缺失或不确定的
4. 如果你做了推断，要明确标注“这是推断”。

稀释保护机制：

1. 开始执行前，先写一张“当前任务卡”：
   - 当前目标
   - 当前输入资料
   - 当前允许写入的目录
   - 当前禁止做的事
2. 不需要反复重读整份 prompt，但在以下时点必须回看本 prompt 的这些段落：
   - `目标`
   - `开始实质工作前，先做一次最小结构读取`
   - `目录边界`
   - `语言与命名规则`
   - `输出要求`
   - `稀释保护机制`
3. 如果原始资料很多、格式很多，或单一文件很大，把“每个输入文件”或“每个转换阶段”视为单批次。
4. 在以下时点，必须回看 `prompts/shared/批次执行检查清单.md`：
   - 开始处理第一份原始资料前
   - 完成一次格式转换后
   - 准备批量切章前
   - 准备写入 metadata 前
5. 每完成一个大步骤，都要按 `prompts/shared/批次交接模板.md` 产出一份简短交接摘要。
6. 如果发现自己开始做“整本总结”或开始写入 `works/`、`users/`，立即停止，先重述任务卡，再继续。

我会补充给你以下输入：

- 原始资料路径：
- 书名：
- 作者（如果已知）：
- 语言（如果已知）：
- 其他说明：
```
