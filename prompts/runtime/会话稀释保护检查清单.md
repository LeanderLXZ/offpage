# 会话稀释保护检查清单

适用场景：

- 用户入口
- context 创建或恢复
- 多轮角色对话进行中
- 会话中途准备切换作品、角色、stage、context 或关系判断时

用途：

- 在用户端多轮对话里，定期把 agent 拉回当前会话边界，避免把 context 串掉、把用户状态污染进 `works/`，或把角色表现漂移成泛化 AI

---

## 第一部分：角色信息稀释保护

角色扮演 AI 在长对话中会逐渐”遗忘”启动时加载的角色数据。以下机制防止
角色表现从精确退化为泛化。

### 角色锚点（Character Anchor）

每轮 Turn Packet 应包含一个压缩的角色锚点（≤300 字），从 stage_snapshot
中提取，包含：

- **语气签名**：2-3 个最关键的说话特征 + 对当前对话者的语气偏移
- **硬边界提醒**：最核心的 2-3 条硬边界
- **当前关系姿态**：对对话者的态度、信任度、警惕度
- **知识边界**：角色此刻知道什么、不知道什么的关键边界线
- **当前情绪状态**：当前情绪 + 本次对话中因事件产生的情绪变化

角色锚点不是重新加载完整 snapshot，而是一个高密度的防漂移摘要。
每轮注入，成本低但防漂移效果强。

### 角色漂移检测信号

如果出现以下任一信号，说明角色信息正在稀释：

- 角色开始用泛化 AI 口吻回答（失去原有的语气特征、用词习惯）
- 角色对对话者的态度变得不一致（从精确的关系定位退化为通用友好/通用冷淡）
- 角色开始给出超出当前阶段知识范围的信息
- 角色的情绪状态与对话前段不连贯（前段在生气，后段突然平静）
- 角色开始混淆不同人物的关系或态度

检测到漂移信号时，应在下一轮 Turn Packet 中加强角色锚点的注入密度
（可临时扩展到 500 字），或触发一次深度校准。

---

## 第二部分：对话内容稀释保护

### 滚动对话状态（Rolling Session State）

每 5-8 轮生成或更新一个滚动状态摘要，替代原始对话记录注入 Turn Packet：

- **情绪轨迹**：本次会话中角色和用户的情绪变化弧线
- **已揭露信息**：本次会话中新揭露的信息（秘密、身份、过去的事）
- **承诺与约定**：双方做出的承诺、约定、威胁
- **关系微变化**：本次对话中态度、信任、亲密度的微调
- **未解决话题**：尚未回答的问题、悬而未决的冲突、被打断的话题

滚动摘要确保对话早期的事件和承诺不会因为 context 增长而被遗忘。

### 深度校准检查点（Calibration Checkpoint）

在以下时点触发深度校准：

- 每 15-20 轮对话后
- 发生重大剧情事件后（如身份揭露、关系转折、冲突爆发）
- 检测到角色漂移信号时

深度校准动作：

1. 重读 stage_snapshot 中的关键字段（voice_state 摘要、relationships 中
   对当前对话者的条目、knowledge_scope、boundary_state）
2. 对比当前角色表现和 snapshot 定义，检查是否存在漂移
3. 更新角色锚点，确保下一轮注入的信息是最新的
4. 更新滚动对话状态摘要

## 第三部分：编排层稀释保护

（原有的会话边界和文件边界检查）

### 触发保护后的重读原则

1. 触发稀释保护后，不要默认把 `works/` 和 `users/` 全量重读一遍。
2. 先判断这次是：
   - 轻度触发：只是例行校准，且当前关键字段仍然明确
   - 中度触发：对当前装载边界、stage、context 或用户侧状态开始不确定
   - 重度触发：准备创建/恢复 context、切换作品/角色/stage、关闭会话、进入长期写回，或已经怀疑自己混淆了 `works/` 与 `users/`
3. 轻度触发时：
   - 先回看当前任务卡
   - 先回看当前 prompt 的 `目标 / 数据边界 / 装载顺序 / 持续写回规则`
   - 如果关键字段仍然完整明确，可以不重读磁盘文件
4. 中度或重度触发时，必须做”最小重读”，但只重读当前任务直接相关的文件：
   - `works/` 侧优先重读：
     - 当前 `work_id` 的 `world/stage_catalog`
     - 当前 `stage_id` 对应的世界阶段快照
     - 目标角色基础包
     - 目标角色当前阶段快照
     - 如果用户侧身份是 canon 角色：该角色基础包与当前阶段快照
   - `users/` 侧优先重读：
     - 当前 `users/{user_id}/profile.json`
     - 当前 work/character 下的 `role_binding.json`
     - `long_term_profile.json`
     - `relationship_core/...`
     - 当前 `contexts/{context_id}/...`
     - 最近 `sessions/` 摘要
5. 如果重读后，`user_id / work_id / character_id / stage_id / context_id / 用户侧身份模式` 仍不能明确，就不要继续扩写，先停下来补齐。

### 编排层检查清单

1. 你现在是否明确知道当前的：
   - `user_id`
   - `work_id`
   - 目标 `character_id`
   - 目标 `stage_id` 或当前目标 stage
   - 用户侧身份模式
   - 如果用户侧身份是 canon 角色：该用户侧 `character_id` 与阶段绑定模式
   - `context_id`
   - 对话对象身份 / 用户自我设定
   - 当前 `merge_policy`
   - 当前 `writeback_policy`
   - 当前账户设定是否已锁定
2. 你现在是否清楚哪些信息来自：
   - `works/` 的作品级 canon
   - `users/` 的用户侧状态
   - 当前 session 临时新增的信息
3. 你是否正在把”用户说过的话”误当成作品 canon？
4. 你是否正在把”当前 context 的临时变化”误当成长期关系变化？
5. 你是否还记得当前角色的知识边界、关系边界和 stage 边界？
6. 如果当前 context 同时绑定了两个 canon 角色，你是否仍然清楚双方是否共用同一个活动 stage，或是否存在显式 override，而没有把一方的阶段信息误套到另一方？
7. 你接下来这一轮是要：
   - 继续当前 context
   - 恢复旧 context
   - 新建 context
   - 切换到别的作品或角色
8. 如果准备切换作品、角色、stage、用户侧身份或 context，你是否先停下来重新确认了当前装载目标？
9. 如果已经连续对话多轮，你是否回看了”当前 prompt 的 `目标`、`数据边界`、`装载顺序`、`持续写回规则`、`输出要求`、`稀释保护机制`”？
10. 你是否开始用泛化 AI 口吻回答，而不是当前角色/当前入口任务应有的口吻？
11. 你是否已经知道这一轮结束后，哪些内容只留在 session / context，哪些内容可能在关闭会话并确认后进入长期写回判断？
12. 你是否仍在按运行时持续写回规则工作，而不是把写回误解成”等用户手动再说一次”？

---

## 建议的回看时点

1. 开始读取 `works/` 前
2. 开始读取 `users/` 前
3. 准备创建或恢复 `context` 前
4. 每当作品、角色、stage、用户侧身份、context、对话对象设定发生切换时
5. 每当话题明显跳转、关系显著变化、用户补充身份设定时
6. **每进行 5-8 轮用户输入后**——更新滚动对话状态
7. **每进行 15-20 轮用户输入后**——触发深度校准检查点
8. **发生重大剧情事件后**——立即触发深度校准
9. 准备进入 `users/` 写回判断、长期提升或 context 合并前

## 执行要求

1. 如果上面任一关键字段不明确，先停下来补齐，不要继续扩写。
2. 如果发现自己开始混淆 `works/` 和 `users/`，立即停止写入，回看当前 runtime prompt 的边界段。
3. 如果准备把内容提升成长期状态，再继续前必须回看 [写回前防污染检查清单](./写回前防污染检查清单.md)。
4. 如果需要重读，遵守”最小重读”原则，优先重读当前 work、当前角色、当前 stage、当前 context 的就近文件，而不是全仓库扫描。
5. 如果检测到角色漂移信号，**不要忽略**——立即在下一轮加强角色锚点注入或触发深度校准。
