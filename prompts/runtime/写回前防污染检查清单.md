# 写回前防污染检查清单

适用场景：

- 准备把一次用户互动的结果写入 `users/{user_id}/...`
- 准备把内容提升成长期关系变化、长期记忆或 pinned memory

用途：

- 在真正落盘前，检查这次写回是否过度、是否写错层、是否把用户影响污染进作品级 canon

检查清单：

1. 这次准备写入的内容，是否全部属于 `users/{user_id}/...`？
2. 你是否正在尝试修改：
   - `works/{work_id}/world/`
   - `works/{work_id}/characters/{character_id}/`
   - 任何作品级 canon 文件
3. 这条信息属于哪一类：
   - 当前 session 临时摘要
   - 当前 context 的共享状态
   - 用户 work-scoped 长期画像变化
   - 用户与角色的长期关系变化
   - pinned memory
   - context 提升或 full merge
   - 仍不足以下结论，不应写入
4. 这条信息是否真的稳定、可复用、对未来对话重要？
5. 这条信息是否只是一次情绪波动、一次口头玩笑、一次未确认设定？
6. 如果你准备更新长期关系或 pinned memory，是否有足够明确的对话证据支持？
7. 如果你准备把整个 context 或 context 的核心内容并入长期状态，是否已经明确：
   - 合并范围
   - 合并原因
   - 哪些内容仍应只留在原 context
   - 当前 context 合并后要如何标记生命周期
8. 这次写入是否会让未来 agent 误以为“作品 canon 已经改变”？
9. 这次写入后，是否还能清楚区分：
   - canon
   - `long_term_profile`
   - 用户侧长期状态
   - 当前 context 临时状态
   - 当前 session 临时状态
10. 你是否已经回看了“当前 prompt 的 `目标`、`边界`、`写回原则`、`输出要求`、`稀释保护机制`”？
11. 你是否已经决定每一条信息的落点：
   - `long_term_profile`
   - `relationship_core`
   - `pinned_memories`
   - `contexts/{context_id}`
   - `sessions/{session_id}`
   - 不落盘
12. 你是否记得“持续写回”默认先写 `session / context`，而不是每一轮都冲进 `long_term_profile` 或 `relationship_core`？

建议的回看时点：

1. 开始判断本轮是否需要写回前
2. 准备把信息提升成长期关系变化前
3. 准备写入 pinned memory 前
4. 准备执行 context 提升或 full merge 前
5. 准备正式落盘前
6. 一旦发现自己想把用户影响写回 `works/` 时

执行要求：

1. 如果不能明确证明某条信息应进入长期状态，默认降级为 context / session 级，或暂不写入。
2. 如果发现自己准备改写 `works/`，立即停止，并回看当前 runtime prompt 的边界段。
3. 写回完成后，应产出一份简短说明：
   - 写了什么
   - 为什么写
   - 为什么没有把其他内容升级为长期状态
4. 如果执行了 context 提升或 full merge，还应说明：
   - 合并了什么
   - 为什么这些内容值得进入长期状态
   - 当前 context 的生命周期是否已更新
