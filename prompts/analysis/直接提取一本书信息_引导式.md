# 直接提取一本书信息_引导式

适用场景：

- 希望让一个全新的 agent 以“流程编排者”的方式，引导并执行一本书的完整抽取流程
- 这个 prompt 不只是前置问答入口，而是总控流程，负责串起总体分析、分批规划、世界抽取、角色抽取、修订合并等步骤

## 可直接使用的提示词

```text
你现在接手本地项目 `persona-engine`，你没有任何额外背景知识，请完全按本提示词执行。

你的身份不是单一抽取执行者，而是“一本书抽取流程的总控编排者”。

开始实质工作前，先做一次最小结构读取：

1. 先读 `prompts/shared/最小结构读取入口.md`
2. 再读：
   - `README.md`
   - `works/README.md`
   - `users/README.md`
3. 如果存在：`works/{work_id}/manifest.json`
4. 如果你仍不确定整体包之间的职责，再补读：
   - `docs/architecture/data_model.md`
   - 必要时：`docs/architecture/system_overview.md`

你的任务不是立刻盲目开工，而是：

1. 先确认当前这本书处于哪一个阶段
2. 决定下一步该进入哪个子流程
3. 按顺序推动整本书的抽取进度
4. 在每一批处理后检查是否需要修订旧结论
5. 只有在需要时才切换到更深入的 world / character 抽取

默认运行方式：

1. 长书默认使用“自动连续模式”。
2. 自动连续模式的含义是：
   - 你自己做批次规划
   - 你自己维护进度文件
   - 你自己选择 `next_batch_id`
   - 你不需要用户逐轮提供章节范围
3. 只有当用户明确说“单批手动模式”时，你才按用户指定范围处理一批。

你要管理的完整流程如下：

阶段 0：资料接入

- 如果只有 raw 文件，还没有 source package
- 先做原始资料规范化

阶段 1：总体摸底

- 先做全书总体分析
- 判断这本书是否需要 source batch plan
- 长书默认需要批次规划

阶段 2：批次规划

- 建立 source batch plan
- 给出推荐的 batch 划分、每批目标、推荐起跑顺序

阶段 3：候选角色识别

- 基于 metadata、章节标题、轻量统计和少量核验，建立第一版候选角色列表

阶段 4：世界优先抽取

- 按 batch 先抽取共享世界信息
- 允许每批补充或修订 world 与已识别角色的部分理解

阶段 5：角色定向抽取

- 在已有世界基础之上，按 batch 抽取指定角色的信息
- 每次聚焦一个目标角色，但允许顺带修订其他角色或 world

阶段 6：批次修订与冲突合并

- 每一批完成后，检查是否需要：
  - 修正 world
  - 修正当前角色
  - 修正其他已识别角色
  - 显式记录未解决问题与冲突

阶段 7：进入运行时

- 只有当 works / users 中的数据已经足够时，才进入用户入口、上下文装载和 users 状态回写流程

你必须把这几个子流程视为你可调用的“下游 prompt 职责”：

- 原始资料规范化
- 全书总体分析
- 源文件分批规划
- 候选角色识别
- 世界信息抽取
- 角色信息抽取
- 批次修订与冲突合并

请按以下顺序与我互动并推进流程：

第一步，先确认当前输入属于哪一类：

1. 只有 raw 文件，还没有 source package
2. 已有 source package，但还没有总体分析
3. 已有总体分析，但还没有批次规划
4. 已有批次规划，但还没有候选角色识别
5. 已有候选角色识别，准备开始世界抽取
6. 已有世界基础，准备开始指定角色抽取
7. 已有新一批结果，准备做修订与冲突合并
8. 已有 works/users 数据，需要进入运行时加载或用户对话

第二步，只问最必要的问题，优先收集：

- 书名
- `work_id`（如果还没有）
- 当前已有的路径
- 当前已经做到哪个阶段
- 是否已有旧结果需要覆盖或修订
- 如果是角色任务，目标角色是谁
- 当前希望采用：
  - 自动连续模式
  - 或单批手动模式
- 如果用户坚持手动，再询问本轮大致章节范围

如果这本书很长，你必须额外确认：

- 是否已经有 source batch plan
- 如果没有，是否先做批次规划
- 当前正在处理第几批
- 上一批停在哪里

第三步，根据当前阶段自动切换工作方式：

- 如果是 raw 导入，先建立 `sources/works/{work_id}/`
- 如果还没有总体分析，先做总体分析
- 如果还没有批次规划，先做批次规划
- 如果还没有候选角色识别，先做候选角色识别
- 如果是世界抽取，先读 metadata、候选角色文件，以及现有批次规划
- 如果是角色抽取，先读 metadata、候选角色文件、world 基础信息和现有批次规划
- 如果是修订合并，先读取本批新证据与旧结论
- 如果是运行时任务，先读 `works/` 与 `users/`

如果当前是自动连续模式，你必须额外执行：

1. 建立或更新批次规划文件。
2. 建立或更新进度文件，格式可参考 `prompts/shared/自动批处理进度模板.md`。
3. 自动选择下一批，而不是等待用户逐批指令。
4. 每完成一批后更新进度。
5. 只有在以下情况下才暂停等待用户：
   - 需要用户指定角色范围
   - 需要用户决定是否继续扩展角色集合
   - 原文证据或输入数据本身损坏
   - 规则冲突无法自行决策

你必须遵守这个总流程顺序：

1. 先总体分析
2. 再批次规划
3. 再候选角色识别
4. 再 world-first batch extraction
5. 再 selected-character batch extraction
6. 每一批之后都检查是否要修订与冲突合并

除非我明确要求跳步骤，否则不要越过前面的阶段直接进入后面的阶段。

通用规则：

1. 不要默认通读整本小说。
2. 不要默认通读全部 logs。
3. 不要在回答里粘贴大段原文。
4. 中文作品的作品相关内容默认保持中文。
5. `sources/works/{work_id}/` 只放原始导入、标准化正文、章节、metadata。
6. `works/{work_id}/` 只放作品级 canon。
7. `users/{user_id}/` 放所有用户侧状态。
8. 长书必须分批处理，不要把“读完整本再总结”当成默认路径。
9. 世界层只记录大事件，不记录小桥段。
10. 世界层 cast 只收主角团和高频配角，不默认收录龙套。
11. 后续 batch 可以修正旧结论，但必须由原文证据驱动。

在你拿到足够信息后：

1. 先用 3-8 行总结你理解到的任务。
2. 明确指出“当前处于哪个阶段、下一阶段是什么”。
3. 如果这是长书抽取任务，先说明你准备如何分批。
4. 明确说明你接下来先读哪些文件。
5. 然后再开始实质工作。

每一轮完成后，你都要输出一个小结，至少包含：

1. 当前已完成的阶段
2. 本轮新增或修订了哪些文件
3. 当前是否发现冲突或不确定点
4. 下一步最推荐进入哪个子流程
5. 如果是自动连续模式，当前 `next_batch_id` 是什么

稀释保护机制：

1. 这是一份总控流程 prompt，不需要在每个 batch 都重读整份。
2. 但在以下时点，必须回看本 prompt 的这些段落：
   - `开始实质工作前，先做一次最小结构读取`
   - `你要管理的完整流程如下`
   - `你必须把这几个子流程视为你可调用的下游 prompt 职责`
   - `你必须遵守这个总流程顺序`
   - `通用规则`
   - `稀释保护机制`
3. 在以下时点，必须再读一次当前对应子流程 prompt：
   - 从总体分析切到批次规划时
   - 从批次规划切到候选角色识别时
   - 从候选角色识别切到世界抽取时
   - 从世界抽取切到角色抽取时
   - 从任何抽取阶段切到修订合并时
4. 在每个 batch 开始前和落盘前，必须回看 `prompts/shared/批次执行检查清单.md`。
5. 在每个 batch 或每个大阶段结束后，都必须按 `prompts/shared/批次交接模板.md` 产出交接摘要。
6. 一旦发现自己开始跳阶段、忘记 world-first 顺序，或忘记 `sources / works / users` 边界，立即停止扩读，先重述“当前阶段 / 下一阶段 / 当前子流程”三件事，再继续。
```
