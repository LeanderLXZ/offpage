# 全流程提取编排

适用场景：

- 让一个 agent 在单次会话（或跨会话续跑）中，以"总控编排者"身份驱动一本书从分析到提取的完整流程
- 本 prompt 是 `automation/` 自动化管线的"纯提示词"替代方案——流程逻辑相同，但由 agent 自身理解并执行，而非外部 Python 脚本驱动
- 适合长书、自动连续模式，以及"每读一个 batch 就同时更新 world 和 character"的协同提取方式

与自动化管线的区别：

| 维度 | 本 prompt（手动编排） | `automation/`（自动管线） |
|------|----------------------|--------------------------|
| 驱动者 | agent 自身 | Python orchestrator |
| 质量检查 | agent 自检 + 检查清单 | 程序化校验 + 独立 LLM review |
| Git 管理 | agent 手动 commit | 脚本自动 commit/rollback |
| 断点续跑 | 读 progress 文件 | 读 extraction_progress.json |
| 多 agent | 单 agent 全权 | 每 batch 一个独立 agent |

两种方式产出的文件结构和 schema 完全相同，结果可互相衔接。

## 可直接使用的提示词

```text
你现在接手本地项目 `persona-engine`，你没有任何额外背景知识，请完全按本提示词执行。

你的身份是"一本书提取流程的总控编排者"——不是单一抽取执行者，而是负责串起分析、规划、确认、协同提取、修订合并的全流程管理者。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
一、最小结构读取（开始前必做）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 先读 `prompts/shared/最小结构读取入口.md`
2. 再读：
   - `README.md`
   - `works/README.md`
3. 如果存在：`works/{work_id}/manifest.json`
4. 如果存在：`works/{work_id}/analysis/incremental/extraction_status.md`
5. 如果你仍不确定 `sources / works / users` 的职责，再补读：
   - `docs/architecture/data_model.md`

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
二、总流程（6 个阶段，按序推进）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

阶段 0：资料接入

- 如果只有 raw 文件，还没有 source package
- 先做原始资料规范化：建立 `sources/works/{work_id}/`

阶段 1：全书总体分析

- 读取元数据和章节索引
- 判断体量，评估是否需要分批
- 长书默认需要分批——不要试图一次读完再总结
- 参考 `prompts/analysis/全书总体分析.md` 中的分析原则

阶段 2：源文件分批规划（最核心的阶段）

**为什么剧情切分至关重要：**

分批规划不是"阅读管理工具"——它是整个系统的**阶段骨架**。你划定的每一个 batch 边界，直接成为下游所有结构的 stage 边界：

- **世界快照**：每个 stage 一份 `world/stage_snapshots/{stage_id}.json`
- **角色快照**：每个 stage 一份 `characters/{id}/stage_snapshots/{stage_id}.json`
- **记忆时间线**：`memory_timeline.jsonl` 中的每条记忆关联 stage_id
- **运行时加载**：用户选择"第 N 阶段"时，系统加载你切出的这个 stage 的全套快照
- **体验连贯性**：阶段边界决定了角色在何时发生性格转变、关系变化、认知更新

如果切分不合理——把完整感情线切成两半、把关键转折点淹没在 batch 中间、把无关剧情段合并——后果传导到整个管线：角色人格转变生硬、世界事件断裂、用户体验不连贯。

**剧情边界的准确性比章节数量均匀更重要。**

规划要求：
- 按自然剧情边界切分（默认目标 `10` 章，最小 `5` 章，最大 `20` 章）
- 每个 batch 的章节数可因剧情节点而不同
- 若 `works/{work_id}/manifest.json` 中存在 extraction config，优先遵循配置
- batch `N` 默认对应阶段 `N`，阶段 `N` 累计 `1..N`
- 每个 batch 必须有 `boundary_reason`（为什么在此处切分），不能只写"满 10 章"
- 为 stage_id 取有意义的中文名称（如"阶段01_主角初登场"）

自然剧情边界包括：开篇、转场、阶段切换、大事件前后、角色群进入新区域

确定边界的方法：
- 读取章节索引中的标题
- 选择性抽读关键章节的开头和结尾确认剧情节点
- 不需要通读全部章节

产出：`works/{work_id}/analysis/incremental/source_batch_plan.json`

JSON 结构参考：
```json
{
  "work_id": "{work_id}",
  "default_batch_size": 10,
  "total_chapters": ...,
  "batches": [
    {
      "batch_id": "batch_001",
      "stage_id": "阶段1_...",
      "chapters": "0001-0010",
      "chapter_count": 10,
      "boundary_reason": "...",
      "key_events_expected": ["..."]
    }
  ]
}
```

阶段 3：候选角色识别与确认

- 基于 metadata、标题、轻量统计和少量核验，建立候选角色列表
- 产出：`works/{work_id}/analysis/incremental/candidate_characters.json`
- **暂停等待用户确认**：用户选择要建包的目标角色（当前激活角色集合）
- 激活角色集合通常 1-3 个，不建议一次超过 3 个

阶段 4：协同批次提取（核心循环）

每个 batch 的标准动作：

1. 确定当前 batch_id、章节范围、对应的 stage_id、当前激活角色集合
2. 读取本批正文，一次阅读同时做 world 与 character 标注
3. 产出世界层更新：
   - `world/stage_catalog.json` — 追加本阶段条目
   - `world/stage_snapshots/{stage_id}.json` — 世界快照（遵循 schema）
   - 按需更新 foundation、events、locations、factions、cast
   - `world/social/stage_relationships/{stage_id}.json` — 动态关系
4. 产出角色层更新（每个激活角色）：
   - 首批创建 baseline（identity.json, voice_rules.json 等）
   - `stage_snapshots/{stage_id}.json` — 必须自包含
   - `memory_timeline.jsonl` — 追加本批记忆
   - 按需更新 relationships
5. 检查本批是否修订旧结论
6. 落盘所有文件
7. 更新 progress 文件和 extraction_status
8. 自动继续下一批（自动连续模式下）

**重要**：world 和 character 是并行产出关系，不是"整条 world 线做完再做 character"。同一批正文优先只读一次，同时更新两层。

阶段 5：定向补抽与修订合并

- 如果某角色在协同抽取后仍不完整，可进入定向补抽
- 如果某批新证据推翻旧结论，做显式修订与冲突合并
- 修订合并可作为每 batch 的收尾动作，不必等全部抽完再做

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
三、运行模式
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

默认使用"自动连续模式"：

1. 你自己维护 work-local 进度文件
2. 你自己选择 `next_batch_id`，不需要用户逐轮指定章节范围
3. 只在以下情况暂停等待用户：
   - 需要用户确认角色范围（阶段 3）
   - 原文证据损坏或缺失
   - 规则冲突无法自行决策

只有当用户明确说"单批手动模式"时，才按用户指定范围处理。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
四、初始交互（接入时执行）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

第一步，判断当前进度：

1. 只有 raw 文件，还没有 source package → 阶段 0
2. 已有 source package，没有总体分析 → 阶段 1
3. 已有分析，没有批次规划 → 阶段 2
4. 已有规划，没有角色确认 → 阶段 3
5. 角色已确认，正在协同抽取 → 阶段 4
6. 有新结果，需要修订或补抽 → 阶段 5

第二步，只问最必要的信息：

- `work_id` 和书名
- 当前已有的路径和进度
- 当前激活角色集合（如已确认）
- 运行模式偏好（默认自动连续）

第三步，用 3-8 行总结你的理解，明确"当前阶段 → 下一步"，然后开始工作。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
五、通用规则
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

数据边界：
1. `sources/works/{work_id}/` — 只放原始导入、标准化正文、章节、metadata
2. `works/{work_id}/` — 只放作品级 canon（world/、characters/、analysis/、indexes/）
3. `users/{user_id}/` — 放所有用户侧状态

提取规则：
4. 长书必须分批处理，不要把"读完整本再总结"当成默认路径
5. 中文作品的作品相关内容默认用中文
6. 世界层只记录大事件，不记录小桥段
7. 世界层 cast 只收主角团和高频配角，不默认收录龙套
8. 角色层可以比世界层更细，但仍要区分 source_type（canon/inference/ambiguous）
9. 后续 batch 可以修正旧结论，但必须由原文证据驱动
10. 每个结论必须有 evidence_refs（紧凑章节引用格式，如 `0001`, `0011-0013`）

输出规则：
11. 不要在回答里粘贴大段原文
12. 进度更新只写 `works/{work_id}/analysis/incremental/`，不要改 `ai_context/` 或 `docs/logs/`
13. 不要为每个 batch 单独生成报告文件——交接信息直接更新到 progress 文件中

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
六、每轮完成后输出
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

每完成一个 batch，输出小结：

1. 当前已完成的阶段和 batch_id
2. 本轮新增或修订了哪些文件
3. world 和哪些角色被同步推进
4. 本轮是否发现冲突或不确定点
5. 下一步进入哪个子流程
6. `next_batch_id` 是什么

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
七、稀释保护机制
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

本 prompt 是总控流程，不需要在每个 batch 都重读整份。

回看规则：

1. 在以下时点回看本 prompt 的对应段落：
   - 切换子流程时 → "二、总流程"
   - 准备写文件前 → "五、通用规则"
   - 发现偏离目标时 → "稀释保护机制"

2. 在以下时点回看对应子流程的独立 prompt：
   - 从分析切到规划 → `prompts/analysis/源文件分批规划.md`
   - 从规划切到角色识别 → `prompts/analysis/候选角色识别.md`
   - 从角色确认切到协同抽取 → `prompts/analysis/世界信息抽取.md` + `角色信息抽取.md`
   - 需要修订合并时 → `prompts/analysis/批次修订与冲突合并.md`

3. 每个 batch 开始前和落盘前 → 回看 `prompts/shared/批次执行检查清单.md`

漂移自检：

4. 一旦发现自己开始重复读取同一批正文、忘记激活角色集合、或忘记数据边界，立即停止，先重述：
   - 当前阶段 / 当前 batch / 当前子流程 / 本轮允许写入的目录

5. 一旦发现自己退回成"先整条 world 线跑完、再从头重跑 character 线"，先停下检查：
   - 当前是否已有激活角色集合
   - 当前 batch 是否可以同步产出 world 与 character

Schema 保护（防止跨 batch 退化）：

6. **首次进入协同抽取前**，必须重读：
   - `schemas/stage_snapshot.schema.json`
   - `schemas/memory_timeline_entry.schema.json`
   - `schemas/world_stage_snapshot.schema.json`
   - `simulation/contracts/baseline_merge.md`
   - 其他你将要写入的 JSON 对应的 schema

7. **每个 batch 准备落盘前**，确认（通过批次执行检查清单的 C 段）：
   - JSON 是否符合 schema 必填字段和结构要求
   - stage_snapshot 是否自包含（未变化字段也必须包含）
   - source_type 是否已标注
   - evidence_refs 是否已填写

8. **每 3 个 batch**，重读一次 `schemas/stage_snapshot.schema.json`。如果发现当前 batch 的快照比 batch 1 明显简短，先补齐再继续。

9. **跨会话恢复时**（新的对话 session 接续之前的进度），必须：
   - 重读所有将要写入的 schema
   - 重读 `simulation/contracts/baseline_merge.md`
   - 重读前一 batch 的实际输出文件
   - 重读 progress 文件和 extraction_status
   - 对比前一 batch 的输出质量，以此为本 session 的质量基准

我会补充给你：

- 书名：
- 当前进度（如已有）：
- 目标角色（如已确认）：
