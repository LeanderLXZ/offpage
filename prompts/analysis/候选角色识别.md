# 候选角色识别

适用场景：

- 从一本书的数据中先识别值得持续跟踪和后续建包的角色候选

## 可直接使用的提示词

```text
你现在接手本地项目 `persona-engine`，你没有任何额外背景知识，请完全按本提示词执行。

目标：
为指定作品做第一轮候选角色识别，并把结果写入 `works/{work_id}/analysis/incremental/`。

开始实质工作前，先做一次最小结构读取：

1. 先读 `prompts/shared/最小结构读取入口.md`
2. 再读：
   - `README.md`
   - `works/README.md`
3. 如果存在：`works/{work_id}/manifest.json`
4. 如果你仍不确定 `analysis/` 在 work package 里的职责，再补读 `docs/architecture/data_model.md`

你必须优先读取：

1. `sources/works/{work_id}/manifest.json`
2. `sources/works/{work_id}/metadata/book_metadata.json`
3. `sources/works/{work_id}/metadata/chapter_index.json`

如有必要，你可以再做：

- 全书标准化章节中的表层提及统计
- 少量定点章节核验

不要默认做的事：

- 不要通读整本小说
- 不要回看全部日志
- 不要把大量一次性小角色都塞进候选清单

角色筛选原则：

1. 优先识别：
   - 主角
   - 主要女主 / 男主
   - 高频配角
   - 结构性重要配角
   - 特殊高频非人角色
2. 不要因为一章高光就把一次性路人误判成长期候选。
3. 要区分“可长期建包的角色”和“只需要记住但不必重点追踪的小角色”。

输出文件建议：

- `works/{work_id}/analysis/incremental/candidate_characters_initial.md`

输出结构至少包括：

1. 作品概况
2. 本轮边界
3. 本轮实际核验的章节
4. 候选角色分层
5. 每个候选角色的：
   - 暂定定位
   - 直接依据
   - 标题信号
   - 提及统计
   - 暂定结论
6. 当前建议的优先顺序
7. 后续建议

语言规则：

- 中文作品的分析内容默认中文
- 文件内容尽量不要出现多余英文说明
- 路径和字段名可以保留技术形式

稀释保护机制：

1. 开始前先写一张“当前任务卡”：
   - 当前目标
   - 当前阶段
   - 当前允许写入的目录
   - 当前禁止做的事
2. 不需要反复重读整份 prompt，但在以下时点必须回看本 prompt 的这些段落：
   - `目标`
   - `开始实质工作前，先做一次最小结构读取`
   - `优先读取`
   - `不要默认做的事`
   - `角色筛选原则`
   - `输出结构`
   - `稀释保护机制`
3. 在以下时点，必须回看 `prompts/shared/批次执行检查清单.md`：
   - 开始 metadata 扫描前
   - 完成全书提及统计后
   - 每完成一组定点章节核验后
   - 准备写候选文件前
4. 如果核验章节开始变多，把“每一组核验章节”视为一个小批次。
5. 每完成一轮候选识别，都要按 `prompts/shared/批次交接模板.md` 产出一份简短交接摘要。
6. 一旦发现自己开始把大量小角色塞进候选清单，立即停止扩展名单，先重述筛选原则。

我会补充给你：

- `work_id`：
- 书名：
- 是否已有旧候选文件需要重做：
```
