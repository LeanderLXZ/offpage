# 批次修订与冲突合并

适用场景：

- 新一轮 source batch 读完后，需要修正已有的 world / character / analysis 结论

## 可直接使用的提示词

```text
你现在接手本地项目 `persona-engine`，你没有任何额外背景知识，请完全按本提示词执行。

目标：
根据一轮新的 source batch，检查并修订已有的 canon 结论，显式记录冲突、纠正和未解决问题。

开始实质工作前，先做一次最小结构读取：

1. 先读 `prompts/shared/最小结构读取入口.md`
2. 再读：
   - `README.md`
   - `works/README.md`
3. 如果这轮修订涉及 `users/` 判断，再补读 `users/README.md`
4. 如果存在：相关 package 的 `manifest.json`
5. 如果你要修改结构化 JSON，再补读 `schemas/README.md` 与对应 schema
6. 如果你仍不确定不同包之间谁该承接这次修订，再补读 `docs/architecture/data_model.md`

你必须遵守：

1. 后续 batch 可以修正旧结论。
2. 但修正必须由原文证据驱动。
3. 用户对话不能改写作品级 world 或角色 canon。
4. 世界层只记录大事件；角色层承接更细的个体信息。
5. 长书必须依赖批次检查点和批次修订，而不是整书重跑。

执行模式：

1. 默认采用“自动连续模式”。
2. 在自动连续模式下，本 prompt 应跟随当前已完成的抽取批次自动执行，不需要用户每轮手动指定范围。
3. 如果没有显式给出 `batch_id`，你应从当前相关的 progress 文件中读取最近完成批次或当前待合并批次。

请按这个顺序执行：

1. 列出本轮 `batch_id` 与新证据来自哪些章节。
2. 列出本轮可能影响的目标：
   - `world/`
   - 当前目标角色
   - 其他已识别角色
3. 对每一条旧结论做判断：
   - 保持不变
   - 补充
   - 修正
   - 暂时降级为不确定
4. 把真正需要更新的文件写回对应位置。
5. 把未解决冲突写进分析或 world mystery / conflict 记录。

输出要求：

1. 明确写出“旧结论是什么、新证据是什么、现在如何处理”。
2. 如果证据不足，不要强行定论。
3. 不要默默覆盖掉旧理解而不说明原因。
4. 不要把小修小补全堆成大事件。
5. 最后给出“下一批最推荐继续处理哪里”。

稀释保护机制：

1. 开始前先写一张“当前任务卡”：
   - 当前目标
   - 当前阶段
   - `batch_id`
   - 本轮允许修改的文件范围
   - 本轮禁止做的事
2. 不需要反复重读整份 prompt，但在以下时点必须回看本 prompt 的这些段落：
   - `目标`
   - `开始实质工作前，先做一次最小结构读取`
   - `你必须遵守`
   - `请按这个顺序执行`
   - `输出要求`
   - `稀释保护机制`
3. 在以下时点，必须回看 `prompts/shared/批次执行检查清单.md`：
   - 开始对照旧结论前
   - 每完成一类文件的修订判断后
   - 准备正式写回前
4. 每完成一轮修订合并，都必须按 `prompts/shared/批次交接模板.md` 产出一份交接摘要。
5. 如果发现自己开始“直接覆盖旧文件而不解释原因”，立即停下，先补全“旧结论 / 新证据 / 现在如何处理”。
6. 如果冲突仍未解决，不要硬合并，保留冲突并写明下一步核验建议。

我会补充给你：

- `work_id`：
- `batch_id`（可选）：
- 本轮新读取的章节（可选）：
- 需要重点复核的旧文件：
```
