# 世界信息抽取

适用场景：

- 从一本书的数据中按 batch 抽取作品级世界信息

## 可直接使用的提示词

```text
你现在接手本地项目 `persona-engine`，你没有任何额外背景知识，请完全按本提示词执行。

目标：
对指定作品执行 world-first batch extraction，把共享世界信息写入 `works/{work_id}/world/`。

开始实质工作前，先做一次最小结构读取：

1. 先读 `prompts/shared/最小结构读取入口.md`
2. 再读：
   - `README.md`
   - `works/README.md`
3. 如果存在：`works/{work_id}/manifest.json`
4. 如果你要创建或更新结构化 JSON，再补读 `schemas/README.md` 与对应 schema
5. 如果你仍不确定 `world/` 和 `characters/` 的分工，再补读 `docs/architecture/data_model.md`

你必须遵守这些核心规则：

1. `sources/works/{work_id}/` 只放原始导入、标准化正文、章节、metadata。
2. `works/{work_id}/world/` 只放作品级、由原文证据驱动的世界 canon。
3. 世界信息允许在后续 batch 中被补充或修正，但只能由原文证据驱动，不能被用户对话改写。
4. 世界层只记录大的、共享的、跨角色重要的事件。
5. 小事件、小桥段、小型互动不默认写入世界层，这些通常由角色层承接。
6. 世界层 cast 只保留主角团和高频配角，不默认收录龙套和一次性小角色。
7. 中文作品默认输出中文名称、中文 identifier value、中文路径段。
8. 世界阶段抽取必须强调时间性：
   - 当前阶段的世界状态要按“现在进行时”写清
   - 更早阶段的内容应作为历史事件保留
   - 不要把历史与当前状态混成一个扁平总结

推荐起点：

1. 如果存在：`works/{work_id}/analysis/incremental/extraction_status.md`
2. `sources/works/{work_id}/manifest.json`
3. `sources/works/{work_id}/metadata/book_metadata.json`
4. `sources/works/{work_id}/metadata/chapter_index.json`
5. `works/{work_id}/analysis/incremental/candidate_characters_initial.md`
6. 如果存在：`works/{work_id}/analysis/incremental/source_batch_plan.md`
7. 如果存在：`works/{work_id}/analysis/incremental/world_batch_progress.md`

批次原则：

1. 不要默认通读整本小说。
2. 必须按 source batch 处理，禁止把整本书一次性当成一个 batch。
3. 每批只读有限章节或有限片段。
4. 普通批次建议：
   - 默认 `10` 章
   - 若 `works/{work_id}/manifest.json` 中存在 extraction config，则优先遵循配置
   - 尽量贴近自然剧情边界
5. 如果当前还没有批次规划，先建立 `source batch plan`，再开始本批抽取。
6. 每个 batch 都可能修正：
   - world
   - 当前目标角色
   - 其他已识别角色
7. 默认把 batch `N` 视为阶段 `N` 候选。
8. 当用户要求提取世界的某个阶段 `N` 时，默认只需累计处理 `1..N`，并把第 `N` 阶段的状态写成当前世界状态。
9. 如果已经存在 `extraction_status` 或 `world_batch_progress`，先把它们视为当前续跑入口；除非文件缺失、明显冲突或用户明确要求重做，否则不要从 `batch_001` 重新开始。

执行模式：

1. 默认采用“自动连续模式”。
2. 只有当用户明确要求“单批手动模式”时，才接受用户手动指定本轮章节范围。
3. 在自动连续模式下，如果用户没有给本轮范围，你必须：
   - 先读 `extraction_status`
   - 读取 `source_batch_plan`
   - 读取并优先沿用 `world_batch_progress`
   - 自动选择当前阶段中下一个未完成 batch
   - 处理完成后更新进度文件
   - 同步更新 `extraction_status`
   - 再决定是否继续下一批

本批开始前，请先确认：

1. 当前 `batch_id`
2. 章节范围
3. 本批目标
4. 上一批已经写下的结论
5. 本批可能会修订哪些旧文件
6. 本批对应哪个阶段候选

建议输出方向：

- `world/manifest.json`
- `world/stage_catalog.json`
- `world/stage_snapshots/`
- `world/foundation/`
- `world/history/`
- `world/events/`
- `world/locations/`
- `world/factions/`
- `world/cast/`
- `world/social/fixed_relationships/`
- `world/social/stage_relationships/`

本轮输出时必须明确：

1. `batch_id`
2. 本批实际读取的章节范围
3. 新增了哪些世界事实
4. 修正了哪些旧理解
5. 哪些内容是原文明示
6. 哪些内容仍然不确定
7. 本批对应的阶段候选是什么
8. 截至当前阶段，哪些内容属于历史事件
9. 当前阶段下，哪些内容属于当前世界状态
10. 下一批建议从哪里继续

证据引用要求：

1. 如果证据来自当前作品的章节正文，`evidence_refs` 默认写紧凑章节引用，例如：
   - `0001`
   - `0004`
   - `0011-0013`
2. 不要在每个 JSON 中重复写完整章节路径，除非证据来自：
   - 当前作品之外的文件
   - 非章节文件
   - 或路径本身具有区分意义

不要做的事：

- 不要用用户对话去改 world canon
- 不要把角色心理活动大量堆进 world
- 不要把所有事件都写成 world 事件
- 不要把一堆小人物塞进 cast
- 不要在 `world/` 里重复维护角色认知摘要层
- 不要额外建立 `world/mysteries/`

稀释保护机制：

1. 开始每个 batch 前，先写一张“当前任务卡”：
   - 当前目标
   - 当前阶段
   - `batch_id`
   - 本批允许写入的目录
   - 本批禁止做的事
2. 不需要每次重读整份 prompt，但在以下时点必须回看本 prompt 的这些段落：
   - `目标`
   - `开始实质工作前，先做一次最小结构读取`
   - `核心规则`
   - `批次原则`
   - `本批开始前，请先确认`
   - `本轮输出时必须明确`
   - `不要做的事`
   - `稀释保护机制`
3. 在以下时点，必须回看 `prompts/shared/批次执行检查清单.md`：
   - 开始本批第一章前
   - 每完成一个阅读小段后
   - 准备扩大阅读范围前
   - 准备落盘前
4. 每个 batch 结束后，必须更新 `world_batch_progress.md` 和 `extraction_status.md`，不要为每个 batch 单独生成报告文件。交接信息应直接写入 progress 文件。
5. 如果在处理中发现自己开始记录大量小事件、龙套角色或角色心理活动，立即停止继续写 world，先回看本 prompt 的 `核心规则` 与 `不要做的事`。
6. 如果本批需要修订旧 world 结论，先写清“旧结论 / 新证据 / 新处理”，再继续写入文件。
7. 如果你开始把“早期已经发生过的事”误写成“当前阶段仍在发生的状态”，立即停下，先重述：
   - 截至本阶段的历史
   - 当前阶段的现状
   - 哪些只是背景，不是现在状态

我会补充给你：

- `work_id`：
- 书名：
- 如果要强制手动：本轮 batch 范围（可选）
- 如果已有 world 文件，是否允许你修订：
```
