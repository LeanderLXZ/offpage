# 源文件分批规划

适用场景：

- 一本书很长，不能也不应该一次性通读
- 在正式做 world / character extraction 前，先为 source 制定可执行的批次计划

## 可直接使用的提示词

```text
你现在接手本地项目 `persona-engine`，你没有任何额外背景知识，请完全按本提示词执行。

目标：
为一本书建立“源文件分批处理计划”，让后续 agent 可以按批次、按检查点、按修订节奏持续处理，而不是试图一次读完整本书。

开始实质工作前，先做一次最小结构读取：

1. 先读 `prompts/shared/最小结构读取入口.md`
2. 再读：
   - `README.md`
   - `works/README.md`
3. 如果存在：`works/{work_id}/manifest.json`
4. 如果你仍不确定分析产物应写在哪一层，再补读 `docs/architecture/data_model.md`

你必须遵守：

1. 不要默认通读整本小说。
2. 先做批次规划，再做深度抽取。
3. 批次规划必须优先依据：
   - `sources/works/{work_id}/manifest.json`
   - `sources/works/{work_id}/metadata/book_metadata.json`
   - `sources/works/{work_id}/metadata/chapter_index.json`
4. 如果已有分析文件，也应参考：
   - `works/{work_id}/analysis/`
5. 如果存在 `works/{work_id}/analysis/incremental/extraction_status.md`，应先读它，优先判断是“续写 / 修订现有规划”，还是“首次建立规划”。

核心思路：

1. 先做“轻扫描”：
   - 章节数
   - 章节标题
   - 章节长度
   - 角色名信号
   - 世界线索信号
2. 再把全书拆成连续的 source batches。
3. 每个 batch 必须有明确用途，例如：
   - 开局设定建立
   - 候选角色确认
   - 世界基础规则抽取
   - 某角色阶段信息抽取
   - 中后期修订批次
4. 默认把每个 batch 视为一个“阶段候选”：
   - batch `N` 默认对应阶段 `N`
   - 阶段 `N` 的抽取范围默认累计到 `1..N`

批次划分建议：

1. 优先按连续章节划分，不要乱跳。
2. 普通批次建议：
   - 默认 `10` 章
   - 若 `works/{work_id}/manifest.json` 中存在 extraction config，则优先遵循配置
   - 章节极短时可适度合并，章节极长时可适度缩小
3. 如果章节非常短，可以合并更多章。
4. 如果章节特别长或场景特别密，可以缩小批次。
5. 尽量贴近自然剧情边界：
   - 开篇
   - 转场
   - 阶段切换
   - 大事件前后
   - 角色群进入新区域

你需要产出：

1. 一本书的总处理策略
2. 推荐批次表
3. 每批的：
   - `batch_id`
   - 对应 `stage_id`
   - 章节范围
   - 大致字量或体量判断
   - 本批主要目标
   - 预期可能影响：
     - `world`
     - 哪些角色
     - 哪些关系
4. 推荐先做哪几批
5. 哪些批次可能需要后续回修
6. 一个可用于自动连续运行的起始进度骨架

建议写入：

- `works/{work_id}/analysis/incremental/extraction_status.md`
- `works/{work_id}/analysis/incremental/source_batch_plan.md`
- `works/{work_id}/analysis/incremental/source_batch_progress.md`

输出要求：

1. 明确说明为什么这样切 batch。
2. 不要把所有批次都写得一模一样。
3. 不要假装已经读过整本书；只能基于 metadata、标题、轻扫描和少量核验做规划。
4. 如果有推断，要标注为推断。
5. 这份规划必须让后续 agent 在没有人工指定每轮范围的情况下，也能自动选出下一批。
6. 这份规划还必须让后续 agent 知道：
   - 当前 batch 对应哪个阶段候选
   - 阶段 `N` 是否默认累计读取到 `1..N`
7. 如果仓库里已经存在旧规划或旧进度，你默认应该在其基础上续写或修订，而不是直接从头覆盖。

稀释保护机制：

1. 开始前先写一张“当前任务卡”：
   - 当前目标
   - 当前阶段
   - 当前允许写入的目录
   - 当前禁止做的事
2. 不需要反复重读整份 prompt，但在以下时点必须回看本 prompt 的这些段落：
   - `目标`
   - `开始实质工作前，先做一次最小结构读取`
   - `你必须遵守`
   - `核心思路`
   - `批次划分建议`
   - `输出要求`
   - `稀释保护机制`
3. 在以下时点，必须回看 `prompts/shared/批次执行检查清单.md`：
   - 开始 chapter_index 扫描前
   - 完成轻扫描后
   - 准备给出最终 batch 表前
4. 当前任务本身不是正文抽取，但仍应把“规划阶段”视为单批次执行。
5. 完成规划后，按 `prompts/shared/批次交接模板.md` 产出一份简短交接摘要。
6. 一旦发现自己开始假装已经读过大量正文，立即停下，改回基于 metadata 和轻扫描的规划模式。

自动连续模式要求：

1. 默认把“自动连续模式”视为长书处理的推荐模式。
2. 你不能要求用户逐轮提供 `batch` 范围，除非用户明确要求单批手动。
3. 你产出的批次规划必须足够清楚，使后续 agent 可以：
   - 先读 `extraction_status`
   - 读入 `source_batch_plan`
   - 读入或创建 `source_batch_progress`
   - 自动确定 `next_batch_id`
   - 自动确定 `next_stage_id`
4. 完成规划后，建议按 `prompts/shared/自动批处理进度模板.md` 初始化进度文件，并同步更新 `extraction_status`。

我会补充给你：

- `work_id`：
- 书名：
- 是否已有旧批次规划：
```
