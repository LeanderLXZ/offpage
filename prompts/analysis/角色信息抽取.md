# 角色信息抽取

适用场景：

- 已经有 source package，准备针对一个指定角色做可持续修订的角色 canon 抽取

## 可直接使用的提示词

```text
你现在接手本地项目 `persona-engine`，你没有任何额外背景知识，请完全按本提示词执行。

目标：
围绕一个指定角色，把原文支撑的角色 canon 抽取到 `works/{work_id}/characters/{character_id}/`。

开始实质工作前，先做一次最小结构读取：

1. 先读 `prompts/shared/最小结构读取入口.md`
2. 再读：
   - `README.md`
   - `works/README.md`
3. 如果存在：`works/{work_id}/manifest.json`
4. 如果你要创建或更新结构化 JSON，再补读 `schemas/README.md` 与对应 schema
5. 如果你仍不确定 `characters/`、`world/` 和 `users/` 的边界，再补读 `docs/architecture/data_model.md`

你必须遵守这些边界：

1. 先加载 `works/{work_id}/characters/{character_id}/` 的基础角色信息，再考虑后续叠加用户侧状态。
2. 当前这一步只处理用户无关的基础 canon，不写用户漂移、不写用户互动后果。
3. 用户侧状态只能写入 `users/{user_id}/`，不要写回角色基础包。
4. 中文作品默认使用中文名称、中文 identifier value、中文路径段。

推荐顺序：

1. 先看：
   - 如果存在：`works/{work_id}/analysis/incremental/extraction_status.md`
   - `sources/works/{work_id}/manifest.json`
   - `sources/works/{work_id}/metadata/book_metadata.json`
   - `sources/works/{work_id}/metadata/chapter_index.json`
   - `works/{work_id}/analysis/incremental/candidate_characters_initial.md`
   - 已有 `works/{work_id}/world/` 信息
   - 如果存在：`works/{work_id}/analysis/incremental/source_batch_plan.md`
   - 如果存在：`works/{work_id}/analysis/incremental/character_batch_progress/{character_id}.md`
2. 再按 batch 定点读和该角色高度相关的章节

长书处理规则：

1. 不要试图把这个角色在整本书中的全部信息一次性抽完。
2. 先依据章节索引与已有 world / analysis 结果做分批规划。
3. 批次大小由分析阶段的分批规划确定，按自然剧情边界切分：
   - 默认目标 `10` 章，最小 `5` 章，最大 `15` 章
   - 每个 batch 的章节数可因剧情节点而不同
   - 若 `works/{work_id}/manifest.json` 中存在 extraction config，则优先遵循配置
   - 以该角色高相关章节为主
4. 每一批都应当是“可修订、可续跑、可回滚判断”的。
5. 默认把 batch `N` 视为该角色的阶段 `N` 候选，阶段 `N` 的抽取范围默认累计到 `1..N`。
6. 如果已经存在 `extraction_status` 或该角色的 `character_batch_progress`，先把它们视为续跑入口，不要默认从头重新规划。

执行模式：

1. 默认采用“自动连续模式”。
2. 只有当用户明确要求“单批手动模式”时，才接受用户手动指定本轮章节范围。
3. 在自动连续模式下，如果用户没有给本轮范围，你必须：
   - 先读 `extraction_status`
   - 读取 `source_batch_plan`
   - 读取或创建该角色的 batch progress
   - 自动选择该角色当前最合适的下一个未完成 batch
   - 处理完成后更新进度文件
   - 同步更新 `extraction_status`
   - 再决定是否继续下一批

角色抽取时请至少关注：

1. 身份与别名
2. 阶段变化
3. 角色记忆线
4. 关系变化（态度、信任度、亲密度、语气和行为模式变化、驱动事件）
5. 知识边界与误解
6. 语气、行为、偏好、底线
7. 该角色视角下的重要事件
8. 当前阶段的状态、心情、性格与口吻
9. 当前阶段前已经历了哪些关键事件
10. 当前阶段下与关键人物的关系状态
11. **自包含阶段快照**：每个阶段快照必须包含该阶段的完整角色状态，包括：
    - `voice_state`：完整的语气基调、语言习惯、用词偏好、口头禅、禁忌用语、
      情绪语气矩阵、对象语气矩阵、典型对话示例
    - `behavior_state`：完整的核心驱动力、决策风格、情绪触发器、情绪反应
      矩阵、关系行为矩阵、习惯性行为、压力应对
    - `boundary_state`：当前阶段有效的软边界、容易被误判的点
    - `relationships`：对每个重要角色的完整关系状态（态度、信任、亲密度、
      语气变化、行为变化、驱动事件、关系演变概述）
    第一批（阶段 1）同时填充 baseline 文件和阶段 1 快照。
    后续批次以 baseline 和前一阶段快照为参照，产出新的自包含快照——
    未变化的内容也应包含在快照中，不可省略。

可写入的典型文件：

- `manifest.json`
- `canon/identity.json`
- `canon/bible.md` 或 `bible.json`
- `canon/memory_timeline/{stage_id}.jsonl`
- `canon/relationships.json`
- `canon/voice_rules.json`
- `canon/behavior_rules.json`
- `canon/boundaries.json`
- `canon/failure_modes.json`
- `canon/stage_catalog.json`
- `canon/stage_snapshots/{stage_id}.json`

抽取原则：

1. 不要默认通读整本小说。
2. 必须按 source batch 处理，而不是一次覆盖全书。
3. 每批只读真正和该角色相关的章节。
4. 每批开始前都先确认：
   - `batch_id`
   - 本批章节范围
   - 本批目标
   - 是否要修订旧结论
   - 本批对应哪个阶段候选
5. 角色包里可以比世界层更细，但仍要区分：
   - 原文明示
   - 推断
   - 暂不确定
6. 如果本批原文足以修正 world 或其他角色的结论，要明确记录并同步说明。
7. 时间性必须写清：
   - 截至当前阶段已经发生过什么
   - 当前阶段角色现在是什么状态
   - 当前阶段角色现在会怎么说话、怎么看人、怎么反应
8. 每个阶段都应至少产出一句话 summary，供用户在运行时选择。

证据引用要求：

1. 如果证据来自当前作品章节，`evidence_refs` 默认写紧凑章节引用，例如：
   - `0008`
   - `0033`
   - `0151-0160`
2. 只有当证据来自：
   - 当前作品之外的文件
   - 非章节文件
   - 或路径本身需要保留
   时，才优先写完整路径。

不要做的事：

- 不要把用户互动造成的关系变化写回基础角色包
- 不要把大量世界小事件照抄进角色包
- 不要把未核验的二手推断写成硬事实

稀释保护机制：

1. 开始每个 batch 前，先写一张”当前任务卡”：
   - 当前目标
   - 当前阶段
   - `character_id`
   - `batch_id`
   - 本批允许写入的目录
   - 本批禁止做的事
2. 不需要每次重读整份 prompt，但在以下时点必须回看本 prompt 的这些段落：
   - `目标`
   - `开始实质工作前，先做一次最小结构读取`
   - `边界`
   - `推荐顺序`
   - `长书处理规则`
   - `抽取原则`
   - `不要做的事`
   - `稀释保护机制`
3. 在以下时点，必须回看 `prompts/shared/批次执行检查清单.md`：
   - 开始本批第一章前
   - 每完成一组角色高相关章节后
   - 准备扩大读取范围前
   - 准备写角色文件前
4. 每个 batch 结束后，都必须按 `prompts/shared/批次交接模板.md` 产出一份交接摘要。
5. 如果在处理中发现自己开始把用户状态写回基础角色包，或开始把大量世界小事件直接灌进角色包，立即停止写入，先回看 `边界` 与 `不要做的事`。
6. 如果本批足以修订 world 或其他角色，先记录修订点，再继续当前角色包写入。
7. 如果你开始把”过去发生过的事”写成”当前阶段仍然在发生的状态”，立即停下，先拆开：
   - 截至当前阶段的经历
   - 当前阶段的现状
   - 当前阶段的口吻 / 心境 / 关系

Schema 与架构规范保护：

8. **首次写角色文件前**，必须重读：
   - `schemas/stage_snapshot.schema.json` — 完整字段结构
   - `schemas/memory_timeline_entry.schema.json` — 记忆条目字段和 source_type
   - `simulation/contracts/baseline_merge.md` — 自包含快照模型规则
9. **每个 batch 准备落盘前**，通过批次执行检查清单的 C 段确认：
   - stage_snapshot 是否自包含（未变化字段也必须包含）
   - source_type 是否已标注
   - evidence_refs 是否已填写
10. **每 3 个 batch**，强制重读 `schemas/stage_snapshot.schema.json`。
    对比当前 batch 和 batch 1 的快照详细度，如果出现退化，先补齐再继续。
11. **batch 2+ 开始前**，必须重读前一阶段的 stage_snapshot 输出，
    以此作为风格、详细度和术语一致性的参照基准。
12. **跨会话恢复时**，必须重读所有相关 schema、前一 batch 输出、
    progress 文件，不要仅依赖交接摘要中的文字描述。

我会补充给你：

- `work_id`：
- 书名：
- `character_id`：
- 角色名：
- `batch_id`（可选；自动模式下可自行确定）：
- 本轮 batch 范围（可选；自动模式下可自行确定）：
```
