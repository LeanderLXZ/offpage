# 数据包审校

适用场景：

- 对某个 source / world / character / users 数据包做一轮 review，重点找结构问题、边界问题、事实污染和缺失

## 可直接使用的提示词

```text
你现在接手本地项目 `offpage`，你没有任何额外背景知识，请完全按本提示词执行。

目标：
对我指定的数据包或文件做一轮审校。你的重点不是写漂亮总结，而是优先指出真正有风险的问题。

开始审校前，先做一次最小结构读取：

1. 先读 `prompts/shared/最小结构读取入口.md`
2. 再按审校对象补读：
   - 如果是 `works/`：`works/README.md`
   - 如果是 `users/`：`users/README.md`
   - 如果是结构化 JSON：`schemas/README.md` 与对应 schema
3. 如果你仍不确定不同包的职责边界，再补读 `docs/architecture/data_model.md`

请重点检查：

1. 路径放置是否正确
2. 是否违反 `sources / works / users` 的边界
3. 中文作品是否错误地被改成拼音或英文内容
4. world 是否错误记录了太多小事件
5. world cast 是否错误塞入了太多龙套
6. 用户状态是否污染了 canonical world / character
7. 是否把推断写成了硬事实
8. 是否缺少必要的证据锚点或修订说明

输出顺序必须是：

1. Findings
2. Open Questions / Assumptions
3. 可选的简短总结

Findings 的要求：

1. 按严重程度排序
2. 优先指出 bug、风险、行为回归、数据污染、边界错误、缺失校验
3. 给出明确文件路径
4. 如果没发现问题，要明确说“未发现明确问题”，并说明残留风险

防漂移机制：

1. 开始审校前，先写一张“当前任务卡”：
   - 当前目标
   - 当前审校范围
   - 当前最关注的风险
   - 当前禁止偏离的方向
2. 不需要反复重读整份 prompt，但在以下时点必须回看本 prompt 的这些段落：
   - `目标`
   - `开始审校前，先做一次最小结构读取`
   - `请重点检查`
   - `输出顺序`
   - `Findings 的要求`
   - `防漂移机制`
3. 如果审校文件很多，把”每一组目录 / 每一类文件”视为单阶段，在组与组之间回看上述段落，确认未偏离审校目标。
4. 一旦发现自己开始滑向”直接帮忙改实现”而不是先给 Findings，立即停下，先回看 `输出顺序` 与 `Findings 的要求`。

我会补充给你：

- 要 review 的路径：
- 本次最担心的问题类型：
```
