# 手动补抽与修复

适用场景：

- 自动化提取（automation）完成后，针对 Phase 3.5 一致性报告指出的问题做局部修复
- 补充遗漏的关系、别名、记忆条目等
- 修正与原文有出入的字段

## 可直接使用的提示词

```text
你现在接手本地项目 `persona-engine`，你没有任何额外背景知识，请完全按本提示词执行。

目标：
对我指定的角色或世界数据做一次定向补充或修复。你不是从零提取，而是在已有
自动化提取结果的基础上做增量修正。

开始前，先做一次最小结构读取：

1. 先读 `prompts/shared/最小结构读取入口.md`
2. 读取目标角色/世界的现有数据文件
3. 读取对应的 schema（`schemas/` 目录下）
4. 如果有 Phase 3.5 一致性报告，读取报告了解需修复的问题

修复原则：

1. **最小变更** — 只修改需要修复的字段，不要重写整个文件
2. **证据优先** — 修复内容必须能从原文中找到依据，标注 evidence_ref
3. **自包含快照** — 如果修改了某个 stage_snapshot，确保它仍然是自包含的
   （包含该阶段的完整状态，不依赖 baseline merge）
4. **不跨阶段污染** — 修复某阶段数据时，不要引入该阶段角色尚不知道的信息
5. **内容语言一致** — 中文作品的数据内容使用中文
6. **长度硬门控** — memory_timeline `event_description` 150–200 字、
   `digest_summary` 30–50 字、世界 `stage_events` 每条 50–100 字、
   角色 `stage_events` 每条 50–80 字，均为 schema 硬门控

常见修复类型：

1. **别名补充** — 在 identity.json 的 aliases 中添加遗漏的名称
2. **关系补充** — 在 stage_snapshot 的 relationships 中添加遗漏的关系
3. **记忆补充** — 在 memory_timeline 中添加遗漏的重要事件
4. **跨 stage 一致性** — 修复前后阶段之间的矛盾（如关系状态不连续）
5. **voice/behavior 细化** — 补充 target_voice_map 或 target_behavior_map 的细节

输出要求：

1. 先列出你发现的问题和修复计划
2. 等我确认后再执行修改
3. 每个修改说明：改了什么、为什么改、依据哪章

我会补充给你：

- 要修复的角色/世界：
- 问题描述或 Phase 3.5 报告路径：
- 需要参考的原文章节范围：
```
