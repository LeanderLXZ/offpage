# user 侧 id 字段统一加 `user_` 前缀

## 背景

user 侧 schema 与 character / world 正典层存在同名 id（`memory_id` /
`event_id` 等），语义上属不同命名空间、互不引用。之前未撞到问题是因为
merge / retrieval 代码尚未实装；一旦 runtime 动工，同名字段会使"指向用户
侧还是正典侧"的意图在读代码时需要额外推断。

决策：方案 A（全量前缀化）。user/runtime 命名空间内所有自由 id 统一加
`user_` 前缀，字段名直接承担作用域语义。`runtime/context_character_state`
归属为 user / runtime 命名空间（既不是正典，也不跨层引用正典），同样加
前缀。

讨论背景见历史 todo 条目 `T-USER-ID-RENAME`（本次改动后从
`docs/todo_list.md` 清除）。

## Schema 变更

### `schemas/user/long_term_profile.schema.json`

- `event_history[].event_id` → `user_event_id`（required）
- `memory_history[].memory_id` → `user_memory_id`（required）
- `persona_shift_history[].shift_id` → `user_shift_id`（required）
- `character_drift_history[].drift_id` → `user_drift_id`（required）

### `schemas/user/pinned_memory_entry.schema.json`

- `memory_id` → `user_memory_id`（required）
- description 明确为"用户命名空间内的钉选记忆 ID，与 works/* 正典
  memory_timeline（M-S###-##）不是同一命名空间，不跨层引用"。

### `schemas/user/relationship_core.schema.json`

- `event_refs[].event_id` → `user_event_id`（required）
- `profile_shift_refs[].shift_id` → `user_shift_id`（required）
- `mutual_agreements[].agreement_id` → `user_agreement_id`（required）

### `schemas/runtime/context_character_state.schema.json`

- `mutual_agreements[].agreement_id` → `user_agreement_id`（可选）
- `context_events[].event_id` → `user_event_id`（可选）
- `context_memories[].memory_id` → `user_memory_id`（可选）

每个重命名字段都补上 description，写明所在命名空间以及与正典 id 家族的
边界。

## 正典层保持不变

`M-S###-##` / `E-S###-##` / `SC-S###-##` / `SN-S###-##` id 家族属 works/*
正典层，在以下文件里保留原字段名（`memory_id` / `event_id` /
`scene_id` / `note_id`）：

- `schemas/character/memory_timeline_entry.schema.json`
- `schemas/character/memory_digest_entry.schema.json`
- `schemas/world/world_event_digest_entry.schema.json`
- 所有引用这些 id 的架构文档与 loader 说明

这些 id 带 pattern 校验，与加前缀后的 user 侧自由 id 在字段名层面已经
消歧。

## 模板影响

`users/_template/` 下现有模板均为空数组 / 空对象，不含任何具体 id 值，
因此无需联动修改。模板向量（`long_term_profile.json`、
`relationship_core/manifest.json`、`contexts/{context_id}/character_state.json`）
在 jsonschema 下仍校验通过（`character_state.json` 的 `{stage_id}`
占位失败属预存在问题，与本次改动无关）。

## 代码影响

`automation/` 与 `simulation/` 现有代码均未消费这些 user 侧 id 字段
（user merge / runtime 尚未实装）。`automation/repair_agent/checkers/
structural.py` 里的 `memory_id` / `event_id` 引用针对 character / world
正典层，不受影响。
