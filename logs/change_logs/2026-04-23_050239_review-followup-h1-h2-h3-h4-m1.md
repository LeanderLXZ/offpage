## H3 — commit-and-save 崩溃窗口

原流程（`orchestrator.py:1810-1815`）：`commit_stage` → 内存赋值
`committed_sha` → `transition(COMMITTED)` → `phase3.save`。若进程在
`commit_stage` 返回后、`save` 前崩溃，git 已有提交但 progress 文件
`committed_sha=""` / state≠COMMITTED，reconcile 会把 stage 打回
PENDING 触发无限 "commit returns no SHA → FAILED" 循环。

### 修复（推荐方案 A）

- `orchestrator.py:1809-1817`：在 `commit_stage` 成功后**先
  `phase3.save`** 把 SHA 落盘，再做 `transition(COMMITTED)` + 第二次
  `phase3.save`。崩溃窗口从"SHA 完全丢"缩到"SHA 写入但 state 未升"
- `progress.py`：新增 `StageEntry.force_promote_to_committed(reason)`
  escape hatch（与既有 `force_reset_to_pending` 对称），标准 transition
  表仍禁止任意 → COMMITTED
- `progress.py:reconcile_with_disk`：在 COMMITTED 早退支之后、非
  COMMITTED 通用 reconcile 之前插入中间分支——当
  `committed_sha` 非空 ∧ `_git_object_exists(sha)` ∧ 所有 per-stage
  产物齐全时，`force_promote_to_committed` 自动升级，`reason` 写
  "reconcile promote from {state} — committed_sha present and live in git"

### 验证

- `python -c 'from automation.persona_extraction import progress, orchestrator'` 通过
- `StageEntry.force_promote_to_committed` 存在，transition 表未改动
