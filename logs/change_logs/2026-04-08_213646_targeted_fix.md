# Targeted Fix：审校失败分级与定点修复机制

**日期**: 2026-04-08
**范围**: automation (orchestrator, progress, prompt_builder, prompt template), docs, ai_context

## 动机

batch_003 测试中，reviewer 发现 `experienced_events` 中一个字段值与其他来源
矛盾。原机制对此做全量回滚+重试，浪费 25 分钟提取时间和全量 token。大多数
reviewer FAIL 是几个字段级的事实错误，不需要重新生成全部产出。

## 变更内容

### 1. 失败分级逻辑

reviewer FAIL 后，`_is_fixable()` 对 findings 做分类：

- **局部问题（fixable）**：≤5 个 error 级 findings，全部指向具体字段位置，
  不涉及文件缺失或结构性错误
- **系统性问题（not fixable）**：文件缺失、大段内容为空、整体语义偏差、
  结构性错误、或 error 数 >5

### 2. 新增 FIXING 状态

状态机新增 `fixing` 状态：

```
reviewing → fixing → passed → committed
               │
               └→ failed → retrying → extracting
```

- REVIEWING → FIXING：局部问题，进入定点修复
- FIXING → PASSED：修复后校验通过
- FIXING → FAILED：修复后仍有错误，回滚到全量重试路径

### 3. 定点修复 prompt (`targeted_fix.md`)

- 只传有问题的文件 + 具体错误描述 + 原文证据
- 指令为"最小化修改，只改审校指出的问题"
- 信息权威层级：原文 > 摘要 > memory_timeline > snapshot > world_snapshot
- 修复后只跑程序化校验（零 token），不再跑 semantic review

### 4. Orchestrator 流程变更

- 步骤从 5 步扩展为 6 步（新增 Step 5: Targeted fix）
- reviewer FAIL → `_is_fixable()` 判定 → fixable 则 transition 到 FIXING
- FIXING 步骤：运行 fix agent → re-validate → passed 或 fallback to FAILED
- FIXING 中断恢复：resume 时保持 FIXING 状态，重新运行 fix agent

### 5. 文档更新

- `docs/requirements.md` §11.5：失败分级流程图、局部 vs 系统性判定条件、
  targeted fix 设计原则
- `ai_context/current_status.md`：状态机描述、failure triage、prompt 列表
- `ai_context/decisions.md`：新增决策 25a
- `docs/architecture/extraction_workflow.md`：流程图和设计决策
- `automation/README.md`：架构图和 batch 流程描述

## 向后兼容性

- `fixing` 状态对已有 progress 文件透明（旧文件中不存在该状态）
- `_is_fixable()` 保守判定——不确定时走全量回滚路径
- 现有 batch_003 (failed, retry=2) 不受影响，resume 时会被自动重置
